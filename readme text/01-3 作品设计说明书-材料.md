## 第一章 需求分析

本作品面向普通家庭用户、高频用嗓职业人群、慢性咽喉不适人群及术后随访用户，解决“轻症阶段不知道是否需要关注、线下喉镜检查门槛较高、健康数据难以连续追踪、专业结果难理解”的问题。作品不是替代医生诊断的医疗系统，而是以本地音频识别模型为核心初筛能力，结合喉镜影像档案、AI 辅助解读、健康档案、症状记录、饮食建议、就医辅助和隐私数据权利管理，形成居家自查与长期管理闭环。

同类作品主要包括健康问答类应用、医院线上问诊平台、单一语音检测或喉镜识别原型。本作品对标“家庭自查 + 风险提醒 + 数据沉淀 + 就医准备”的综合健康管理工具，强调可追踪、可解释、可运营和隐私合规。

| 对比维度 | 普通健康问答应用 | 医院线上问诊平台 | 单一模型演示原型 | 本作品 |
| --- | --- | --- | --- | --- |
| 核心目标 | 科普咨询 | 医患沟通 | 算法展示 | 居家初筛与随访管理 |
| 检测能力 | 无本地检测 | 依赖医生判断 | 单点检测 | 本地音频模型 + 喉镜影像预测 |
| 结果解释 | 通用文本 | 医生解读 | 多为模型输出 | 图谱、热力图、AI 辅助解读 |
| 数据闭环 | 弱 | 依赖医院系统 | 弱 | 档案、症状、饮食、就医、标注授权闭环 |
| 隐私能力 | 基础协议 | 医疗平台规则 | 通常较弱 | 隐私设置、导出、删除、授权明细 |

主要功能包括：录音/上传音频初筛、MFCC/Mel 图谱生成、喉镜图片上传与本地模型预测、Qwen-VL 图像分析、健康档案与趋势追踪、症状日志、饮食推荐、健康科普、就医建议、报告导出、模型标注闭环、后台内容和数据管理。主要性能目标为：常规音频检测 3 到 10 秒内有反馈，模型调用失败时返回明确错误阶段；前端支持微信小程序和 H5；后端接口具备登录校验、文件校验、软删除、日志留痕和可扩展配置。

### 2.4 人机界面结构

人机界面按“用户自查端”和“后台管理端”两条线展开。用户端以首页为统一入口，围绕“开始自查、查看结果、继续管理、必要时就医”组织页面，包括录音检测、检测结果、AI 报告、喉镜影像详情、健康档案、饮食建议、就医辅助和隐私设置等。页面风格采用轻医疗绿色视觉，重点突出当前风险、下一步行动和“仅供参考”的医学边界提示，避免把模型结果包装成确诊结论。

管理端面向管理员、运营人员或后续研究人员，重点不在移动端体验，而在数据查询、配置和追踪。后台提供检测记录、用户管理、科普内容、饮食库、问卷、医院医生、模型日志、确诊标注、隐私申请等页面，支撑内容维护、异常排查、数据导出和合规处理。模型与 AI 服务不直接暴露给普通用户，而是通过后端接口统一调度，输出结构化 JSON，保证前端展示、数据库归档和失败追踪能够使用同一套结果格式。

### 2.5 模块间接口概述

系统模块之间采用“前端统一请求、后端统一调度、模型能力独立封装”的接口组织方式。用户端和管理端均通过 REST 接口访问 Spring Boot 后端，登录态由统一请求封装携带 Token，后端根据当前用户身份完成权限校验和数据归属判断。

音频自查接口：

- `/api/audio/detect`：小程序上传音频文件，后端保存临时文件后调用 Python 音频推理脚本，返回初筛结果、置信度、图谱地址和错误阶段信息。

AI 辅助解读接口：

- `/FrontAI/AnalyzeAudioRecord`：基于已有检测记录生成音频图谱 AI 解读报告。
- `/FrontAI/AnalyzeSpectrumUpload`：上传 MFCC/Mel 图谱并生成 AI 解读。
- `/FrontAI/Chat`：提供喉部健康相关的问答与建议。

喉镜影像接口：

- `/Front/Laryngoscope/List`、`/Front/Laryngoscope/SavePhoto`：完成喉镜图片档案的查询、上传和编辑。
- `/Front/Laryngoscope/LocalPredict`：调用本地 RepAlexNet 喉镜预测模型，生成分类结果、概率分布和 Grad-CAM 热力图。
- `/Front/Laryngoscope/QwenAnalyze`：调用 Qwen-VL 对喉镜图片进行结构化辅助分析。

健康管理与运营接口：

- `/Front/Diet/*`：承接饮食推荐、食物库浏览和饮食记录。
- `/Front/Medical/*`：承接就医建议、医生医院信息和就诊准备报告。
- `/Front/Privacy/*`：承接隐私设置、数据导出申请和删除申请。
- `/Front/ModelLabel/*`：承接确诊标注、授权状态和模型优化数据闭环。

后端内部按照“Controller—Service—Mapper—模型/AI工具”的方式组织。Controller 负责接收参数和校验权限，Service 负责编排业务流程，Mapper 负责 MySQL 读写，Python 模型脚本和 Qwen-VL 客户端作为外部能力由 Service 层统一调度。这样的结构使前端接口保持稳定，同时方便后续替换模型脚本、调整 Prompt、扩展数据库字段或接入异步任务队列。

## 第三章 详细设计

### 3.1 典型使用流程

【图 3-1 典型使用流程图，需用 Gemini 绘制】

本图用于展示普通用户从居家自查到长期管理的闭环流程。主线为：进入首页 → 录音或上传音频 → 本地模型检测 → 查看良性/疑似风险、置信度和图谱。查看结果后分出三个动作：生成 AI 辅助解读、同步健康档案和趋势、记录症状。健康档案继续连接饮食建议、就医建议/就诊准备报告；用户线下就诊后可提交确诊标注，若选择匿名授权，则进入模型优化闭环。图中应使用回环箭头体现“标注授权 → 模型优化 → 后续检测能力提升”的闭环。

### 3.2 界面设计重点

用户端界面围绕移动端真实使用路径设计。首页 `Index.vue` 承担功能分发作用，将音频自查、健康档案、饮食建议、就医辅助、隐私设置等入口集中展示，减少用户寻找功能的成本。音频自查由 `AudioRecord.vue` 和 `AudioResult.vue` 承接，前者强调录音/上传和检测状态反馈，后者展示结果、置信度、模型信息、健康建议和后续操作入口。

AI 解读页面主要解决普通用户看不懂图谱和模型术语的问题，相关页面包括 `AudioAIReport.vue`、`AudioAIChat.vue`、`AudioAISpectrumUpload.vue` 等。页面内容按“风险等级、通俗摘要、健康建议、何时就医、免责声明”分段展示，使检测结果从专业模型输出转化为用户能够理解的健康参考信息。

喉镜影像模块由 `LaryngoscopePhotoList.vue`、`LaryngoscopePhotoDetail.vue` 和 `LaryngoscopePhotoEdit.vue` 组成。列表页负责历史喉镜图片的归档、摘要和入口展示；详情页集中展示喉镜原图、本地模型预测结果、类别概率分布、Grad-CAM 热力图和 Qwen 图像分析报告，便于用户对同一张喉镜图片进行连续查看和对比理解。

健康档案 `HealthRecord.vue` 是长期管理入口，汇总风险等级、趋势标签、近期症状和检测历史。饮食、就医、隐私、标注等页面分别承担“日常管理、线下就诊准备、数据权利控制、模型优化反馈”的任务。整体界面设计遵循摘要优先原则：先给用户一个可理解的结论，再提供图谱、概率、日志等细节，避免移动端一次性堆叠过多专业信息。

### 3.3 数据库设计

系统采用 MySQL，核心设计以用户、检测记录、健康档案、喉镜影像、AI报告、标注授权为主线。为便于历史追踪，多数业务表保留 `CreationTime`、`UpdateTime`、`IsDelete` 等字段，删除操作以软删除为主。

【图 3-2 核心数据库关系 ER 图，需用 Gemini 绘制】

本图用于说明核心业务表之间的主关系。`APP_USER` 与 `TAudioScreenRecord`、`TLaryngoscopePhoto`、`TPersonalLaryngealHealthRecord`、`TSymptomLog`、`TModelOptimizeLabel` 均为一对多关系。`TAudioScreenRecord` 与 `TAudioAiReport`、`TModelInterfaceCallLog`、`TModelOptimizeLabel` 为一对多关系。`TLaryngoscopePhoto` 与 `TLaryngoscopeLocalPredictionRecord`、`TLaryngoscopeQwenAnalysisRecord` 为一对多关系。图中只需展示核心实体名和一对多关系，字段明细保留在下方表格中。

数据库按照业务闭环分为四类核心数据。第一类是用户与自查数据，包括用户信息、音频自查记录和喉镜影像档案。`TAudioScreenRecord` 保存音频地址、图谱地址、初筛结果、置信度和检测状态；`TLaryngoscopePhoto` 保存喉镜图片、检查时间、医院、检查方式、视角和备注。第二类是结果解释数据，包括模型调用日志、AI 报告、本地喉镜预测记录和 Qwen 图像分析记录，用于保存模型版本、运行耗时、原始 JSON、热力图地址、风险摘要和失败原因。

第三类是长期健康管理数据，包括健康档案、症状日志、饮食记录、问卷答题、提醒消息和就医报告。健康档案通过最新检测和症状信息计算风险等级与趋势，避免把一次检测孤立展示。第四类是合规和优化闭环数据，包括隐私设置、数据导出/删除申请、模型优化标注和授权状态。用户可在就诊后提交确诊结果，并选择是否匿名授权用于后续模型优化。

整体设计基本遵循关系型数据库范式：用户、检测、喉镜图片、报告、标注等均拆分为独立实体，通过用户 ID、检测 ID、喉镜图片 ID 建立关联。少量字段采用 JSON 或快照方式保存，例如 `RawOutputJson`、`ProbabilitiesJson`、`ReportJson`、`RawResponse` 等。这类设计有意保留模型原始输出、AI 报告结构和历史上下文，便于审计、复现和后续解析；虽然不完全追求字段原子化，但符合模型结果和 AI 报告版本变化频繁的实际需要。

### 3.4 关键技术与算法

本作品的核心技术难点在于把模型检测、可解释结果、AI 辅助说明和健康管理数据连成一条稳定链路。音频自查部分使用 Python + PyTorch 的 Siamese 多任务模型，后端通过 `ProcessBuilder` 调用 `detect_audio_siamese.py`，脚本加载 `voice_best_model.pth` 后输出肿瘤/非肿瘤概率、置信度、语音性别等结构化字段。为了让结果更直观，脚本同时生成 MFCC、Mel 频谱和波形图，供前端结果页和 AI 解读模块使用。

喉镜影像部分使用 `predict_larynscope.py` 调用 RepAlexNet 权重 `repalexnet_heatmap_n100.pth`，对历史喉镜图片进行分类预测，并生成 6 类概率分布。为了避免模型输出变成黑箱，系统使用 Grad-CAM 生成热力图，将模型关注区域叠加到喉镜图像上。该结果并不作为诊断结论，而是作为用户理解影像风险提示和后续就医沟通的辅助材料。

AI 辅助解读部分接入 Qwen / Qwen-VL。后端把音频图谱或喉镜图片转为 Base64，并配合严格 Prompt 要求模型返回 JSON。报告中包含风险等级、通俗摘要、观察点、建议、复查提示和免责声明。这里的 AI 只负责解释和建议，不覆盖本地模型输出，也不替代医生诊断。

健康趋势和标注闭环是作品从“单次检测工具”升级为“长期管理平台”的关键。系统会根据近期检测、症状严重程度和持续时间更新健康档案风险等级；用户就诊后可提交确诊标注，并自主选择是否匿名授权。这样既能帮助用户管理自身记录，也为后续模型优化提供合规的数据来源。

## 第四章 测试报告

### 4.1 主要测试用例

测试围绕“能否完整跑通真实用户路径”和“异常情况能否被清楚反馈”展开。音频自查链路重点测试了文件格式、大小限制、模型脚本调用、JSON 解析、图谱地址生成和检测记录入库。测试过程中分别上传 mp3、wav、flac、m4a 文件以及超限文件，系统能够让合法文件进入检测，并对不支持格式或超 10MB 文件给出明确提示。使用样例音频调用 `/api/audio/detect` 时，接口能够返回初筛结果、置信度、图谱 URL 和检测记录 ID。

喉镜影像链路重点测试了图片归属校验、本地模型预测、热力图生成和 Qwen-VL 图像分析。用户只能分析自己名下的喉镜照片；本地预测成功后返回类别、置信度、概率分布和热力图地址；Qwen-VL 分析成功后返回风险等级、观察点、建议和免责声明。健康档案、症状日志、饮食建议、就医报告、隐私申请和确诊标注也进行了联动测试，确认检测完成后可更新档案，症状记录会影响风险等级，导出/删除申请能够留痕，标注授权不会越权访问。

管理后台测试主要覆盖分页查询、条件筛选、详情查看、编辑、软删除、导出和统计展示。对用户数据相关接口进行了越权测试，使用其他用户身份访问检测记录、AI 报告、喉镜记录、标注详情时，后端能够拒绝或返回无权限结果。

### 4.2 修正过程与技术指标

测试中发现的主要问题包括：音频脚本输出非 JSON 时前端难以定位错误，喉镜图片本地路径与访问 URL 映射不一致，Qwen 返回内容偶尔不是严格 JSON。针对这些问题，项目统一了 Python 脚本输出格式，要求成功和失败都输出单一 JSON；后端增加文件路径解析兜底，兼容项目根目录和后端工程目录两种运行方式；AI 报告解析时增加 JSON 主体截取和失败记录保存，避免用户只看到“生成失败”而没有原因。

从测试结果看，常规音频检测链路通常能在 3 到 10 秒内返回，模型调用超时阈值设置为 120 秒，避免请求长期挂起。安全性方面，系统使用 JWT 登录校验、用户数据归属校验、文件类型和大小校验，并避免公开真实密钥配置。可用性方面，前端提供加载、成功、失败和空状态；后端错误信息包含阶段和说明，便于定位。扩展性方面，音频模型、喉镜模型和 Qwen 接口均通过服务层解耦，模型路径、输出目录和超时时间可配置。部署方面，前端、管理端、后端和 Python 模型脚本分层清晰，当前可本机联调，后续可继续向 Docker 和异步队列演进。

## 第五章 安装及使用

### 5.1 环境要求

默认部署需要准备 JDK 17、Maven、MySQL 8.x、Node.js 16+、Python 3.8+。后端运行 Spring Boot 服务，管理端运行 Vue3 + Vite 项目，小程序端建议使用 HBuilderX 或微信开发者工具。模型环境需要安装 PyTorch、torchvision、librosa、matplotlib、OpenCV、Pillow、NumPy 等依赖，并保证 `voice/voice_best_model.pth`、`voice/detect_audio_siamese.py`、`laryngoscope prediction/predict_larynscope.py`、`laryngoscope prediction/repalexnet_heatmap_n100.pth` 等文件路径可访问。若启用 AI 辅助解读，还需要配置阿里云百炼 DashScope API Key 及 Qwen / Qwen-VL 模型参数。

### 5.2 默认安装过程

1. 初始化数据库：导入 `healthcontrol.sql` 及相关增量 SQL，如喉镜预测表结构 `laryngoscope_prediction_db_changes.sql`。
2. 配置后端：复制 `HealthControl.springboot/src/main/resources/application.example.yml` 为 `application.yml`，填写数据库、邮件、AI Key、文件目录等配置。
3. 启动后端：

```bash
cd HealthControl.springboot
mvnw.cmd spring-boot:run
```

4. 启动管理后台：

```bash
cd HealthControl.elementui
npm install
npm run dev
```

5. 运行小程序端：使用 HBuilderX 打开 `HealthControl.uniapp`，配置接口地址后运行到微信小程序或 H5。
6. 检查 Python 模型：确认 `voice/voice_best_model.pth`、`voice/detect_audio_siamese.py`、`laryngoscope prediction/repalexnet_heatmap_n100.pth` 等文件存在，并安装所需 Python 依赖。

### 5.3 典型使用流程

## 第六章 项目总结

本项目从单一“音频识别”逐步扩展为“居家自查 + 健康档案 + AI辅助解释 + 就医辅助 + 隐私治理 + 标注闭环”的综合系统。开发过程中最大的难点不是某一个页面或接口，而是多端、多模型、多数据表之间的闭环协同：音频检测结果要写入档案，档案要影响饮食和就医建议，用户确诊标注又要回流模型优化，同时还必须处理健康数据的隐私边界。

项目制作过程中，前端、后端、数据库和模型脚本需要同步推进。前端负责把复杂流程压缩成用户能理解的操作路径；后端负责统一权限、文件、模型调用和数据归档；模型脚本负责稳定输出结构化结果；数据库则承担追踪历史和支撑后台运营的任务。为了避免“算法演示”和“健康产品”之间脱节，项目把检测结果、症状记录、档案趋势、饮食建议、就医报告和标注授权串成一个连续流程，使每一次自查都能沉淀为后续管理依据。

项目开发也提升了对医疗健康类产品边界的理解：AI 和本地模型可以帮助用户更早关注风险、理解结果、准备就医资料，但不能替代医生诊断。因此作品在结果页、AI 报告、喉镜分析和就医建议中均强调“仅供健康参考”。在工程实现上，项目完成了 UniApp 小程序、Vue 管理端、Spring Boot 后端、MySQL 数据库、Python 模型脚本和 Qwen-VL 多模态接口的整合，具备进一步试点和演示基础。

后续升级方向包括：扩大脱敏样本和标注数据规模，继续优化音频模型泛化能力；完善喉镜影像类别标签和热力图展示；增强异步任务队列、Docker 部署和模型版本管理；引入更细粒度的隐私授权和审计机制；与社区、学校、高频用嗓职业群体或基层健康管理机构合作，探索健康宣教和早期风险筛查场景。商业推广上，可先以高频用嗓人群健康管理、社区筛查宣教和基层随访辅助为试点场景，逐步验证用户留存、模型稳定性和合规流程。
