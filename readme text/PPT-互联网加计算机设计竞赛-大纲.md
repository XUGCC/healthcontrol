## 3. 行业背景与用户痛点

**页标题建议**

喉部健康管理存在高门槛、低持续性、难解释三大痛点

**核心要点**

- 医院喉镜检查和专科就诊成本较高，普通用户往往“有不适但不确定是否就医”。
- 传统自查方式主观性强，缺少量化结果和连续追踪能力。
- 检测结果常常停留在“一个数值或一个结论”，用户难以理解，也难以转化为行动。
- 健康数据分散在不同环节，缺少统一档案和长期管理机制。

**讲述要点**

- 评委最关心的是“为什么现在需要这个产品”。
- 重点强调：居家自查降低门槛，长期管理提升价值，AI 解读解决“看不懂”的问题。

## 4. 项目定义与目标用户

**页标题建议**

我们做的是一个面向居家场景的喉部健康管理平台

**项目定义**

- 本项目聚焦“居家喉部健康管理”场景。
- 用户在小程序内录音或上传音频，系统完成嗓音风险初筛，并联动健康档案、症状记录、饮食建议、问卷评估、就医辅助和隐私管理。
- 核心目标是把一次性检测变成可持续的健康管理服务。

**目标用户**

- 普通用户，尤其是经常用嗓、嗓音不适、需要自我监测的人群。
- 慢病随访或术后恢复人群，需要低成本、可持续的居家追踪方式。
- 基层健康管理场景下的运营人员、医生或管理者。

**讲述要点**

- 这一页要讲清楚“我们到底解决谁的问题”。

## 5. 解决方案与产品闭环

**页标题建议**

从音频自查到健康管理，我们做的是完整闭环

**核心流程**

1. 用户录音或上传音频。
2. 后端调用 Python 音频推理脚本生成初筛结果、置信度和图谱。
3. 系统展示波形 / MFCC / Mel 等可视化结果。
4. 用户查看 AI 解读报告，获取更易理解的建议。
5. 结果沉淀到健康档案、症状日志和历史记录中。
6. 结合提醒、科普、饮食、问卷、就医辅助等模块，形成持续管理。

**讲述要点**

- 强调“检测只是入口，管理才是核心价值”。
- 说明项目不是单点模型，而是完整产品闭环。

## 6. 核心功能与使用场景

**页标题建议**

核心功能不是堆模块，而是围绕健康管理形成组合能力

**核心功能**

- 音频自查：录音 / 上传音频，生成初筛结果和图谱。
- AI 解读：把图谱和结果转化成用户能理解的风险等级、摘要和建议。
- 健康档案：沉淀用户基础信息、检测记录和长期健康变化。
- 症状追踪：记录不适、变化趋势和历史对比。
- 个性化提醒：支持定期自查、护嗓提醒和风险提示。
- 科普与饮食：提供喉部健康内容与饮食建议。
- 问卷与就医辅助：支持风险评估、就诊准备和报告导出。
- 隐私与数据权利：支持偏好设置、导出申请和删除申请。

**典型场景**

- 用户嗓子不舒服，先在家做初筛，再决定是否就医。
- 用户长期高频用嗓，定期检测并查看趋势变化。
- 用户术后或慢病随访，通过档案和提醒保持持续管理。

## 7. 技术方案与创新点

**页标题建议**

我们不仅能做出来，还能解释为什么这样做

**技术方案**

- 小程序端使用 UniApp + Vue3 负责用户交互和结果展示。
- 管理后台使用 Vue3 + Element Plus 负责运营、医生和管理员侧管理。
- 后端使用 Spring Boot 3 + MyBatis Plus + MySQL 承接业务与数据流转。
- Python 负责音频模型推理和图谱生成。
- AI 解读模块用于将检测结果转化为结构化报告。

**创新点**

- 检测结果 + 图谱展示 + AI 解读三层联动，而不是只给一个分类结果。
- 支持健康档案、症状追踪、提醒、饮食、问卷、就医辅助等连续管理能力。
- 引入模型调用日志与标注闭环，支持后续优化和版本迭代。
- 加入隐私设置与数据权利模块，具备真实落地所需的合规意识。

**讲述要点**

- 这一页要体现“技术不是炫技，而是为产品价值服务”。
- 如果现场强调算法，可补一句“模型推理脚本统一输出 JSON，便于后端稳定集成和追踪”。
