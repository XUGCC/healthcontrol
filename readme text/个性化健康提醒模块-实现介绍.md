## 个性化健康提醒模块 - 实现介绍文档

## 一、模块概述

本模块实现了“个性化健康提醒”能力，通过用户可配置的提醒规则和后端定时任务，为居家喉部健康自查小程序提供：

- 自查提醒、护嗓提醒、风险随访提醒等多种提醒类型；
- 小程序内的消息中心查看、筛选和已读管理；
- 与健康档案（风险等级、末次自查时间）的联动。

本文档介绍已实现的主要功能、关键代码位置、运行方法和测试步骤。

## 二、主要实现功能

### 2.1 后端功能实现

#### 2.1.1 个性化提醒配置服务（TPersonalHealthRemind）

**核心文件：**

- 实体与 DTO  
  - `HealthControl.springboot/src/main/java/com/example/web/entity/TPersonalHealthRemind.java`  
  - `HealthControl.springboot/src/main/java/com/example/web/dto/TPersonalHealthRemindDto.java`
- Mapper 与 Service  
  - `HealthControl.springboot/src/main/java/com/example/web/mapper/TPersonalHealthRemindMapper.java`  
  - `HealthControl.springboot/src/main/java/com/example/web/service/TPersonalHealthRemindService.java`  
  - `HealthControl.springboot/src/main/java/com/example/web/service/impl/TPersonalHealthRemindServiceImpl.java`
- 控制器  
  - `HealthControl.springboot/src/main/java/com/example/web/controller/TPersonalHealthRemindController.java`
- 数据表脚本  
  - `HealthControl.springboot/create_tpersonalhealthremind_table.sql`

**主要功能：**

1. **提醒配置管理**
   - 通过 `TPersonalHealthRemind` 表为每个用户记录多条提醒配置：  
     - `RemindType`：0=定期自查，1=护嗓，4=风险随访（扩展）；  
     - `RemindTime`：字符串 `HH:mm`；  
     - `RepeatFrequency`：简化为业务层映射的频率（天数）；  
     - `RemindStatus`：0=关闭，1=开启；  
     - `LastPushTime`：最近一次生成提醒消息的时间，用于去重控制。

2. **分页查询与条件过滤**
   - `TPersonalHealthRemindServiceImpl.BuilderQuery` 中基于 `UserId`、`RemindType`、`RemindStatus`、`RemindTimeRange` 等构建查询条件；
   - 支持关键词模糊查询（针对内容字段），便于管理端后续扩展。

3. **新增/修改提醒配置**
   - `CreateOrEdit(TPersonalHealthRemindDto input)`：  
     - 若 `Id` 为空，则新增记录；  
     - 若 `Id` 不为空，则更新对应提醒；  
     - 内部通过 `saveOrUpdate` 实现统一处理。

**如何调用：**

- 前端通过 `/TPersonalHealthRemind/List` 拉取当前用户的所有提醒配置；
- 前端通过 `/TPersonalHealthRemind/CreateOrEdit` 将用户在设置页上的修改保存回数据库。

#### 2.1.2 消息通知服务（MessageNotice）

**核心文件：**

- 实体与 DTO  
  - `HealthControl.springboot/src/main/java/com/example/web/entity/MessageNotice.java`  
  - `HealthControl.springboot/src/main/java/com/example/web/dto/MessageNoticeDto.java`
- 查询模型与 Mapper  
  - `HealthControl.springboot/src/main/java/com/example/web/dto/query/MessageNoticePagedInput.java`  
  - `HealthControl.springboot/src/main/java/com/example/web/mapper/MessageNoticeMapper.java`
- Service 与实现  
  - `HealthControl.springboot/src/main/java/com/example/web/service/MessageNoticeService.java`  
  - `HealthControl.springboot/src/main/java/com/example/web/service/impl/MessageNoticeServiceImpl.java`
- 控制器  
  - `HealthControl.springboot/src/main/java/com/example/web/controller/MessageNoticeController.java`
- 数据表脚本  
  - `HealthControl.springboot/create_messagenotice_table.sql`

**主要功能：**

1. **消息记录存储**
   - 使用 `MessageNotice` 表存储所有类型的提醒消息：  
     - `Title`：消息标题（如“定期自查提醒”、“护嗓小贴士”、“高风险随访提醒”）；  
     - `Content`：消息正文；  
     - `Type`：CHECKUP / VOICE_CARE / FOLLOWUP / ANNOUNCEMENT 等；  
     - `UserId`：接收用户；  
     - `IsRead`：是否已读；  
     - `ActionType` / `ActionParam`：前端点击后的跳转行为配置。

2. **消息列表查询**
   - `MessageNoticeServiceImpl.List`：  
     - 根据 `UserId`、`Type`、`IsRead`、时间范围等条件构建查询；  
     - 支持分页（Page + Limit）；  
     - 默认按 `CreationTime` 倒序返回最近的消息。

3. **已读状态更新**
   - `MarkRead(Integer id, Integer userId)`：校验归属用户后，将单条消息标记为已读；  
   - `MarkAllRead(Integer userId)`：将某个用户所有未读消息批量标记为已读。

4. **通用创建/编辑接口**
   - `CreateOrEdit(MessageNoticeDto input)`：用于手动创建或更新消息记录；  
   - 定时任务中通过 `MessageNotice.MapToDto()` + `CreateOrEdit` 快速插入消息。

**如何调用：**

- 小程序消息中心页通过 `/MessageNotice/List` 拉取消息列表；  
- 点击单条消息时调用 `/MessageNotice/MarkRead`；  
- 点击“全部已读”调用 `/MessageNotice/MarkAllRead`。

#### 2.1.3 定时任务：护嗓提醒、自查提醒、风险随访

本模块按照需求文档第 8 章，实现了三个后端定时任务，负责根据用户配置和健康档案自动生成相应的提醒消息。

##### (1) 每日护嗓提醒任务（VOICE_CARE）

**实现文件：**

- `HealthControl.springboot/src/main/java/com/example/web/jobs/PersonalHealthRemindJob.java`

**核心逻辑：**

1. 定时调度
   - 使用 `@Scheduled(cron = "0 */5 * * * ?")`，每 5 分钟执行一次；  
   - 计算当前时间的 `HH:mm` 字符串，用于与 `RemindTime` 匹配。

2. 查询护嗓提醒配置
   - 过滤条件：  
     - `RemindType = 1`（护嗓）；  
     - `RemindStatus = 1`（开启）；  
     - `RemindTime = 当前 HH:mm`；  
     - `IsDelete != 1`（未删除）。

3. 去重与消息生成
   - 使用配置的 `LastPushTime` 做日级去重：同一条配置同一天只推送一次；  
   - 为每条配置构建一条 `MessageNotice`：  
     - `Type = "VOICE_CARE"`；  
     - `Title = "护嗓小贴士"`；  
     - `Content`：优先使用配置中的 `RemindContent`，否则使用默认护嗓提示文案；  
     - `UserId` 为配置对应用户；  
     - `IsRead = false`，`IsSend = false`，`IsSuccess = true`。
   - 通过 `messageNoticeService.CreateOrEdit(msg.MapToDto())` 插入消息；  
   - 更新该配置的 `LastPushTime` 为当前时间。

##### (2) 周期自查提醒任务（CHECKUP）

**实现文件：**

- `HealthControl.springboot/src/main/java/com/example/web/jobs/PeriodicCheckupReminderJob.java`

**核心逻辑：**

1. 定时调度
   - 使用 `@Scheduled(cron = "0 30 0 * * ?")`，每天 00:30 执行一次；

2. 查询自查提醒配置
   - 过滤条件：  
     - `RemindType = 0`（定期自查）；  
     - `RemindStatus = 1`；  
     - `IsDelete != 1`。

3. 联动健康档案
   - 对每个配置，根据 `UserId` 在 `TPersonalLaryngealHealthRecord` 中查找该用户最新健康档案；  
   - 获取 `LastScreenTime`（最近一次自查时间），若为空则视为“尚未自查”，优先提醒。

4. 根据频率判断是否需要提醒
   - 将配置的 `RepeatFrequency` 映射为间隔天数（如：每天、每周一次、每 30 天一次等，当前实现中以简化方式处理）；  
   - 如果 `当前日期 - LastScreenTime ≥ 频率天数`，则满足提醒条件；  
   - 再结合配置的 `LastPushTime`，避免同一天重复推送。

5. 生成消息
   - 为用户插入一条 `MessageNotice`：  
     - `Type = "CHECKUP"`；  
     - `Title = "定期自查提醒"`；  
     - `Content`：提示用户完成一次喉部自查；  
     - `ActionType = "GOTO_AUDIO_DETECT"`，前端点击后可直接跳到录音检测页。  
   - 更新自查配置的 `LastPushTime`。

##### (3) 风险随访提醒任务（FOLLOWUP）

**实现文件：**

- `HealthControl.springboot/src/main/java/com/example/web/jobs/RiskFollowupReminderJob.java`

**核心逻辑：**

1. 定时调度
   - 使用 `@Scheduled(cron = "0 0 9 ? * MON")`，每周一 09:00 执行一次；

2. 选出中高风险用户
   - 从 `TPersonalLaryngealHealthRecord` 中查询：  
     - `RiskAssessmentLevel` ∈ {1（中风险）, 2（高风险）}；  
     - `IsDelete = 0`。

3. 匹配随访提醒配置
   - 按 `UserId` 在 `TPersonalHealthRemind` 中查找该用户的随访提醒配置：  
     - 约定 `RemindType = 4`；
     - `RemindStatus = 1` 且未删除；
   - 没有配置则不推送，保留可配置性。

4. 按风险等级控制随访频率
   - 高风险（2）：默认每 7 天一次随访提醒；  
   - 中风险（1）：默认每 14 天一次随访提醒；  
   - 使用配置自身的 `LastPushTime` 进行间隔判断，未到间隔天数则不推送。

5. 生成消息
   - 为用户插入一条 `MessageNotice`：  
     - `Type = "FOLLOWUP"`；  
     - `Title`：高风险用户为“高风险随访提醒”，中风险为“风险随访提醒”；  
     - `Content`：包含风险等级的说明、建议尽快就医或复查，以及最近一次自查时间（如有）；  
     - `ActionType = "GOTO_HEALTH_RECORD"`，点击后跳转到健康档案页查看详细情况。  
   - 更新对应随访配置的 `LastPushTime`。

### 2.2 前端功能实现（Uni-app）

#### 2.2.1 健康提醒总览页（HealthReminder.vue）

**文件路径：**

- `HealthControl.uniapp/pages/Front/HealthReminder.vue`

**主要功能：**

1. **总览卡片**
   - 顶部展示“提醒总览”：自查、护嗓、随访三类当前启用状态与简要描述；  
   - 通过计算属性 `allEnabled` 判断是否至少开启了一类提醒。

2. **自查提醒设置**
   - 开关 `checkupEnabled` 控制是否启用定期自查提醒；  
   - 提醒频率：通过 `picker` 选择频率（每天/工作日/每周/每 30 天），前端使用 `checkupFrequencyOptions` 映射；  
   - 提醒时间：通过 `picker mode="time"` 选择 `HH:mm`；  
   - 根据加载到的 `RiskAssessmentLevel` 动态展示推荐文案 `checkupRecommendText`。

3. **护嗓提醒设置**
   - 开关 `voiceCareEnabled` 控制护嗓提醒；  
   - 提醒时间 `voiceCareTime` 通过 `picker` 设置；  
   - “今天不再提醒”按钮：前端展示 Toast，后续可结合后端 `LastPushTime` 规则做更精细控制。

4. **风险随访提醒设置**
   - 开关 `followupEnabled` 控制是否启用风险随访提醒；  
   - 文案提示会说明“系统会根据风险等级与最近一次检测时间，适时推送随访提醒”。

5. **历史提醒入口**
   - “查看历史提醒”按钮跳转到消息中心页 `ReminderMessageList.vue`。

**与后端的交互：**

- `onLoad` 时：
  1. 调用 `/TPersonalLaryngealHealthRecord/Get` 获取当前用户的健康档案与风险等级，调整自查推荐频率；  
  2. 调用 `/TPersonalHealthRemind/List` 获取当前用户的所有提醒配置，并通过 `mapRecordToState` 映射到本地状态。
- 点击“保存设置”：
  - 构建自查、护嗓、随访三个 DTO（`buildCheckupDto` / `buildVoiceDto` / `buildFollowupDto`）；  
  - 依次调用 `/TPersonalHealthRemind/CreateOrEdit` 保存到后端；  
  - 成功后通过 `uni.showToast` 给出“已保存”提示。

#### 2.2.2 消息中心页（ReminderMessageList.vue）

**文件路径：**

- `HealthControl.uniapp/pages/Front/ReminderMessageList.vue`

**主要功能：**

1. **消息列表展示**
   - 顶部 `uni-nav-bar` 标题“健康提醒”；  
   - 支持标签筛选：全部 / 未读 / 自查 / 护嗓 / 随访；  
   - 每条消息以卡片方式展示：标题、类型标签、正文摘要、时间、未读红点。

2. **分页与上拉加载**
   - 通过 `Page`、`Limit` 分页加载消息；  
   - `scrolltolower` 和 `onReachBottom` 事件触发 `loadMore`，向后端请求下一页数据。

3. **消息统计**
   - 每次加载列表后，前端计算：  
     - `total`：当前已加载的消息总数；  
     - `read`：其中 `IsRead = true` 的数量；  
   - 在顶部统计卡片中展示“最近 30 天小结”的简版总数与已读数。

4. **单条已读与点击行为**
   - 点击单条消息时：  
     - 若 `IsRead == false`，先调用 `/MessageNotice/MarkRead` 标记为已读；  
     - 再根据 `ActionType` 判断跳转行为：  
       - `GOTO_AUDIO_DETECT` → 录音检测页；  
       - `GOTO_SYMPTOM_FORM` → 症状记录表单；  
       - `GOTO_HEALTH_RECORD` → 健康档案页；  
       - 其他/空 → 弹出文本弹窗展示提醒内容。

5. **全部已读**
   - 底部“全部标记为已读”按钮：  
     - 使用 `uni.showModal` 二次确认；  
     - 确认后调用 `/MessageNotice/MarkAllRead` 将该用户所有未读消息设为已读；  
     - 刷新列表并更新统计。

**与后端的交互：**

- 列表加载：调用 `/MessageNotice/List`，传入 `UserId`、`Page`、`Limit`、`IsRead` 或 `Type` 等过滤条件；
- 单条已读：`/MessageNotice/MarkRead`；  
- 全部已读：`/MessageNotice/MarkAllRead`。

## 三、运行方法

### 3.1 后端准备

1. **数据库脚本执行**
   - 确保已在目标数据库中执行以下脚本（按需修改库名）：  
     - `HealthControl.springboot/create_tpersonalhealthremind_table.sql`  
     - `HealthControl.springboot/create_messagenotice_table.sql`

2. **启动 Spring Boot 应用**
   - 在 `HealthControl.springboot` 目录下运行：  
     ```bash
     mvn spring-boot:run
     ```  
     或使用 IDE 直接启动主类。  
   - 确认 `WebApplication` 上已启用 `@EnableScheduling`，定时任务会自动生效。

3. **验证任务是否运行**
   - 查看控制台日志，确认应用启动无异常；  
   - 可通过手动调整系统时间或临时修改 Cron 表达式来快速验证任务是否触发。

### 3.2 前端准备

1. **依赖安装与 API 地址**
   - 在 `HealthControl.uniapp` 下执行：  
     ```bash
     npm install
     ```  
   - 检查 `utils/http.js` 中 `baseURL` 是否指向正确的后端地址。

2. **运行到浏览器或微信小程序**
   - H5 调试：  
     ```bash
     npm run dev:h5
     ```  
   - 微信小程序调试：  
     ```bash
     npm run dev:mp-weixin
     ```  
   - 使用 HBuilderX 也可以直接“运行到浏览器 / 小程序模拟器”。

3. **登录准备**

   - 使用现有账号或注册新账号登录小程序；  
   - 确保 `UserId` 在前端 `commonStore` 中能够正确获取，这样提醒配置和消息中心才能正常工作。

## 四、功能测试指南

### 4.1 提醒配置与保存测试（健康提醒页）

**测试步骤 1：首次进入健康提醒页**

1. 登录后，进入“个人中心”；  
2. 点击“健康提醒”卡片，进入 `HealthReminder.vue`；  
3. 预期结果：
   - 页面顶部显示“提醒总览”；  
   - 自查提醒、护嗓提醒、随访提醒卡片显示默认状态；  
   - 若用户已有健康档案，推荐文案会根据风险等级给出不同建议。

**测试步骤 2：修改提醒设置并保存**

1. 在“定期自查提醒”卡片中：  
   - 开启开关；  
   - 选择频率（如“每周一次”）；  
   - 设置提醒时间（如 `09:00`）。  
2. 在“护嗓提醒”卡片中：  
   - 开启开关；  
   - 设置提醒时间（如 `21:00`）。  
3. 在“就医/复查提醒”卡片中：  
   - 按需开启或关闭随访提醒。  
4. 点击底部“保存设置”。  
5. 预期结果：
   - 弹出“已保存”提示；  
   - 再次进入健康提醒页，设置保持不变；  
   - 数据库 `TPersonalHealthRemind` 表中可看到对应 `UserId` 的 0/1/4 三种 `RemindType` 记录。

   - `MessageNotice` 表中新增 `Type="VOICE_CARE"` 且 `UserId` 对应当前用户的记录；  
   - 对应护嗓提醒配置的 `LastPushTime` 已被更新为当前时间。

### 4.3 周期自查提醒任务验证（CHECKUP）

**准备条件：**

1. 当前用户至少有一条健康档案记录（可通过完成一次音频检测自动生成）；  
2. 健康档案中 `LastScreenTime` 已有值。

**测试步骤：**

1. 在数据库中手动调整某个用户的：  
   - `TPersonalHealthRemind` 中 `RemindType=0`（定期自查）的配置：  
     - 设置 `RemindStatus=1`；  
     - 设置合适的 `RepeatFrequency`（可简化为每天或每 7 天）；  
   - `TPersonalLaryngealHealthRecord.LastScreenTime` 为较久之前的日期，使 `当前日期 - LastScreenTime` 大于频率天数。  
2. 等待到第二天 00:30，或临时修改 `PeriodicCheckupReminderJob` 的 Cron 为短周期用于测试。  
3. 预期结果：
   - 在消息中心中出现类型为“自查提醒”的消息（CHECKUP）；  
   - 消息内容提示用户完成一次喉部自查，点击后可跳转至音频检测页面；  
   - 自查配置的 `LastPushTime` 被更新，短时间内不会再次推送。

### 4.4 风险随访提醒任务验证（FOLLOWUP）

## 五、注意事项与后续优化

1. **数据类型一致性**
   - `TPersonalHealthRemind.RemindType` 在数据库中为 `TINYINT(1)`，在实体中当前使用 `Boolean`，实际业务中按数值枚举使用时要注意转换；  
   - 如需更精确表达多种频率类型，建议将 `RepeatFrequency` 改为标准枚举 Int，并在映射时统一管理。

2. **Cron 表达式调整**
   - 生产环境中应避免过于频繁的任务执行；  
   - 测试环境可以临时缩短 Cron 周期以便快速验证功能。

3. **打扰控制与去重**
   - 当前通过 `LastPushTime` 控制日级别的去重；  
   - 若后续提醒种类增多，可在 `MessageNotice` 增加业务键（如 `BizKey`）用于更精细的去重与合并。

4. **埋点与效果评估**
   - 前端可在关键操作（打开提醒页、保存配置、点击消息、全部已读等）处新增埋点事件；  
   - 后端可根据 `MessageNotice` 表聚合发送量、阅读率、行为转化率等指标，为运营和产品优化提供数据支持。

5. **与健康档案的更深联动**
   - 当前已利用风险等级和末次自查时间驱动提醒；  
   - 后续可以结合症状记录频率、检测结果趋势等指标，对提醒频率和文案进行更智能的个性化调整。
