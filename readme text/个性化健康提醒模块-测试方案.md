# 个性化健康提醒模块 - 测试方案

## 一、测试环境准备

### 1.1 数据库准备
1. 确保已执行以下 SQL 脚本：
   - `create_messagenotice_table.sql`（创建消息通知表）
   - `create_tpersonalhealthremind_table.sql`（创建提醒配置表）
   - `alter_messagenotice_add_missing_fields.sql`（添加缺失字段）

2. 确认数据库表结构正确：
   ```sql
   -- 检查 MessageNotice 表字段
   DESC MessageNotice;
   -- 应包含：Id, Title, Content, Type, TargetKey, PlanSendTime, ActualSendTime, UserId, IsSend, IsSuccess, IsRead, ResultMsg, RelationNo, ActionType, ActionParam, CreationTime
   
   -- 检查 TPersonalHealthRemind 表字段
   DESC TPersonalHealthRemind;
   -- 应包含：Id, UserId, RemindType, RemindTime, RepeatFrequency, RemindStatus, LastPushTime, RemindContent, IsDelete, CreationTime, UpdateTime
   ```

### 1.2 后端服务准备
1. 确保后端 Spring Boot 服务已启动
2. 确认定时任务已启用（`@EnableScheduling`）
3. 检查定时任务 Cron 表达式（测试阶段）：
   - `PersonalHealthRemindJob`：每 1 分钟执行一次（`0 */1 * * * ?`）
   - `PeriodicCheckupReminderJob`：每 1 分钟执行一次（`0 */1 * * * ?`）
   - `RiskFollowupReminderJob`：每 2 分钟执行一次（`0 */2 * * * ?`）
   - `MessageNoticeJob`：每 10 秒执行一次（`0/10 * * * * ?`）

### 1.3 前端小程序准备
1. 确保小程序已编译并运行
2. 确保用户已登录（有有效的 `UserId`）

## 二、三种提醒类型测试方案

### 2.1 护嗓提醒（VOICE_CARE）测试

#### 测试目标
验证护嗓提醒能够按时生成并在历史提醒中显示。

#### 前置条件
1. 用户已登录
2. 数据库中已存在护嗓提醒配置：
   ```sql
   INSERT INTO TPersonalHealthRemind 
   (UserId, RemindType, RemindTime, RepeatFrequency, RemindStatus, IsDelete, CreationTime)
   VALUES 
   (你的用户ID, 1, 'HH:mm', 0, 1, 0, NOW());
   -- 注意：RemindTime 设置为当前时间的 "HH:mm" 格式，例如当前是 10:30，则设置为 '10:30'
   ```

#### 测试步骤
1. **配置提醒**：
   - 打开小程序，进入"健康提醒"页面
   - 切换到"护嗓提醒"标签
   - 开启"护嗓饮食/用声提醒"开关
   - 设置提醒时间为当前时间的后 1-2 分钟（例如当前是 10:30，设置为 10:31 或 10:32）
   - 保存配置

2. **等待触发**：
   - 等待 1-2 分钟，观察后端控制台日志
   - 应看到 `PersonalHealthRemindJob.generateReminderMessages()` 的执行日志
   - 检查是否有异常信息

3. **验证消息生成**：
   ```sql
   -- 查询是否生成了消息
   SELECT * FROM MessageNotice 
   WHERE UserId = 你的用户ID 
   AND Type = 'VOICE_CARE' 
   ORDER BY CreationTime DESC 
   LIMIT 5;
   ```
   - 应能看到新生成的 `Type='VOICE_CARE'` 的消息记录
   - `Title` 应为 "护嗓小贴士"
   - `IsSend` 应为 0（未发送）或 1（已发送）

4. **验证前端显示**：
   - 打开小程序，进入"健康提醒历史"页面（或"消息中心"）
   - 切换到"全部"标签，应能看到"护嗓小贴士"的提醒
   - 切换到"护嗓"标签，应只显示护嗓类型的提醒
   - 验证提醒内容、时间显示是否正确

#### 预期结果
- ✅ 定时任务正常执行，无异常日志
- ✅ 数据库中生成 `Type='VOICE_CARE'` 的消息记录
- ✅ 前端历史提醒列表中正确显示护嗓提醒
- ✅ 提醒标题为"护嗓小贴士"，类型标签显示"护嗓提醒"

### 2.2 定期自查提醒（CHECKUP）测试

#### 测试目标
验证定期自查提醒能够根据用户风险等级和上次自查时间正确生成，并在历史提醒中显示。

#### 前置条件
1. 用户已登录
2. 数据库中已存在定期自查提醒配置：
   ```sql
   INSERT INTO TPersonalHealthRemind 
   (UserId, RemindType, RemindTime, RepeatFrequency, RemindStatus, IsDelete, CreationTime)
   VALUES 
   (你的用户ID, 0, '10:00', 3, 1, 0, NOW());
   -- RemindType=0 表示定期自查
   -- RepeatFrequency=3 表示每30天一次（根据风险等级，低风险用户）
   ```

3. 用户有健康档案记录（可选，用于测试有历史记录的情况）：
   ```sql
   -- 如果有健康档案，确保 LastScreenTime 距离今天已超过 RepeatFrequency 对应的天数
   -- 例如 RepeatFrequency=3（30天），则 LastScreenTime 应设置为 30 天前或更早
   UPDATE TPersonalLaryngealHealthRecord 
   SET LastScreenTime = DATE_SUB(NOW(), INTERVAL 31 DAY)
   WHERE UserId = 你的用户ID;
   ```

#### 测试步骤
1. **配置提醒**：
   - 打开小程序，进入"健康提醒"页面
   - 开启"定期自查提醒"开关
   - 设置提醒频率（根据风险等级，低风险建议每30天一次）
   - 保存配置

2. **准备测试数据**（确保满足触发条件）：
   ```sql
   -- 方案1：用户没有健康档案（首次自查）
   -- 确保 TPersonalLaryngealHealthRecord 表中没有该用户的记录，或 IsDelete=1
   
   -- 方案2：用户有健康档案，但 LastScreenTime 已过期
   -- 将 LastScreenTime 设置为 30 天前（或更早）
   UPDATE TPersonalLaryngealHealthRecord 
   SET LastScreenTime = DATE_SUB(NOW(), INTERVAL 31 DAY),
       RiskAssessmentLevel = 0
   WHERE UserId = 你的用户ID;
   
   -- 方案3：清除 LastPushTime，允许重新触发
   UPDATE TPersonalHealthRemind 
   SET LastPushTime = NULL
   WHERE UserId = 你的用户ID AND RemindType = 0;
   ```

3. **等待触发**：
   - 等待 1-2 分钟，观察后端控制台日志
   - 应看到 `PeriodicCheckupReminderJob.generatePeriodicCheckupReminders()` 的执行日志
   - 检查是否有异常信息

4. **验证消息生成**：
   ```sql
   -- 查询是否生成了消息
   SELECT * FROM MessageNotice 
   WHERE UserId = 你的用户ID 
   AND Type = 'CHECKUP' 
   ORDER BY CreationTime DESC 
   LIMIT 5;
   ```
   - 应能看到新生成的 `Type='CHECKUP'` 的消息记录
   - `Title` 应为 "定期自查提醒"
   - `ActionType` 应为 "GOTO_AUDIO_DETECT"

5. **验证前端显示**：
   - 打开小程序，进入"健康提醒历史"页面
   - 切换到"全部"标签，应能看到"定期自查提醒"
   - 切换到"自查"标签，应只显示自查类型的提醒
   - 点击提醒，应能跳转到音频检测页面（`ActionType='GOTO_AUDIO_DETECT'`）

#### 预期结果
- ✅ 定时任务正常执行，无异常日志
- ✅ 数据库中生成 `Type='CHECKUP'` 的消息记录
- ✅ 前端历史提醒列表中正确显示定期自查提醒
- ✅ 提醒标题为"定期自查提醒"，类型标签显示"自查提醒"
- ✅ 点击提醒能正确跳转到音频检测页面

#### 前置条件
1. 用户已登录
2. 用户有健康档案，且风险等级为中风险（1）或高风险（2）：
   ```sql
   -- 确保用户有健康档案，且风险等级为中或高

   (UserId, RiskAssessmentLevel, IsDelete, CreationTime, LastScreenTime)
   VALUES 
   (你的用户ID, 1, 0, NOW(), DATE_SUB(NOW(), INTERVAL 15 DAY))
   ON DUPLICATE KEY UPDATE 
   RiskAssessmentLevel = 1,
   LastScreenTime = DATE_SUB(NOW(), INTERVAL 15 DAY);
   -- RiskAssessmentLevel: 0=低风险, 1=中风险, 2=高风险
   ```

3. 数据库中已存在随访提醒配置：
   ```sql
   INSERT INTO TPersonalHealthRemind 
   (UserId, RemindType, RemindTime, RepeatFrequency, RemindStatus, IsDelete, CreationTime)
   ```

## 四、常见问题排查

### 4.1 提醒未生成

**可能原因**：
1. 定时任务未执行（检查 Cron 表达式和 `@EnableScheduling`）
2. 提醒配置条件不满足（检查 `RemindStatus`、`IsDelete`、`RemindType`）
3. 业务逻辑条件不满足（如定期自查的 `LastScreenTime` 未过期）

**排查步骤**：
1. 检查后端控制台日志，查看定时任务是否执行
2. 检查数据库中的提醒配置：
   ```sql
   SELECT * FROM TPersonalHealthRemind WHERE UserId = 你的用户ID;
   ```
3. 检查业务逻辑条件（如健康档案的 `LastScreenTime`、风险等级等）

### 4.2 提醒未显示在历史列表中

**可能原因**：
1. 消息未生成（参考 4.1）
2. 前端查询条件错误（检查 `Type` 筛选）
3. 数据库字段缺失（检查是否执行了 `alter_messagenotice_add_missing_fields.sql`）

**排查步骤**：
1. 直接查询数据库：
   ```sql
   SELECT * FROM MessageNotice WHERE UserId = 你的用户ID ORDER BY CreationTime DESC;
   ```
2. 检查前端网络请求，查看 `/MessageNotice/List` 的返回数据
3. 检查前端筛选逻辑（`ReminderMessageList.vue` 的 `buildFilter` 方法）

### 4.3 定时任务执行异常

**可能原因**：
1. 数据库字段类型不匹配（`Boolean` vs `Integer`）
2. SQL 语法错误（字段不存在）
3. 实体类字段映射错误

**排查步骤**：
1. 查看后端控制台的完整异常堆栈
2. 检查实体类字段类型是否与数据库一致
3. 检查 SQL 脚本是否完整执行
