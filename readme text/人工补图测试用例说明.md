# 人工补图测试用例说明

这套测试用例是配合 [healthcontrol.sql](/c:/Users/徐国城/Desktop/居家喉部健康自查小程序/healthcontrol.sql) 使用的，目标是把“小程序里需要人工补图的落库图片位”做成一套可以直接写库、直接访问、前台能看到的可用夹具数据。

## 1. 已生成内容

- 测试图片目录：
  [HealthControl.springboot/external-resources/test-fixtures](/c:/Users/徐国城/Desktop/居家喉部健康自查小程序/HealthControl.springboot/external-resources/test-fixtures)
- 可重复生成脚本：
  [scripts/generate-test-fixture-images.ps1](/c:/Users/徐国城/Desktop/居家喉部健康自查小程序/scripts/generate-test-fixture-images.ps1)
- SQL 测试脚本：
  [人工补图测试用例.sql](/c:/Users/徐国城/Desktop/居家喉部健康自查小程序/人工补图测试用例.sql)

当前共生成 `48` 张 PNG 图片，全部落在后端静态目录映射下，数据库里统一使用：

`http://localhost:7245/test-fixtures/...`

## 2. 覆盖范围

本次覆盖了 6 类真正落库的人工补图位：

| 图片位 | 表 / 字段 | 测试策略 |
| --- | --- | --- |
| 用户头像 | `appuser.ImageUrls` | 更新现有用户 `Id=2/3/4` |
| 科普封面图 | `thealthscience.CoverUrl` | 新增测试文章 `Id=90001` |
| 科普正文插图 | `thealthscience.ScienceContent` | 在测试文章正文里内嵌两张 `<img>` |
| 饮食食物配图 | `tlaryngealfood.PicUrl` | 覆盖每个可见分类前 3 条示例食物，共 30 条 |
| 医院/医生配图 | `totolaryngologyhospitaldoctor.PicUrl` | 覆盖现有 `Id=1~10` 医生数据 |
| 喉镜照片 | `tlaryngoscopephoto.LaryngoscopePhotoUrl` | 插入 2 条独立测试记录 `Id=90001/90002` |

不包含：

- `AI 图谱上传页` 的临时图片，因为这一块当前不落固定业务表。

## 3. 执行顺序

1. 先保证后端以 `HealthControl.springboot` 为工作目录运行，端口仍是 `7245`。
2. 先导入 [healthcontrol.sql](/c:/Users/徐国城/Desktop/居家喉部健康自查小程序/healthcontrol.sql)。
3. 再执行 [人工补图测试用例.sql](/c:/Users/徐国城/Desktop/居家喉部健康自查小程序/人工补图测试用例.sql)。

如果你后续删掉了 `test-fixtures` 目录，可以重新执行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/generate-test-fixture-images.ps1
```

当前这版脚本优先重绘的是 `foods` 目录下的 30 张食物图，用来解决饮食卡片页“同底图仅换字”的问题。
头像、科普封面/正文图、医生图、喉镜图仍然保留在 `test-fixtures` 目录中，可直接配合 SQL 使用。

## 4. 关键说明

- 这批图片是“稳定可访问的测试夹具图”，不是通过上传接口随机生成目录，因此 URL 可重复、可直接写 SQL。
- SQL 脚本按“幂等”思路写的：
  更新类数据直接 `UPDATE`
  新增类数据使用 `INSERT ... ON DUPLICATE KEY UPDATE`
- 科普测试文章使用了独立主键 `90001`，喉镜测试记录使用了独立主键 `90001/90002`，方便后续识别和清理。
- 饮食和医生模块优先覆盖了前台最容易被展示到的示例数据，这样你执行完 SQL 后，进入列表页更容易直接看到测试图。
