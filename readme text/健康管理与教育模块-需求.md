## 健康管理与教育模块（科普 + 互动 + 风险问卷）- 详细需求文档（结合现有框架与数据库）

> 版本：v1.0  
> 编写日期：2026-03-05  
> 适用项目：居家喉部健康自查小程序（`HealthControl.uniapp` / `HealthControl.springboot` / `HealthControl.elementui` / `voice`）  
> 关联数据库：`healthcontrol`（见 `healthcontrol.sql`）  
> 现有后端基础：已存在 `THealthScience*`、`TScienceLike/Collect/Comment*`、`TRiskAssessmentQuestionnaire`、`TUserQuestionnaireAnswer` 的 CRUD Controller/Service/Mapper（SpringBoot）

## 一、模块定位与目标

### 1.1 模块定位

健康管理与教育模块围绕“**居家自查 + 科学护嗓 + 风险自评**”的闭环，提供：

- **知识科普**：内容分层组织、搜索与推荐，帮助用户理解喉部健康与自查意义；
- **用户互动**：点赞/收藏/评论（含楼中楼），强化参与感与内容传播；
- **风险问卷**：用户自评风险并得到解释与建议，结果同步到健康档案（`tpersonallaryngealhealthrecord.RiskAssessmentLevel`），用于后续提醒与随访。

### 1.2 业务目标（可量化）

- 科普模块：阅读转化（进入详情/列表曝光）、收藏率、评论率提升；
- 风险问卷：完成率、重复测评率、与风险提醒模块联动后的自查频次提升；
- 内容生态：普通用户贡献内容比例、审核通过率、违规内容拦截率。

## 二、范围与角色

### 2.1 用户角色

- **普通用户（RoleType=2）**：浏览科普、互动、发布科普/评论、参与风险问卷；
- **管理员（RoleType=1）**：内容与分类管理、审核、屏蔽违规、查看统计与导出；
- **（可选）医生/专家角色**：如后续新增角色，可用于“权威内容”标签与优先展示。

### 2.2 业务边界与约束

- 小程序真机网络：前端请求必须走 HTTPS 合法域名（已在 `HealthControl.uniapp/utils/config.js` 处理 baseUrl 切换）；
- 内容安全：必须预留审核与屏蔽机制，避免医疗误导与敏感信息传播；
- 现有代码结构：后端偏“通用 CRUD”风格（`/XXX/List|Get|CreateOrEdit|Delete|BatchDelete|Export`），需求将尽量复用该风格，并补充必要的“面向用户”的业务接口。

## 三、数据库现状与差距（基于 `healthcontrol.sql`）

### 3.1 现有表（已存在）

- 科普分类：`thealthsciencecategory`
- 科普文章：`thealthscience`（`ScienceContent` 目前为 `varchar(128)`，不足以承载富文本）
- 互动：
  - 点赞：`tsciencelike`
  - 收藏：`tsciencecollect`
  - 评论：`tsciencecomment`（支持 `ParentCommentId` 自引用）
  - 评论点赞：`tsciencecommentlike`
- 风险问卷：
  - 问卷主表：`triskassessmentquestionnaire`（仅元信息，无题目明细）
  - 用户答题：`tuserquestionnaireanswer`（固定维度字段：吸烟/饮酒/职业用嗓/反酸/其他，适合“固定问卷”，不支持多问卷题目配置）

### 3.2 主要差距（为了满足你提出的目标能力）

- **科普内容富文本**：需要 `LONGTEXT`/`TEXT` 承载完整内容（含图片/段落/列表等）；现表 `ScienceContent varchar(128)` 不足。
- **审核流程**：现表缺少文章审核状态、审核人、审核时间、驳回原因等字段；评论审核也缺少标准状态枚举与原因字段。
- **多级分类**：现分类表无 `ParentId`，无法构建“多级分类管理（声带保护/疾病科普/护喉贴士/发音训练 → 子类）”。
- **互动防重复**：点赞/收藏表建议加唯一键（`UserId + ScienceId` / `UserId + CommentId`），避免重复记录导致统计失真。
- **问卷题目体系**：若要“管理员/用户创建标准化问卷 + 题目/选项 + 自动评分”，需要新增问卷题目/选项表，或在答题表增加 JSON 存储。

> 本需求文档会给出**单独 SQL 脚本**用于新增/改表（不直接改 `healthcontrol.sql`）。

#### 4.2.5 发布科普（必须）

发布/编辑字段（对齐 `thealthscience` 与本次改表脚本）：

- **Title**：标题（必填，5–40 字）
- **CoverUrl**：封面（必填，建议 16:9）
- **CategoryId**：分类（必填；支持二级分类选择）
- **KnowledgeType**：知识类型（必填）
- **ScienceContent**：正文（富文本，必填）
- **Summary**：摘要（可选；为空则自动截取正文前 120 字）

发布动作：

- 保存草稿：`AuditStatus=0`
- 提交审核：`AuditStatus=1`

内容规范（必须在发布页提示）：

- 禁止夸大疗效、替代诊断、诱导就医/药品购买
- 禁止泄露隐私（手机号/身份证/病历图片等）
- 免责声明：科普仅供参考，建议以医生诊断为准

#### 4.2.5.1 发布页 UI 布局（必须）

- **顶部**：标题“发布科普”，右侧按钮“保存草稿”
- **表单区域（卡片化）**：
  - 标题输入（计数器）
  - 分类选择（树选择器：显示“一级/二级”）
  - 知识类型选择（radio）
  - 封面上传（单图，支持裁剪可选）
  - 摘要输入（可选，120 字提示）
  - 正文富文本编辑器
    - 图片上传调用 `/File/BatchUpload`
    - 图片插入后默认宽度 100%
- **底部按钮**：
  - 主按钮：提交审核
  - 次按钮：预览（本地渲染富文本，确认排版）
- **提交校验**：
  - 必填项缺失时，定位到对应控件并提示
  - 正文最小长度建议 ≥ 200 字（可配置）

#### 4.2.3 科普发布与审核（普通用户 + 管理员）

普通用户发布流程：

- 新建草稿 → 编辑（标题/封面/分类/类型/正文） → 提交审核 → 等待审核结果
- 驳回后可修改重新提交；通过后可再次编辑，但编辑将触发“重新审核”（可配置）。

审核状态（建议）：

- 0=草稿
- 1=待审核
- 2=通过（可展示）
- 3=驳回
- 4=下架/屏蔽（管理员操作）

文案与体验要求：

- 以“居家护嗓、温和陪伴”为基调，不出现恐吓式内容；
- 发布页增加“医学免责声明”与“内容规范提示”。

### 4.3 科普社交互动

#### 4.3.1 点赞

- 点赞/取消点赞可反复操作，需记录时间（`LikeTime` / `CancelLikeTime`）
- 防重复：同一用户对同一文章最多 1 条有效点赞记录（建议唯一键）
- 点赞数可通过统计计算，也可落字段缓存（需求建议：先统计，后续再做缓存）

#### 4.3.2 收藏

- 收藏/取消收藏，记录时间（`CollectTime` / `CancelCollectTime`）
- 防重复：同一用户对同一文章最多 1 条有效收藏记录（建议唯一键）

#### 4.3.3 评论（一级 + 楼中楼）

- 一级评论：`ParentCommentId = NULL`
- 楼中楼：`ParentCommentId = 上级评论 Id`（现表已支持自引用）
- 建议补充：
  - `ReplyToUserId`（被回复用户，便于提醒与展示“回复 XX”）
  - `RootCommentId`（主楼 Id，便于一次性拉取树）
- 评论审核：
  - 普通用户评论默认进入“待审核”或“敏感词命中待审核”
  - 管理员/系统可自动通过
- 评论点赞：同点赞逻辑（`tsciencecommentlike`）

#### 4.3.4 评论树与分页策略（必须，防止性能问题）

- **主楼分页**：`Page/Limit` 只分页 `ParentCommentId IS NULL` 的主楼评论
- **楼中楼加载**：
  - 默认每个主楼只展示最近 2 条回复
  - 点击“展开更多回复”再请求子评论（`ParentCommentId=主楼Id`）
- **展示字段**：评论内容、用户昵称、时间、是否为作者、是否已点赞、审核状态提示

#### 4.3.5 互动防刷与去重（建议）

- 点赞/收藏：
  - 后端做 `UserId + ScienceId` 唯一约束
  - 重复操作直接返回成功（幂等）
- 评论：
  - 同一用户对同一文章 10 秒内最多发 1 条（限流，防刷）
  - 命中敏感词进入“待审核”

### 4.4 风险评估问卷体系

#### 4.4.1 问卷列表与推荐

- 展示可用问卷（`ShowStatus=展示`，未删除）
- 推荐策略（先简单后复杂）：
  - 默认置顶“标准风险自评问卷”
  - 若用户最近 30 天未测评，提示“建议每月测一次”

#### 4.4.1.1 问卷入口页 UI（`QuestionnaireHome`）（必须）

- **顶部**：标题“风险自评”
- **顶部结果摘要卡**：
  - 最近一次得分、风险等级 tag、测评时间
  - CTA：查看详情 / 再测一次
  - 若无历史：展示“暂无测评，建议完成一次自评” + CTA“开始测评”
- **问卷列表卡**：
  - 标准问卷置顶（默认）
  - 每条展示：标题、描述、题目数、预计耗时（如 1–2 分钟）

#### 4.4.2 用户答题与自动评分（0–100）

评分要求：

- 提交后系统自动计算总分（0–100）
- 生成：
  - 分数解释（分段说明）
  - 风险等级（低/中/高）
  - 个性化建议（建议自查频率、就医提醒等）

建议的等级映射（可配置）：

- 0–34：低风险（RiskAssessmentLevel=0）
- 35–64：中风险（RiskAssessmentLevel=1）
- 65–100：高风险（RiskAssessmentLevel=2）

#### 4.4.2.1 题目体系（两种模式，MVP 先做模式 A）

- **模式 A（兼容现表，MVP）**：固定维度问卷
  - 使用 `tuserquestionnaireanswer` 现有字段（SmokingHistory/DrinkingHistory/OccupationalVoice/AcidReflexSymptom/OtherRiskFactor）
  - 对应你提出的“问卷维度”，最快上线
- **模式 B（可配置问卷，增强）**：
  - 使用新增表 `triskassessmentquestion` / `triskassessmentquestionoption`
  - 用户答案存 `tuserquestionnaireanswer.AnswerJson`
  - 计分明细存 `ScoreDetailJson`

#### 4.4.2.2 默认评分规则（建议，可在管理端配置）

- SmokingHistory=是：+25
- DrinkingHistory=是：+15
- OccupationalVoice=是：+20
- AcidReflexSymptom=是：+20
- OtherRiskFactor 非空：MVP 固定 +10（增强版可按选项/关键字映射 0～20）
- 总分封顶 100

#### 4.4.2.3 答题页 UI（`QuestionnaireAnswer`）（必须）

- **进度条**：显示“已完成 X / 总题数”
- **题目卡片**：
  - 题干 + 说明（可选）
  - 选项（单选/多选/是非/输入）
- **交互**：
  - 必答题未作答，点击下一步提示
  - 允许返回上一题修改
- **提交确认**：
  - 提交前弹窗提示“测评结果会同步到健康档案并影响提醒推荐”

#### 4.4.2.4 结果页 UI（`QuestionnaireResult`）（必须）

- **结果卡**：
  - 得分（大号数字）
  - 风险等级（彩色 tag）
  - 温和解释（分段说明）
- **建议卡**：
  - 护嗓建议（3–5 条）
  - 推荐下一步：
    - 去做一次音频自查
    - 去设置个性化提醒（跳转 `HealthReminder`）
    - 查看科普合集（跳转 `ScienceList` 带 KnowledgeType/CategoryId）

#### 4.4.3 结果同步到健康档案

提交问卷后：

- 若用户不存在 `tpersonallaryngealhealthrecord`：自动创建一条档案（最小字段集），写入 `RiskAssessmentLevel`
- 若存在：更新 `RiskAssessmentLevel`（并记录更新时间/来源）
- 同步后的风险等级用于：
  - 个性化健康提醒模块（自查/随访提醒频率推荐）
  - 科普推荐（优先推送风险对应的内容集合）

同步策略（必须明确）：

- 若用户已手动配置过提醒：只更新风险等级，不强制覆盖用户提醒配置
- 若风险从低→中/高：允许生成 1 条系统提示（复用 `messagenotice`，Type=FOLLOWUP/ANNOUNCEMENT）

#### 4.4.4 历史记录与重复测评

- 用户可查看历史答题记录（按时间倒序）
- 支持重复测评，保留多次记录，不覆盖旧记录

## 六、接口需求（SpringBoot：复用现有 CRUD + 补充用户侧业务接口）

### 6.1 复用现有 CRUD（已存在）

### 6.2 建议新增/补充的用户侧接口（面向小程序前台）

为避免把“管理端 CRUD”直接暴露给用户端，建议新增 **Front API**（可新 Controller 前缀 `/Front`，或在现 Controller 下新增 `Public*` 方法）。

科普：

- `POST /Front/Science/CategoryList`：仅返回 `ShowStatus=1 && IsDelete=0` 的分类，按 `SortNum` 排序，支持树结构
- `POST /Front/Science/List`：仅返回审核通过且可见的文章；支持 `CategoryId/Keyword/KnowledgeType/Sort`
- `POST /Front/Science/Detail`：文章详情 + 用户对该文章是否已赞/藏 + 评论统计
- `POST /Front/Science/Read`：阅读上报（去重策略），用于阅读量/热度
- `POST /Front/Science/LikeToggle`：点赞/取消
- `POST /Front/Science/CollectToggle`：收藏/取消
- `POST /Front/Science/CommentCreate`：发表评论/回复
- `POST /Front/Science/CommentList`：按文章拉取评论树（分页：主楼分页 + 子评论按需展开）
- `POST /Front/Science/MyPublishList`：我的发布（按审核状态）

问卷：

- `POST /Front/Questionnaire/List`：可用问卷列表
- `POST /Front/Questionnaire/Detail`：问卷详情 + 题目（若启用题目体系）
- `POST /Front/Questionnaire/Submit`：提交答题，返回分数/等级/建议，并同步健康档案
- `POST /Front/Questionnaire/MyHistory`：历史记录列表
- `POST /Front/Questionnaire/LatestResult`：最近一次测评结果摘要（用于个人中心/首页展示）

#### 6.3 接口入参/回参约定（与现有项目保持一致）

- **统一返回结构**：`{ Code, Success, Msg, Data }`
- **分页结构（List）**：`Data.Items + Data.TotalCount`
- **幂等要求**：Like/Collect Toggle 必须幂等（重复点击不应报错）

### 6.4 Front API 详细入参/回参（必须落地，可直接开发）

> 说明：统一采用 `POST`，并复用 `{ Code, Success, Msg, Data }` 结构。下面只列“前台必要字段”，避免把管理端 CRUD 暴露给用户端。

#### 6.4.1 `POST /Front/Science/CategoryList`

- **用途**：前台分类树（一级 tab + 二级网格）
- **过滤**：`ShowStatus=1` 且 `IsDelete=0`
- **入参（Data）**：
  - `IncludeEmpty`（可选）：是否返回无文章分类（默认 true）
- **回参（Data）**：
  - `Items[]`（树结构）：
    - `Id/CategoryName/CategoryDesc/SortNum`
    - `ParentId/Level/Path`（执行 `alter_thealthsciencecategory_add_hierarchy.sql` 后）
    - `Children[]`

#### 6.4.2 `POST /Front/Science/List`

- **用途**：科普列表分页（首页推荐流/列表页）
- **过滤（必须）**：
  - 文章：`AuditStatus=2 && ShowStatus=1 && IsDelete != 1`（兼容历史 NULL）
  - 分类：`ShowStatus=1 && IsDelete=0`
- **排序（Sort）**：
  - `RECOMMEND`：`IsTop desc, RecommendWeight desc, PublishTime desc`
  - `LATEST`：`PublishTime desc`
  - `HOT_7D/HOT_30D`（增强）：未实现热度分时可先用 `ReadCount desc` 兜底
- **入参（Data）**：
  - `Page`（必填）
  - `Limit`（必填）
  - `CategoryId`（可选：二级分类 Id；若传一级分类需包含子类）
  - `Keyword`（可选：匹配 `Title/Summary`）
  - `KnowledgeType`（可选：1/2/3/4/9）
  - `Sort`（可选：RECOMMEND/LATEST/HOT_7D/HOT_30D）
- **回参（Data）**：
  - `Items[]`（列表卡片字段）：
    - `Id/Title/CoverUrl/Summary`
    - `CategoryId`（可选：`CategoryName`）
    - `KnowledgeType/PublishTime`
    - `ReadCount/LikeCount/CollectCount/CommentCount`
    - `IsTop`（可选）
  - `TotalCount`

#### 6.4.3 `POST /Front/Science/Detail`

- **用途**：详情页展示 + 当前用户状态（是否已赞/藏）+ 评论统计
- **过滤（必须）**：同 `/Front/Science/List`（前台不可看未通过/下架内容；作者查看草稿/待审走“我的发布详情”）
- **入参（Data）**：
  - `ScienceId`（必填）
- **回参（Data）**：
  - `Science`：
    - `Id/Title/CoverUrl/ScienceContent/Summary`
    - `CategoryId/KnowledgeType`
    - `CreatorId`（可选：`CreatorName`）
    - `PublishTime`
    - `ReadCount/LikeCount/CollectCount/CommentCount`
  - `UserState`：
    - `IsLiked/IsCollected`
  - `CommentStat`：
    - `TotalCount`

#### 6.4.4 `POST /Front/Science/Read`

- **用途**：阅读上报（去重 + 阅读量/热度）
- **去重（必须）**：
  - 同一用户同一篇文章 **30 分钟窗口只记 1 次**
  - 未登录：用 `ClientKey + ScienceId + WindowStartTime` 去重
- **入参（Data）**：
  - `ScienceId`（必填）
  - `ClientKey`（可选：未登录时必填，前端本地生成并持久化）
  - `ReadScene`（可选：HOME/LIST/SHARE）
  - `ReadDurationSec`（可选：离开页面上报）
- **回参（Data）**：
  - `ReadCount`
  - `Deduped`（是否被去重）

#### 6.4.5 `POST /Front/Science/LikeToggle`（幂等）

- **幂等**：重复点赞/取消都返回成功并保持最终状态
- **入参（Data）**：
  - `ScienceId`（必填）
  - `Action`（可选：LIKE/UNLIKE；不传则 toggle）
- **回参（Data）**：
  - `IsLiked`
  - `LikeCount`

#### 6.4.6 `POST /Front/Science/CollectToggle`（幂等）

- **入参/回参**：同 LikeToggle（字段：`IsCollected/CollectCount`）

#### 6.4.7 `POST /Front/Science/CommentCreate`

- **审核策略（必须）**：
  - 普通用户默认 `AuditStatus=0(待审)`；管理员可自动通过 `AuditStatus=1`
  - 命中敏感词/高风险用户：强制待审（增强）
- **入参（Data）**：
  - `ScienceId`（必填）
  - `ParentCommentId`（可选：回复时传）
  - `RootCommentId`（可选：回复主楼时传主楼 Id；不传则后端计算）
  - `ReplyToUserId`（可选：回复目标用户）
  - `CommentContent`（必填：1–200 字）
- **回参（Data）**：
  - `CommentId`
  - `AuditStatus`
  - `MsgTip`（例如“已提交，审核通过后对他人可见”）

#### 6.4.8 `POST /Front/Science/CommentList`

- **入参（Data）**：
  - `ScienceId`（必填）
  - `Page/Limit`（必填：主楼分页）
  - `ParentCommentId`（可选：传主楼 Id 时拉子评论）
  - `PreviewLimit`（可选：子评论预览条数，默认 2）
- **过滤（前台）**：
  - 对他人：`AuditStatus=1 && IsDelete != 1`（兼容历史 NULL）
  - 对自己：允许看到自己 `AuditStatus=0/2`，但需标识状态
- **回参（Data）**：
  - `Items[]`：
    - `Id/ScienceId/ParentCommentId/RootCommentId/ReplyToUserId`
    - `UserId/UserName/UserAvatar`
    - `CommentContent/CreationTime/AuditStatus`
    - `LikeCount`（可选）
    - `IsLiked`（可选）
    - `ReplyCount`（主楼子回复数）
  - `TotalCount`

#### 6.4.9 `POST /Front/Science/MyPublishList`

- **入参**：`Page/Limit`，可选 `AuditStatus`
- **回参**：列表字段 + `RejectReason`（驳回原因）

#### 6.4.10 问卷提交与档案同步（必须明确）

- **关键要求**：
  - 写入答题记录（保留历史，不覆盖旧记录）
  - 同步更新档案 `RiskAssessmentLevel`
  - 不强制覆盖用户已配置的提醒（只做引导提示/跳转）

## 十二、实现细节与注意事项（不新增功能点，按此即可开发）

### 12.1 前台可见性/权限控制（必须）

- **文章前台可见**：`AuditStatus=2 && ShowStatus=1 && IsDelete != 1`
- **作者本人可见**：
  - 在“我的发布”可看到 `AuditStatus=0/1/2/3/4` 全部状态
  - 进入详情（我的发布详情）允许展示草稿/待审/驳回，但必须带明显状态条
- **评论前台可见**：
  - 他人可见：`AuditStatus=1 && IsDelete != 1`
  - 自己可见：自己发布的评论允许看到 `AuditStatus=0/2`（显示“审核中/已驳回”提示）

### 12.2 点赞/收藏幂等实现（必须）

> 依赖 `alter_science_interaction_add_unique_indexes.sql` 的唯一键保证不重复插入，后端实现只需做到“重复操作直接返回成功”。

- **点赞**：
  - LIKE：若存在记录（`UserId+ScienceId`）则更新 `LikeTime`、`IsDelete=0`；不存在则插入
  - UNLIKE：存在则更新 `CancelLikeTime`、`IsDelete=1`；不存在也返回成功
- **收藏**：同点赞逻辑

### 12.3 阅读去重与阅读量更新（必须）

- **必须支持 30 分钟窗口去重**（防刷/防刷新）
- **落地建议**：使用 `tsciencereadlog`（见 `create_tsciencereadlog_table.sql`）
  - 写入 readlog 成功（未触发唯一键冲突）才 `thealthscience.ReadCount + 1`
  - 触发去重则 `Deduped=true`，不累加阅读量

### 12.4 评论树实现细节（必须）

- **RootCommentId 规则**：
  - 主楼：`RootCommentId = Id`
  - 子楼：`RootCommentId = 主楼 Id`
- **查询策略**：
  - 主楼分页：`ParentCommentId IS NULL`（或 0）按时间倒序
  - 子楼展开：`ParentCommentId = 主楼Id` 按时间正序（更像聊天）
- **ReplyToUserId 用途**：
  - 前端展示“回复 @XX”
  - 生成消息通知（复用 `MessageNotice`，ActionType 可扩展为 `GOTO_SCIENCE_DETAIL`）

### 12.5 评论审核闭环（必须）

- **状态建议**：0=待审，1=通过，2=驳回，3=屏蔽
- **审核通过**：写 `AuditStatus=1`，写 `AuditorId/AuditTime`，清空 `RejectReason`
- **驳回**：写 `AuditStatus=2`，写 `RejectReason`（用户端可见）
- **屏蔽**：写 `AuditStatus=3`（可写 `AuditRemark` 内部说明）

### 12.6 举报与内容安全（若你已纳入范围，则按此实现；否则可暂不做）

- **举报对象**：文章/评论
- **处理结果**：
  - 仅记录（不做动作）
  - 触发屏蔽（评论 AuditStatus=3 或文章 AuditStatus=4）
  - 对作者发送通知（复用 `MessageNotice`）

### 12.7 搜索实现注意事项（若你已纳入范围，则按此实现；否则可暂不做）

- **搜索历史**：前端可缓存最近 10 条；后端 `tsciencesearchlog` 用于运营分析
- **热词**：按近 7 天关键词聚合，过滤敏感词与过短关键词（1 个字）

### 12.8 推荐实现注意事项（若你已纳入范围，则按此实现；否则可暂不做）

- **基础推荐（P1）**：同分类 + 同 KnowledgeType + 最新优先
- **风险相关推荐（P1）**：若用户风险中/高，优先推荐“就医指南/随访”与“疾病科普”
- **专题合集（P2）**：运营配置专题页（后续可新增表或用文章 IsTop+Weight 组合实现）

### 12.9 后端实现总原则（SpringBoot + MyBatisPlus，必须）

- **Controller 分层**：
  - 管理端 CRUD：保留现有 `/THealthScience*`、`/TScience*` 风格
  - 前台接口：新增 `FrontScienceController`、`FrontQuestionnaireController`（或统一 `Front/*` 前缀），避免直接暴露管理端 CRUD
- **鉴权与用户获取**：

  - 未登录允许浏览：仅 `CategoryList/List/Detail/Read/CommentList`（Read 需 ClientKey）；点赞/收藏/发布/评论需要登录

### 12.10 `/Front/Science/CategoryList` 实现细节（必须）

- **查询**：

### 12.11 `/Front/Science/List` 实现细节（必须）

- **过滤条件（必须一致）**：
  - `AuditStatus=2`
  - `ShowStatus=1`
  - `IsDelete != 1`
- **CategoryId 处理**：
  - 若传的是一级分类：需要包含其所有二级分类的文章
    - 实现方式：先取子分类 Id 列表，再 `in(CategoryId, ids)`
- **Keyword 处理**：
  - `trim` + 长度限制（建议 1～40）
  - 仅允许模糊匹配：`Title like %kw% OR Summary like %kw%`
- **分页**：使用 MyBatisPlus `Page<T>` + `IPage`
- **回参 Summary**：
  - 若 `Summary` 为空：后端从 `ScienceContent` 截取前 120（去掉 HTML 标签可选，MVP 可不去标签）
- **计数字段**：
  - 若执行过 `alter_thealthscience_upgrade_content_and_audit.sql` 并启用冗余计数（Like/Collect/CommentCount）：列表直接返回字段
  - 若未启用：统一返回 0 或通过聚合接口补齐（需求建议：先冗余，列表性能更稳）

### 12.12 `/Front/Science/Detail` 实现细节（必须）

- **查询**：按同 List 的可见性过滤，拿到文章主体
- **用户状态**：
  - IsLiked：`tsciencelike` 查询 `UserId + ScienceId` 且 `IsDelete != 1`
  - IsCollected：`tsciencecollect` 同理
- **注意**：Detail 不直接返回评论树（避免首屏太重），由 CommentList 单独拉取

### 12.13 `/Front/Science/Read` 实现细节（必须）

- **窗口时间计算（30 分钟）**：
  - 例：当前时间 `t`，`WindowStartTime = floorTo30Min(t)`（如 10:05→10:00；10:35→10:30）
- **写入策略**：
  - 登录用户：插入 `tsciencereadlog(UserId,ScienceId,WindowStartTime,...)`
  - 未登录：插入 `tsciencereadlog(ClientKey,ScienceId,WindowStartTime,...)`（ClientKey 必填）
  - 唯一键冲突：视为去重命中，返回 `Deduped=true`
- **阅读量更新**：
  - 仅在写 log 成功时：`UPDATE thealthscience SET ReadCount = IFNULL(ReadCount,0)+1 WHERE Id=?`
  - 注意：`ReadCount` 可能为 NULL，必须 `IFNULL`

### 12.14 点赞/收藏 Toggle 的“可实现代码逻辑”（必须）

- **核心**：依赖唯一键（UserId+ScienceId）保证不会插入多条有效记录
- **实现建议（MySQL 兼容）**：
  - 第一步：查是否存在记录（包含 IsDelete=1 的历史）
  - 第二步：
    - 不存在：insert（IsDelete=0，写 LikeTime/CollectTime）
    - 存在：
      - LIKE/COLLECT：update IsDelete=0，写 LikeTime/CollectTime
      - UNLIKE/UNCOLLECT：update IsDelete=1，写 CancelLikeTime/CancelCollectTime
  - 第三步（可选：若启用冗余计数）：
    - 点赞成功变为“已点赞”：`LikeCount + 1`
    - 取消变为“未点赞”：`LikeCount - 1`（下限 0）
  - 上述建议置于同一个事务，防止并发双击导致计数错乱

### 12.15 评论创建与楼中楼字段落地（必须）

- **CommentCreate 入参校验**：
  - `CommentContent`：trim、长度 1～200
  - 回复时：
    - `ParentCommentId` 必须存在且属于同一 `ScienceId`
    - `RootCommentId` 为空则后端自动计算：
      - 若 parent 为主楼：root=parent.Id
      - 若 parent 为子楼：root=parent.RootCommentId
- **RootCommentId 写法**：
  - 主楼：先 insert，再 update `RootCommentId=Id`（或 insert 时先空，落库后补写）
  - 子楼：insert 时直接写 `RootCommentId`
- **审核状态**：
  - 普通用户：默认 `AuditStatus=0`
  - 管理员：可直接 `AuditStatus=1`（按现系统角色判断）

### 12.16 评论列表与分页（必须）

- **主楼分页**：
  - 条件：`ScienceId=? AND (ParentCommentId IS NULL OR ParentCommentId=0) AND IsDelete!=1`
  - 对他人：`AuditStatus=1`
  - 对自己：额外 OR 自己的 `UserId=currentUserId`（并返回状态文案）

  - 对自己：同上允许自己可见
- **ReplyCount**：
  - 推荐后端聚合：`count(*) where RootCommentId=主楼Id and IsDelete!=1 and AuditStatus=1`
