## 居家喉部健康自查小程序｜后续功能迭代路线图（推荐优先级）

> 版本：v1.0  
> 编写日期：2026-03-07  
> 适用项目：`HealthControl.uniapp` / `HealthControl.springboot` / `HealthControl.elementui`  
> 参考：同目录 `饮食管理与推荐模块-需求.md`、数据库结构 `healthcontrol.sql`  

---

## 0. 写在前面：为什么要做“路线图”

你的小程序目前已经具备“自查 → 档案 → 症状/问卷/科普/饮食/提醒”的多数能力，但要真正成为“可行动的健康服务”，下一阶段应优先补齐：

- **风险出现后用户下一步怎么做**（就医建议、就诊准备、报告导出、可追溯通知）
- **服务的持续性**（消息通知中心、个性化提醒闭环）
- **数据闭环与产品护城河**（确诊标注与授权、可解释可审计的模型优化迭代）

本路线图以“**用户价值最大化 + 闭环优先 + 性价比**”为原则，给出推荐的逐步实现顺序。

## 1. 当前项目能力盘点（用于确定下一步缺口）

### 1.1 已实现（用户侧 UniApp）

- **音频自查与结果**：记录、列表、结果页  
- **健康档案**：档案页、症状日志（新增/列表/详情）  
- **科普与互动**：科普分类/列表/详情/发布、点赞收藏、评论/回复  
- **风险问卷**：问卷列表/答题/结果/历史  
- **饮食模块**：饮食推荐首页、食物列表/详情、饮食记录（含日期范围筛选、友好/忌口筛选）  
- **健康提醒配置**：提醒设置页、历史提醒入口（消息列表页已有但需要进一步加强闭环）  

### 1.2 已具备（后端 SpringBoot + 管理端 ElementUI）

从项目结构与 `healthcontrol.sql` 可见，以下模块的表结构、控制器、管理端页面都已经存在：

- **就医辅助**：`TOtolaryngologyHospitalDoctor*`（医院医生库）、`TUserMedicalRecommend*`（就医推荐记录）、`TUserMedicalPrepareReport*`（就诊准备报告）  
- **报告导出**：`TScreenReportExportRecord*`（导出记录）  
- **个性化提醒/通知**：`TPersonalHealthRemind*`、`MessageNotice*` + Job 定时任务  
- **隐私设置**：`TUserPrivacySetting*`  
- **模型优化标注**：`TModelOptimizeLabel*`  

> 结论：下一阶段最“划算”的工作，是把这些**已存在的后端/表/管理端能力**真正做成用户侧可用的闭环页面与统一前台聚合接口（`/Front/*`）。

## 2. 推荐的迭代优先级（P0 → P1 → P2）

### 2.1 P0（最优先）：就医辅助闭环（强用户价值 + 与风险强关联）

#### 2.1.1 目标

当用户出现中高风险或“恶性倾向初筛”时，给到明确的下一步行动：

- 看什么科室、去哪家医院、如何预约  
- 一键生成“就诊准备报告”，把关键数据整理给医生  
- 导出/分享留痕，形成可追溯的健康管理链路  

#### 2.1.2 涉及表（均已存在）

- `totolaryngologyhospitaldoctor`：医院/医生资源库  
- `tusermedicalrecommend`：系统对用户的就医推荐记录（含依据、查看状态）  
- `tusermedicalpreparereport`：用户就诊准备报告（生成状态、文档链接、统计）  
- `tscreenreportexportrecord`：导出记录（导出格式、用途、文件URL）  

#### 2.1.3 用户侧（UniApp）建议页面与交互

- **(新增) 就医建议首页** `pages/Front/MedicalHome.vue`（建议）
  - 展示：当前风险等级、最近一次自查结果摘要、推荐动作（就医建议/生成报告/查看医院）
  - CTA：`查看推荐医生`、`生成就诊准备报告`、`查看历史报告/导出记录`

- **(新增) 医院医生列表** `pages/Front/HospitalDoctorList.vue`
  - 筛选：地区、科室类型、关键字（医院/医生）
  - 操作：一键拨号、复制地址、打开预约链接（外链或复制）

- **(新增) 就医推荐详情** `pages/Front/MedicalRecommendDetail.vue`
  - 展示：推荐原因（风险等级/初筛结果/症状）、推荐医生卡片、最近一次检测时间
  - 行为：标记已查看、收藏（可选）

- **(新增) 就诊准备报告** `pages/Front/MedicalPrepareReport.vue`
  - 选择：简版/详版（TemplateType）
  - 生成：展示生成状态（待生成/生成中/完成/失败 + 失败原因）
  - 完成后：支持查看/下载/分享（并写入导出记录）

> 注意：为降低工作量，可先做“报告生成后返回 URL（PDF/图片）”，前端用 `web-view` 或图片预览展示。

#### 2.1.4 后端（SpringBoot）推荐补齐：统一前台聚合接口 `/Front/Medical/*`

虽然已有 CRUD Controller，但前台更需要“聚合好的业务接口”。建议新增（或在现有基础上封装）：

- `POST /Front/Medical/Home`
  - 返回：当前用户风险概览、最新检测摘要、是否有推荐、最近报告状态、推荐医院/医生Top1

- `POST /Front/Medical/DoctorList`
  - 入参：`Region(可选)`, `DepartmentType(可选)`, `Keyword(可选)`, `Page, Limit`

- `POST /Front/Medical/RecommendList`
  - 入参：`Page, Limit`
  - 返回：按时间倒序的就医推荐记录（可带医生信息）

- `POST /Front/Medical/RecommendMarkViewed`
  - 入参：`RecommendId`

- `POST /Front/Medical/PrepareReportCreate`
  - 入参：`TemplateType(0/1)`, `ContentScope(可选)`
  - 行为：创建生成任务记录（写入 `tusermedicalpreparereport`），同步或异步生成文档

- `POST /Front/Medical/PrepareReportList`
  - 入参：分页

- `POST /Front/Medical/ExportRecordCreate`
  - 入参：`DetectId/ReportId`, `ExportFormat`, `ExportPurpose`
  - 行为：写入 `tscreenreportexportrecord`

#### 2.1.5 管理端（ElementUI）建议增强（可选，但性价比高）

- 医院医生库：支持批量上架/下架、地区/科室枚举统一
- 就医推荐规则（若后续需要）：可在管理端配置“风险等级/初筛结果 → 推荐科室/地区/医生优先级”

#### 2.1.6 验收标准（P0）

- 中高风险用户能在小程序内看到**明确就医建议入口**  
- 能看到**医院/医生列表**并完成拨号/预约链接操作  
- 能生成**就诊准备报告**并在完成后打开预览  
- 导出/分享行为能写入导出记录（后台可查）

---

### 2.2 P1：消息通知中心（把提醒变成可追溯服务）

#### 2.2.1 目标

把“提醒配置页”升级为“**通知中心 + 行为闭环**”：

- 什么时候提醒了什么？用户是否看过？点了后去了哪里？  
- 可用于运营/医疗合规审计（不夸大、不恐吓、有据可查）

#### 2.2.2 涉及表

- `messagenotice`：通知记录（已读/发送状态、ActionType/Param）
- `tpersonalhealthremind`：提醒配置（时间、频率、开关、最后推送时间）

#### 2.2.3 用户侧（UniApp）建议

- **通知列表增强** `pages/Front/ReminderMessageList.vue`
  - 分组/筛选：未读/已读、类型（CHECKUP/VOICE_CARE/FOLLOWUP）
  - 操作：一键标记已读、清理（软删除）
  - 点击跳转：依据 `ActionType/ActionParam` 跳转到自查/档案/饮食/就医建议等页面

- **提醒设置增强** `pages/Front/HealthReminder.vue`
  - “今天不再提醒”建议真正落库（以日期窗口控制）
  - 根据风险等级给出推荐频率提示（已做基础计算，可进一步把推荐值写入提示文案）

#### 2.2.4 后端建议

在 `MessageNotice` 相关接口上补齐：

- `MarkRead`、`BatchMarkRead`、`ListByUser`、`SoftDelete`  
- Job 推送策略：结合 `LastPushTime` 和用户风险等级，控制推送频率  

#### 2.2.5 验收标准（P1）

- 通知列表能正确展示未读/已读  
- 点击通知能跳到对应业务页面  
- 已读状态可追溯，管理员可在后台审计

### 2.3 P2：模型优化标注闭环（产品长期护城河）

#### 2.3.1 目标

把“自查结果”与“真实确诊”连接起来，形成数据闭环：

自查 → 就医 → 确诊 → 标注/授权 → 模型优化 → 更准自查

#### 2.3.2 涉及表

- `tmodeloptimizelabel`：确诊结果标注与授权状态（已存在）
- 可结合 `taudioscreenrecord` 关联 DetectId

#### 2.3.3 用户侧（UniApp）建议

新增入口建议放在：

- 自查结果页 `AudioResult.vue`：增加“我已去医院检查/我有确诊结果”按钮  
- 用户中心：增加“确诊标注与数据授权”入口  

标注内容建议：

- 医院确诊结果（文本/枚举）
- 是否与模型判断一致（误报/确诊）
- 上传/补充说明
- 授权匿名用于模型优化（需与隐私设置联动）

#### 2.3.4 验收标准（P2）

- 用户能提交一次确诊标注  
- 管理端可审核/查看标注数据  
- 授权状态可追溯，未授权数据不进入训练集（原则层面先落地，后续再接训练管线）

---

### 2.4 P2：隐私设置与用户数据权利（合规与信任）

#### 2.4.1 目标

让用户明确知道：

- 哪些数据会被保存、哪些会用于匿名优化  
- 可以随时关闭授权、导出/删除个人数据（至少提供申请与记录）

#### 2.4.2 涉及表

- `tuserprivacysetting`：本地存储开关、匿名授权开关、协议同意状态（已存在）

#### 2.4.3 用户侧建议（UniApp）

新增 `pages/Front/PrivacySetting.vue`：

- 本地存储开关（音频/报告缓存）
- 匿名授权开关（用于模型优化）
- 隐私协议状态展示
- 数据导出/删除说明入口（MVP 可先做“提交申请/工单”）

## 3. 推荐的“实现顺序”（避免返工的最佳路径）

### 3.1 第一阶段（1～2 个迭代周期）：P0 就医辅助闭环

- 先把医院医生库在小程序端做出来（列表+筛选+拨号/预约）
- 再做“我的就医建议”（推荐列表/详情 + 标记已查看）
- 再做“就诊准备报告”（生成 + 状态 + 预览/导出记录）

### 3.2 第二阶段（1 个迭代周期）：P1 通知中心

- 通知列表增强 + 跳转闭环
- 提醒“今天不再提醒”落库 + Job 策略微调

### 3.3 第三阶段（长期迭代）：P2 数据闭环与合规能力

- 确诊标注（含授权）  
- 隐私设置完善（含导出/删除路径）  
- 后续再考虑：更复杂的统计看板、规则引擎、A/B 实验与埋点

## 5. 建议你现在就定下来的 3 个产品原则（减少后续决策成本）

- **闭环优先**：每个新模块上线时至少满足“入口 → 操作 → 可回看/可追溯”
- **风险分层**：低/中/高风险看到的内容深度不同，高风险要更强调就医与复查
- **可解释与克制**：所有建议都要能解释“为什么”，且必须带免责声明，避免制造焦虑
