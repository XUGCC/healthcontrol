## 就医辅助模块（医院医生库 + 就医推荐 + 就诊准备报告 + 导出与通知）需求文档

> 版本：v1.0  
> 编写日期：2026-03-07  
> 适用项目：居家喉部健康自查小程序（`HealthControl.uniapp` / `HealthControl.springboot` / `HealthControl.elementui`）  
> 关联数据库：`healthcontrol`（见 `healthcontrol.sql` 中 `totolaryngologyhospitaldoctor`、`tusermedicalrecommend`、`tusermedicalpreparereport`、`tscreenreportexportrecord`、`messagenotice` 等）  
> 前置能力：音频自查（`taudioscreenrecord`）、健康档案（`tpersonallaryngealhealthrecord`）、症状日志（`tsymptomlog`）、饮食记录（`tuserdietrecord`）已存在  

---

## 一、模块定位与目标

### 1.1 模块定位

就医辅助模块面向“自查后需要行动”的关键阶段，为用户提供：

- **医疗资源库**：可查询正规医院/医生信息（地区、科室、电话、预约链接等）；
- **风险驱动就医建议**：基于风险等级、初筛结果、症状等，生成“建议就医/复查”的推荐记录；
- **就诊准备报告**：一键把用户近一段时间关键数据整理成“简版/详版”报告，方便就诊沟通；
- **导出与可追溯**：报告查看/导出/分享行为留痕（便于运营评估与用户自我回溯）；
- **通知联动**：中高风险触发随访提醒或就医提示，点击可直达“就医建议/报告”。

补充说明（增强目标）：

- **更少焦虑、更可执行**：给“下一步做什么”清晰路径（挂号/就诊准备清单/带什么资料/何时复查）
- **复诊与长期随访**：支持把“就医建议/报告”做成可回看的“医疗沟通档案”
- **更强可追溯与合规**：建议/报告都要可解释、有依据、有免责声明，导出/分享留痕

### 1.2 业务目标（可量化）

- **行动转化**：中高风险用户中，进入就医辅助模块的比例 ≥ 40%  
- **报告使用**：生成就诊准备报告的用户中，查看/导出（或分享）比例 ≥ 30%  
- **服务闭环**：从“风险出现→通知→就医建议→报告生成→导出记录”全链路可追溯  

### 1.3 与其他模块的关系

- 与 **音频自查模块**：使用 `taudioscreenrecord.PrimaryScreenResult/Confidence` 作为建议就医的重要依据  
- 与 **健康档案/症状日志**：使用 `tpersonallaryngealhealthrecord.RiskAssessmentLevel` 与 `tsymptomlog` 作为风险分层与报告内容输入  
- 与 **饮食模块**：报告可包含最近 7 天饮食概况（友好/忌口次数）作为辅助信息（不做医疗结论）  
- 与 **提醒/通知模块**：`messagenotice.ActionType` 引导用户进入就医建议/报告页

## 二、范围与角色

### 2.1 用户角色

- **普通用户（RoleType=2）**
  - 浏览医院/医生信息，进行拨号/打开预约链接
  - 查看系统就医建议（历史记录、已查看状态）
  - 生成/查看/导出就诊准备报告

- **管理员（RoleType=1）**
  - 维护医院/医生资源库（新增/编辑/上下架/软删除）
  - 查看用户就医推荐记录、就诊准备报告记录、导出记录（运营/审计）
  - （可选）配置规则/策略（后续 P2）

### 2.2 业务边界与约束

- **不替代诊断**：任何“就医建议/报告”都必须带免责声明（仅供参考，不能替代医生诊断）
- **不承诺治疗结果**：不出现“确诊”“治愈”等承诺性文案
- **合规展示**：医院/医生信息需真实可核验；预约链接为外部链接时需提示“将离开小程序”
- **隐私最小化**：报告生成仅使用本系统已存在的用户数据；分享/导出需用户确认

强约束（务必落地）：

- **风险分层展示**：低/中/高风险看到的内容“深度不同”，高风险要突出“尽快就医/复查”，但文案克制、不恐吓
- **外链提示**：预约链接/地图导航属于外部能力，必须提示并二次确认
- **异常兜底**：医院电话为空/链接不可用/无推荐记录/无档案，都必须有可操作兜底（例如“去完善档案/去查医院库”）

## 三、数据库现状与差距（基于 `healthcontrol.sql`）

### 3.1 相关表（已存在）

- **医院医生库** `totolaryngologyhospitaldoctor`
  - `HospitalName/DoctorName/DepartmentType/Region/Address/ContactPhone/AppointmentUrl`
  - `ShowStatus/IsDelete`

- **用户就医推荐记录** `tusermedicalrecommend`
  - `UserId/DetectId/RiskLevel/RecommendHospitalId/RecommendTime/ViewStatus/IsDelete`

- **用户就诊准备报告** `tusermedicalpreparereport`
  - `UserId/HealthRecordId/DetectIdList/TemplateType/ContentScope`
  - `StandardDocUrl/GenerateStatus/GenerateFailReason`
  - `ViewCount/DownloadCount/ShareCount/IsDelete`

- **报告导出记录** `tscreenreportexportrecord`
  - `UserId/DetectId/ExportTime/ExportFormat/ExportFileUrl/ExportPurpose/IsDelete`

- **通知表** `messagenotice`
  - `Type/ActionType/ActionParam/IsRead/PlanSendTime/ActualSendTime/UserId`

### 3.2 差距与建议

- 目前表已足够做 MVP；若后续要做更强规则引擎（风险 + 初筛 + 症状 → 推荐医生策略），可新增“就医推荐规则表”（P2）。
- `tusermedicalpreparereport` 已有 `StandardDocUrl`，MVP 可先用“HTML 页面 URL / 图片 URL / PDF URL”任一形式承载。

建议补齐但可分期（不阻塞 MVP）：

- 医院医生库建议补充：地图定位（经纬度）、医院等级/标签、擅长方向、门诊时间等（提升筛选与可信度）
- 就医推荐建议补充：推荐原因快照（风险/检测/症状/规则命中），避免“推荐生成后数据变了导致解释对不上”
- 导出记录建议补充：关联 ReportId（而不仅 DetectId），便于审计与回放
- 报告建议补充：内容快照（JSON/HTML），避免“只存 URL 导致资源失效”

## 四、小程序端（UniApp）功能需求

> 目标：把就医辅助做成“用户能用、能回看、能导出”的闭环模块。

### 4.1 信息架构与入口

建议至少提供 3 个入口（按优先级）：

- **入口 1：自查结果页**（`AudioResult`）  
  - 当检测结果/风险等级达到阈值时，展示“就医建议”卡片与按钮  
- **入口 2：健康档案页**（`HealthRecord`）  
  - 档案概览区增加“就医建议/就诊准备报告”入口  
- **入口 3：个人中心**（`UserCenter`）  
  - 增加“就医辅助”入口（统一承载历史推荐/报告/导出记录）

建议新增页面（命名示例）：

- `pages/Front/MedicalHome.vue`：就医辅助首页（概览 + CTA）
- `pages/Front/HospitalDoctorList.vue`：医院/医生列表（筛选 + 拨号/预约）
- `pages/Front/MedicalRecommendList.vue`：我的就医建议（历史列表）
- `pages/Front/MedicalRecommendDetail.vue`：就医建议详情（原因解释 + 推荐医生卡片）
- `pages/Front/MedicalPrepareReport.vue`：就诊准备报告（生成 + 状态 + 预览 + 导出）
- `pages/Front/MedicalExportRecordList.vue`：（可选）导出记录列表

建议新增（更实用、且成本不高）：

- `pages/Front/HospitalDoctorDetail.vue`：（可选）医生详情页（更好承载地图导航/收藏/就诊准备清单入口）
- `pages/Front/MedicalGuide.vue`：（可选）就医指南页（挂号/就诊当天流程/何时急诊）

### 4.2 MedicalHome（就医辅助首页）

#### 4.2.1 页面目标

给用户 10 秒内明确：

- 我现在是否需要关注就医/复查？
- 系统为什么这么建议？
- 下一步可以做什么（看推荐医生 / 生成报告 / 查看医院库）？

#### 4.2.2 页面结构（建议）

- 顶部：`uni-nav-bar` 标题“就医辅助”
- 卡片 1：**当前风险概览**
  - 风险等级（低/中/高）
  - 最近一次检测摘要（良性/恶性倾向/未知 + 时间）
  - 最近症状（如有）
  - 温和免责声明
- 卡片 2：**系统就医建议（如有）**
  - 最新一条推荐摘要（推荐医院/医生、依据）
  - CTA：查看详情
- 卡片 3：**就诊准备报告**
  - 最近一份报告状态（待生成/生成中/已完成/失败）
  - CTA：生成新报告 / 查看历史报告
- 卡片 4：**医院医生库入口**
  - CTA：查看医院医生库（列表筛选）

建议增强（让首页更“闭环”）：

- 卡片 5：**就诊准备清单（可复用固定模板）**
  - 展示：需要携带的资料（最近报告/既往检查/用药清单/症状记录）
  - CTA：一键生成清单（保存到报告或本地）
- 卡片 6：**导出/分享记录（可选）**
  - 展示：最近一次导出/分享时间与用途（就诊/家属沟通/自留）

### 4.3 HospitalDoctorList（医院/医生列表）

#### 4.3.1 功能

- 列表字段（卡片式）：
  - 医院名称、医生姓名
  - 科室类型（0/1/2 显示：耳鼻喉科/咽喉科/头颈外科）
  - 地区（Region）
  - 电话（ContactPhone）
  - 预约链接（AppointmentUrl）
- 筛选：
  - 地区（关键字匹配 Region）
  - 科室类型（下拉）
  - 关键词（匹配医院/医生）
- 分页：下拉刷新/上拉加载

#### 4.3.2 交互

- 点击电话：`uni.makePhoneCall`
- 点击预约：弹窗确认后 `openURL`（或复制链接）
- 点击条目：进入详情弹窗/页（MVP 可直接在列表卡片展示完整信息）

建议增强交互（强实用）：

- **地图导航**：点击地址 → 打开地图（`uni.openLocation`），需要经纬度；若无经纬度则复制地址
- **收藏**：在列表卡片右上角展示收藏按钮（星标），便于用户建立“就医资源库”
- **快速筛选**：
  - 常用城市快捷入口（本地缓存最近选择）
  - 只看“可预约链接/有电话”的筛选项（避免空数据）

#### 4.3.3 前台可见性

只展示：

- `ShowStatus=1`
- `IsDelete != 1`（兼容 NULL）

### 4.4 MedicalRecommendList / Detail（就医建议）

#### 4.4.1 列表

- 默认展示近 30/60 天内的记录（可分页）
- 展示字段：
  - 推荐时间、风险等级、是否已查看
  - 推荐医生/医院摘要（若存在）
- 支持筛选：未查看/已查看

建议增强（更符合真实使用）：

- 支持“置顶/收藏推荐”（用于用户对某条建议反复查看）
- 支持“反馈按钮”（轻量）：`我已就医` / `暂不需要`（用于后续通知频率/策略优化；MVP 可先只记录不联动）

#### 4.4.2 详情页

展示“为什么建议就医”（可解释）：

- 风险依据：
  - 风险等级（来自档案或推荐记录 RiskLevel）
  - 最近检测（DetectId → `taudioscreenrecord` 摘要）
  - 最近症状（如有）
- 推荐医生卡片：
  - 医院、医生、科室、地区、电话、预约链接
- 操作：
  - 标记为已查看（进入详情即标记）
  - 一键拨号/预约

建议增强内容（更可解释、更可执行）：

- **紧急情况提示**（固定文案，非个性化）：如“呼吸困难、吞咽明显困难、咯血”等建议急诊就医
- **就诊准备建议**：基于模板展示“建议向医生描述的要点”（症状开始时间、加重因素、用嗓情况等）
- **相关科普入口**：跳转到“就医与随访”分类（你已有科普模块的结构）

#### 4.4.3 文案规则（必须）

- 高风险：强调“建议尽快线下就医/复查”
- 中风险：强调“建议近期复查/关注症状变化”
- 低风险：强调“如持续不适或出现加重，建议就医咨询”

### 4.5 MedicalPrepareReport（就诊准备报告）

#### 4.5.1 报告类型

- **简版（TemplateType=0）**：适合门诊快速沟通（1 页/少量模块）
- **详版（TemplateType=1）**：包含更多历史记录与解释（2～4 页）

#### 4.5.2 报告内容范围（ContentScope 约定）

MVP 推荐默认范围（可配置）：

- 自查摘要：最近 1～3 次 `taudioscreenrecord`（时间、结果、置信度、关键声学参数）
- 档案摘要：`tpersonallaryngealhealthrecord` 风险等级、趋势标签、末次自查时间
- 症状摘要：最近 1～5 条 `tsymptomlog`（类型、程度、持续时间）
- 问卷摘要：最近一次 `tuserquestionnaireanswer` 结果摘要（如已接入）
- 饮食摘要（可选）：最近 7 天 `tuserdietrecord` 友好/忌口次数（不下医疗结论）
- 免责声明与就医建议（温和表述）

建议新增“就诊沟通模板”（强实用，且可复用固定模板）：

- 主诉模板：症状类型/持续时间/诱因/缓解因素
- 重要时间线：最近一次自查/症状首次出现/近期加重节点
- 生活方式摘要：用嗓强度、烟酒、反酸相关（来自问卷/档案）
- “我希望医生帮我确认的问题”清单（用户可勾选/编辑）

#### 4.5.3 报告生成流程（状态机）

报告表 `tusermedicalpreparereport.GenerateStatus`：

- 0=待生成  
- 1=生成中  
- 2=已完成（StandardDocUrl 可用）  
- 3=生成失败（GenerateFailReason 有值）  

前端交互：

- 点击“生成报告”：
  - 创建报告记录（状态=待生成/生成中）
  - 页面显示生成中动画与说明（预计 5～20 秒）
  - 定时轮询/手动刷新状态
- 生成完成：
  - 打开预览（web-view / 图片预览 / pdf 预览）
  - 提供“下载/分享”按钮（触发导出记录 + 计数累加）
- 生成失败：
  - 展示失败原因（友好文案）与“重试”按钮

建议增强（更可靠）：

- 生成中提供“后台生成，不影响退出页面”的提示；回到页面可继续查看状态
- 若 `StandardDocUrl` 为空但状态=已完成，需兜底提示“资源生成异常，请重试/联系管理员”
- 提供“选择数据范围”的入口（增强版）：
  - 检测：最近 1 次/最近 3 次/最近 30 天
  - 症状：最近 5 条/最近 30 天
  - 饮食：最近 7 天/近 30 天（仅摘要）

#### 4.5.4 导出/分享与计数

- 查看：`ViewCount + 1`
- 下载：`DownloadCount + 1`（并写入导出记录）
- 分享：`ShareCount + 1`（并写入导出记录，或单独统计）

> MVP 可只做“查看 + 下载/分享中的一种”，但必须至少有一种行为可追溯。

建议增强“导出用途”选择（强实用）：

- 导出前弹窗选择用途：`就诊` / `家属沟通` / `自留备份` / `其他`
- 导出格式：`PDF` / `图片`（MVP 可先做一种）

## 五、后端接口设计（SpringBoot）

### 5.1 设计原则

- **前台业务接口使用 `/Front/Medical/*` 聚合**，避免直接暴露管理端 CRUD
- 统一返回：`{ Code, Success, Msg, Data }`
- 分页返回：`Data.Items + Data.TotalCount`
- 软删除字段统一兼容：`IsDelete != 1`（包括历史 NULL）

补充原则：

- **解释一致性**：推荐/报告的“依据”尽量采用快照字段（否则数据变化后解释会漂移）
- **幂等与防重复**：报告生成、导出记录创建要支持幂等（用户重复点击不应产生多条无意义记录）
- **权限与越权防护**：所有用户数据接口必须校验 `UserId`，禁止访问他人 Recommend/Report/ExportRecord

### 5.2 前台接口清单（建议）

#### 5.2.1 `POST /Front/Medical/Home`

- **用途**：就医辅助首页聚合数据
- **返回 Data**：
  - `RiskLevel`（0/1/2，来自档案或计算）
  - `LatestDetect`（时间、初筛结果、置信度）
  - `LatestSymptom`（类型/程度/持续）
  - `LatestRecommend`（可选：推荐摘要）
  - `LatestReport`（可选：报告状态 + URL）

#### 5.2.2 `POST /Front/Medical/DoctorList`

- 入参：`Page, Limit, Region(可选), DepartmentType(可选), Keyword(可选)`
- 返回：医院医生列表（仅 `ShowStatus=1 && IsDelete!=1`）

建议新增：`POST /Front/Medical/DoctorDetail`

- 入参：`DoctorId`
- 返回：医生详情（含地址/电话/预约链接/标签/经纬度等）

#### 5.2.3 `POST /Front/Medical/RecommendList`

- 入参：`Page, Limit, ViewStatus(可选)`
- 返回：当前用户的就医推荐记录列表（可带医生信息）

#### 5.2.4 `POST /Front/Medical/RecommendDetail`

- 入参：`RecommendId`
- 返回：推荐详情 + 关联医生信息 + 依据摘要（风险/检测/症状）

#### 5.2.5 `POST /Front/Medical/RecommendMarkViewed`

- 入参：`RecommendId`
- 行为：更新 `tusermedicalrecommend.ViewStatus`

建议新增：`POST /Front/Medical/RecommendFeedback`

- 入参：`RecommendId, FeedbackType(已就医/暂不需要/信息不准等), Remark(可选)`
- 行为：记录用户反馈（用于策略优化与通知频率，MVP 可先落库不联动）

#### 5.2.6 `POST /Front/Medical/PrepareReportCreate`

- 入参：
  - `TemplateType`（0/1）
  - `DetectIdList`（可选：不传则默认取最近 1～3 次）
  - `ContentScope`（可选：字符串约定）
- 行为：

## 十三、业务规则与推荐策略（让“推荐”稳定、可解释、可控）

### 13.1 推荐触发时机（建议）

- **触发点 A（强推荐）**：用户完成一次音频自查后（`taudioscreenrecord` 新增记录）
  - 若档案风险等级（`tpersonallaryngealhealthrecord.RiskAssessmentLevel`）为中/高，或本次初筛结果为恶性倾向（`PrimaryScreenResult=1`），则生成一条 `tusermedicalrecommend`
- **触发点 B（弱推荐）**：每日定时任务（Job）扫描中高风险用户
  - 若用户近 N 天未查看/未反馈，则生成“随访/复查提醒”（`messagenotice`）并引导进入就医辅助

### 13.2 推荐去重与频控（必须）

- **同一用户**在一个时间窗口内（如 7 天）最多生成 1 条系统推荐
- **同一次 DetectId** 只生成 1 条推荐（强烈建议加唯一约束，见第 16 章）
- 若用户反馈“已就医”，默认 7～30 天内不再生成系统推荐（可配置）
- 若初筛结果从良性→恶性倾向（或风险显著升级），允许突破窗口去重（更安全）

### 13.3 推荐内容选择（MVP 简化方案）

- **地区优先**：优先命中用户最近选择/常用地区（可先由前端缓存城市传参）
- **科室映射（最小可用）**：
  - 默认：耳鼻喉科（0）
  - 若包含“吞咽困难/颈部肿块/恶性倾向/高风险”信号：优先头颈外科（2）或咽喉科（1）
- **兜底**：无地区匹配则返回“热门/综合”资源

### 13.4 推荐解释文本生成（强烈建议）

建议后端生成 `RecommendReason`（或快照字段），模板示例：

- “依据：风险等级=中；最近一次自查=恶性倾向初筛；近 7 天症状=吞咽不适/咽痛（2 条）。建议尽快到 XX 科室线下就医评估。”
