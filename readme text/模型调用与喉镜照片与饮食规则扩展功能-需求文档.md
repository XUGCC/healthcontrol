> 版本：v1.0  
> 编写日期：2026-03-12  
> 适用项目：居家喉部健康自查小程序（`HealthControl.uniapp` / `HealthControl.springboot` / `HealthControl.elementui`）  
> 关联数据库：`healthcontrol`（`tmodelinterfacecalllog`、`tdietrecommendrule`、`tlaryngoscopephoto`）  
> 依赖模块：音频自查模块、健康档案模块、饮食管理与推荐模块、就医辅助模块、模型优化标注闭环模块

## 一、整体目标与思路

- **当前问题**：`tmodelinterfacecalllog`、`tdietrecommendrule`、`tlaryngoscopephoto` 三张表已建但未形成用户可感知的功能或运营工具。
- **总体目标**：在不大改数据库结构的前提下，基于这三张表补齐几块高价值能力：
  - **模型检测服务透明度与稳定性反馈**（基于 `tmodelinterfacecalllog`）  
  - **喉镜影像档案与对照记录**（基于 `tlaryngoscopephoto`）  
  - **饮食规则驱动的“推荐解释 + 场景卡片”增强**（强化使用 `tdietrecommendrule`）  
- **设计原则**：
  - 尽量利用现有表结构，优先通过接口与前端展示实现价值；
  - 与现有模块解耦为可独立上线的小功能块（方便分批实现）；
  - 注意文案合规与用户情绪，所有结论均“非诊断，仅作参考”。

## 二、模型检测服务透明度与稳定性反馈（基于 tmodelinterfacecalllog）

### 2.1 模块定位

- 面向用户的小功能：在每次音频自查结束后，用简单的可视化方式反馈“本次模型检测服务是否顺畅稳定”，避免用户把网络/接口异常误认为“检测结果很差”。  
- 面向运营/技术的管理端能力：在后台查看模型接口调用日志，快速发现错误高发的链路和时间段，为后续本地网络模型/算法服务优化提供依据（如参数调整、超时策略、预处理流程优化等）。

### 2.2 数据基础（tmodelinterfacecalllog）

- 关键字段回顾：
  - `DetectId`：关联 `taudioscreenrecord`，一条检测可能对应多条模型调用（多阶段、多模型）。
  - `CallLink`：调用环节（如预处理、声纹模型、主诊断模型等，可在代码中约定枚举）。
  - `ModelInterfaceType`：接口类型（HTTP、本地推理、第三方服务等，可在代码中约定）。
  - `InputSpectrumUrl`、`InputParams`：输入图谱和参数快照。
  - `CallCost`：耗时（毫秒）。
  - `ResultSummary`、`ResultComplete`：结果摘要与完整返回（可为 JSON 字符串）。
  - `CallStatus` / `FailErrorCode` / `FailDetailInfo`：成功/失败与错误详情。

> 说明：本需求文档默认不修改该表结构，通过约定和接口解析完成展示与统计。

### 2.3 小程序端能力：本次检测“模型服务状态”展示

#### 2.3.1 页面嵌入位置

- 页面：`pages/Front/AudioResult.vue`（音频自查结果页，已存在）
- 位置：检测结果卡片下方，新增一个“模型服务状态”信息块

#### 2.3.2 展示内容

- **本次服务概览卡片**
  - 标题：**“模型服务状态”**
  - 状态徽标（根据本次关联的日志综合判断）：
    - 正常：`本次检测模型服务运行顺畅`
    - 轻微异常：`本次检测出现轻微波动，结果已按规则兜底处理`
    - 失败重试：`检测过程中出现错误，已为你重试/切换备用模型`
  - 简要说明（从日志聚合生成）：
    - 成功调用次数（成功日志条数）
    - 失败调用次数（失败日志条数）
    - 总耗时范围（`CallCost` 的平均值/区间）

- **“查看详细调用信息”折叠面板（高级用户可点开）**
  - 按 `CallLink` 分组，展示每个环节：
    - 环节名称（由前端根据枚举映射，如“预处理”“声纹分析”“主诊断模型”等）
    - 耗时：`CallCost`（ms）
    - 状态：成功/失败（失败显示“错误码 + 简短错误信息”）
  - 安全说明文案（固定）：  
    - “以上为系统内部模型服务调用信息，仅用于技术透明展示，不代表医疗结论。”

#### 2.3.3 业务规则

1. 日志查询：
   - 根据 `DetectId` 从后端查询 `tmodelinterfacecalllog` 关联记录（仅 `IsDelete IS NULL/0`）。
   - 若无日志记录，则隐藏“模型服务状态”模块（或展示“暂无内部日志记录”）。
2. 状态计算建议（后端统一计算）：
   - 若所有环节 `CallStatus=1`，且平均 `CallCost` 在合理范围内 → 状态=正常。
   - 若存在失败但最终有成功结果 → 状态=失败重试，前端提示“系统已为你自动重试”。
   - 若所有关键环节失败且没有有效检测结果 → 前端整体检测流程本身应已处理，此处仅展示“此次模型服务出现异常，请稍后重试或检查网络”。
3. 异常兜底与备用策略（约定说明）：
   - 当某些环节调用失败，但最终仍产生了检测结果（例如通过自动重试或备用模型完成推断）时：
     - 在聚合结果中将本次检测标记为“有兜底”的状态（例如 `StatusLevel = WARNING`）；
     - 在前端提示文案中增加温和说明：
       - 示例：“本次检测过程中出现过短暂异常，结果已按照系统规则做了兜底处理，仅供参考，建议结合症状持续观察或复查。”
   - 后端建议策略（不影响前端接口约定）：
     - 同一环节调用失败时，可自动重试 1–2 次，再决定失败；
     - 若主模型多次失败，可根据 `ServiceName/ModelVersion` 切换至轻量级备用模型，结果在 `ResultSummary/ResultComplete` 中保留来源信息；
     - 对于由备用模型产生的结果，统一将 `QualityLevel` 设为“轻微异常”或更高等级，便于前端做差异化展示。
3. 合规要求：
   - 不在此处显示模型版本号、第三方服务名称等敏感信息（如需，可在管理端单独展示）。

### 2.4 管理端能力：模型接口调用日志看板

#### 2.4.1 页面与路由

- 路由建议：`/Admin/ModelInterfaceCallLogList.vue`
- 入口：`HealthControl.elementui` 管理端“系统运维 / 模型日志”菜单下新增“模型接口调用日志”

#### 2.4.2 功能点

- 日志列表：
  - 查询条件：
    - 时间范围（默认近 7 天）
    - `CallStatus`（全部/成功/失败）
    - `CallLink`（下拉枚举）
    - `ModelInterfaceType`
    - `DetectId` / `UserId`（通过 join `taudioscreenrecord` 查询）
  - 列表字段：
    - 调用时间（`CreationTime`）
    - 环节、接口类型
    - 耗时（ms）
    - 状态 + 错误码
    - 结果摘要（`ResultSummary`）
    - 操作：查看详情
- 详情弹窗：
  - 展示 `InputParams`、`InputSpectrumUrl`、`ResultComplete`（长文本可折叠）
  - 用于技术排查问题

#### 2.4.3 后续可拓展（P2）

- 对失败率、平均耗时做图表可视化（按日、按环节分布），用于性能监控；
- 结合 `tmodeloptimizelabel` 评估不同模型版本下的误报/漏报情况（需要在结果 JSON 中携带模型版本号）。

## 三、喉镜影像档案与对照记录（基于 tlaryngoscopephoto）

### 3.1 模块定位

- 让用户可以在小程序中 **上传并管理自己的喉镜检查照片**，并与音频自查记录、确诊标注关联，形成更完整的“喉部影像 + 声音 + 诊断”档案。
- 为后续模型优化（如图像模型）打基础，但当前阶段只作为用户侧管理和记录功能，不做自动图像诊断。

### 3.2 数据基础（tlaryngoscopephoto）

- 核心字段：
  - `UserId`：关联用户。
  - `DetectId`：可选关联一次音频自查记录（如“与 2026-03-01 的检测结果配对”）。
  - `LaryngoscopePhotoUrl`：喉镜照片 URL（上传后生成）。
  - `PhotoDesc`：照片描述（医院名称、检查类型、简单备注等均可由前端引导用户填写）。
  - `UploadTime`：上传时间。
  - `AuditStatus`：审核状态（可由运营后台人工审核敏感图片等）。

> 说明：现有字段已能满足“上传、描述、关联、审核”基础需求，本阶段不强制改表。

### 3.3 小程序端：喉镜影像管理入口与页面

#### 3.3.1 信息架构与入口

- 建议新增页面（命名示例）：
  - `pages/Front/LaryngoscopePhotoList.vue`：喉镜影像列表/档案页；
  - `pages/Front/LaryngoscopePhotoEdit.vue`：单条喉镜影像上传/编辑页。
- 入口建议：
  - 入口 1：`UserCenter.vue` 中新增菜单项“喉镜影像档案”；
  - 入口 2：`HealthRecord.vue`（个人喉部健康档案）中增加“喉镜照片”卡片；
  - 入口 3：检测结果页 `AudioResult.vue`，在“我已去医院检查/提交确诊结果”附近增加“上传喉镜照片”按钮，引导与本次检测关联。

#### 3.3.2 LaryngoscopePhotoList（列表页）

- 功能：
  - 列出当前用户所有未删除且审核通过/审核中的喉镜照片记录，按 `UploadTime` 或 `CreationTime` 倒序；
  - 支持将**同一次检查**下的多张照片做“逻辑分组”展示（可按 `CheckTime + HospitalName` 聚合，前端表现为“一张检查卡片 + 多张缩略图”）；
  - 每条记录展示：
    - 缩略图（`LaryngoscopePhotoUrl`）；
    - 标题或简短描述（如“2026-03-01 北京同仁医院 喉镜检查”——可从 `PhotoDesc` 的前若干字符提取）；
    - 关联检测时间：若有 `DetectId`，显示对应 `taudioscreenrecord` 的检测时间；
    - 审核状态：待审核 / 已通过 / 未通过（文案友好说明，不展示具体审核原因给普通用户，避免引起误解）。
  - 支持操作：
    - 点击查看大图（可兼容系统图片预览）；
    - 点击“编辑”进入详情/编辑页；
    - 长按/点击“更多”菜单进行“删除”（软删除，前端需二次确认）。

#### 3.3.3 LaryngoscopePhotoEdit（上传/编辑页）

- 表单内容：
  1. **照片上传**
     - 使用现有上传组件（可复用 `upload-images` 组件），限制单次上传 1 张；
     - 支持从相册选择或直接拍摄；
     - 图片大小和格式限制（例如 ≤5MB，jpg/png）。
  2. **关联检测记录（可选但强推荐）**
     - 下拉/弹窗选择近期音频自查记录（如最近 10 条），显示检测时间 + 结果摘要；
     - 若从 `AudioResult.vue` 中携带 `DetectId` 进入，则自动填充。
  3. **检查时间**
     - 日期选择器，默认当前日期，允许用户调整为实际检查日期。
  4. **医院与检查类型描述（映射到 PhotoDesc）**
     - 文本输入框，示例占位符：“如：北京同仁医院，电子喉镜检查”；
     - 文本长度限制（如 100 字内），由后端保存到 `PhotoDesc`。
  5. **备注（可选）**
     - 文本框，记录医生建议、个人感受等，仅用于个人回顾。
  6. **提交按钮**
     - 文案：“保存到我的喉镜影像档案”

- 业务规则：
  - 至少需上传 1 张图片；
  - 若用户未登录，不允许进入本页；
  - 保存后新建/更新 `tlaryngoscopephoto` 记录，`AuditStatus` 默认为待审核（由后端约定）。
  - 建议在表单底部固定展示隐私说明：
    - 照片仅对本人可见，不会自动分享给第三方或作为模型训练数据；
    - 如后续需要将喉镜照片用于模型优化训练，会在“模型优化标注模块”中单独征得授权。

### 3.4 与其他模块的联动

- **与健康档案模块**：
  - 在健康档案详情页增加“喉镜影像”区块，展示最近 1–2 次喉镜照片的缩略图及检查时间；
  - 点击进入 `LaryngoscopePhotoList.vue`。
- **与模型优化标注闭环模块**：
  - 标注详情页（如 `ModelLabelDetail.vue`）中，可显示与该检测记录关联的喉镜照片缩略图，提示：“此图像仅作就诊记录参考，不参与当前版本模型自动判断”。
- **与就医辅助模块**：
  - 就诊准备报告（`tusermedicalpreparereport` 生成的内容）中，可在“既往检查资料”部分列出最近一次喉镜照片信息，提醒用户就诊时携带原始报告。  

### 3.5 管理端：喉镜影像审核与管理（可选 P1）

- 路由建议：`/Admin/LaryngoscopePhotoList.vue`
- 功能：
  - 列表查看用户上传的喉镜照片，字段包括：用户、上传时间、检查时间（从描述中解析或后续加字段）、审核状态等；
  - 支持审核操作：通过 / 拒绝（拒绝时填写简单原因，仅用于运营记录，前端仅展示通用提醒文案，避免引发误解）；
  - 可按 `AuditStatus`、时间范围等过滤，避免不合规图片残留。

## 四、饮食规则驱动的推荐解释与场景卡片（强化 tdietrecommendrule 的使用）

> 说明：`饮食管理与推荐模块-需求.md` 中已整体规划了食物库与饮食推荐逻辑，本节不重复，而是重点回答“如何更好地把 `tdietrecommendrule` 利用起来，让推荐 **可解释、可运营**”。

### 4.1 目标与思路

- 目标：
  - 让用户理解“为什么系统会推荐这些友好/忌口食物”，提升信任与执行意愿；
  - 让运营/医生可以通过规则配置，快速定义针对不同症状和风险等级的“饮食场景卡片”，并观察命中情况。
- 思路：
  - 前端：在饮食推荐页上增加“推荐解释”和“场景卡片”展示；
  - 后端：在推荐接口内使用 `tdietrecommendrule` 进行匹配，并吐出“命中的规则列表 + 文案”，前端直接展示。

### 4.2 数据基础（tdietrecommendrule）

- 字段回顾：
  - `SymptomType`：匹配症状，字符串枚举，支持多症状用逗号分隔；
  - `PrimaryScreenRelation`：初筛结果关联（良性/恶性/不限）；
  - `RiskLevelScope`：适用风险等级，如 `LOW,MID,HIGH,ANY`；
  - `RecommendCategoryId`：关联 `tlaryngealfoodcategory`（如“润喉类”、“辛辣类”）；
  - `RecommendPriority`：1=最高，5=最低，用于排序；
  - `RuleDesc`：规则说明文案；
  - `IsEnabled` / `IsDelete`：启用与软删除标记。

> 当前结构已足够支持“基于症状 + 风险等级 + 初筛结果 → 推荐食物分类 + 规则解释”。

### 4.3 小程序端：饮食推荐页“推荐解释”增强

#### 4.3.1 适配页面

- 页面：`pages/Front/DietHome.vue`（见“饮食管理与推荐模块-需求.md”）  

#### 4.3.2 新增展示内容

- 在“推荐友好食物卡片区”和“推荐忌口食物卡片区”下方，增加一个折叠区域：
  - 标题：**“为什么会推荐这些？”**
  - 内容：后端返回的已命中 `tdietrecommendrule` 的简化说明列表，每条一行：
    - 示例文案（由后端组合）：
      - “因为你近期常见症状包含【咽干】且当前为【中风险】，系统根据规则推荐增加【润喉类】食物，减少【辛辣类】食物。”  
      - “你的近期初筛结果偏【高风险】，建议减少高油高盐类食物，保护声带恢复环境。”  
  - 展示规则：
    - 默认仅展示前 2 条，点击“展开更多”查看全部；
    - 若没有命中规则（极少场景），展示：“当前暂无针对性饮食规则命中，本页为通用饮食建议，仅供参考”。
  - 在折叠区域上方，可选增加 1–3 张“场景卡片”（基于 `SceneCode`/`DisplayWeight`）：
    - 卡片内容示例：
      - 标题：如“近期咽干 + 中风险的饮食建议”；
      - 摘要：来自对应规则的 `RuleDesc` 前几句；
      - CTA 按钮：
        - “去看推荐食物列表” → 跳转到带好分类和类型筛选条件的食物列表页面；
        - “记录今天吃了什么” → 跳转到饮食记录编辑页。

#### 4.3.3 接口调整建议

- 在原有饮食推荐接口（见“饮食管理与推荐模块-需求.md”中的推荐接口）基础上，扩展返回字段：
  - `MatchedRules`: `[{ RuleId, SymptomType, PrimaryScreenRelation, RiskLevelScope, RecommendCategoryId, RuleDesc, RecommendPriority, SceneCode, DisplayWeight }]`
  - 前端只消费 `RuleDesc` + 一些关键枚举的中文名称，拼装解释文案，并根据 `SceneCode/DisplayWeight` 决定是否生成“场景卡片”及排序。

### 4.4 管理端：饮食规则配置与命中统计简版

#### 4.4.1 规则配置页（P1）

- 路由建议：`/Admin/DietRecommendRuleList.vue`
- 功能：
  - 列表展示 `tdietrecommendrule` 中的规则；
  - 支持新增/编辑/启用停用规则（通过标准表单维护 `SymptomType`、`RiskLevelScope`、`RecommendCategoryId`、`RuleDesc` 等字段）；
  - 支持按症状关键字搜索规则。

#### 4.4.2 命中统计（P2，可后续）

- 通过在推荐接口中记录规则命中日志（不必马上建新表，可先按日聚合埋点到日志系统），后续在管理端增加简单统计：
  - 各规则近 7 天命中次数（或直接利用 `HitCount` 字段做近似统计，并定期做全量校正）；
  - 与用户实际饮食记录的简单对比（如：命中“减少辛辣类”规则的用户中，有多少在后续 7 天内减少了辛辣摄入）。  

> 建议：在推荐接口中，每次命中某条规则时，对其 `HitCount` 做一次自增（可通过异步批量更新的方式降低并发写入压力）。

## 五、接口设计概要（后端 SpringBoot）

> 仅列出与本需求新增功能直接相关的接口，具体 DTO 可按现有项目风格补充。

### 5.1 模型服务状态接口

- **获取单次检测的模型调用日志聚合信息（本地网络模型/算法服务）**
  - 接口：`POST /Front/Model/CallLogSummary`
  - 请求参数：
    - `{ "DetectId": Integer }`
  - 返回数据示例：
    - `{ "Success": true, "Data": { "StatusLevel": "NORMAL|WARNING|ERROR", "TotalCalls": 3, "SuccessCalls": 3, "FailCalls": 0, "AvgCost": 320, "Items": [ { "CallLink": 0, "CallLinkName": "预处理", "CallStatus": 1, "CallCost": 120, "ResultSummary": "ok" }, ... ] } }`

### 5.2 喉镜影像相关接口

- **新增/编辑喉镜照片记录**
  - 接口：`POST /Front/Laryngoscope/SavePhoto`
  - 请求参数（示例）：
    - `{ "Id": Integer|null, "DetectId": Integer|null, "LaryngoscopePhotoUrl": String, "PhotoDesc": String, "UploadTime": DateTime }`
  - 返回：
    - `{ "Success": true, "Data": { "Id": Integer }, "Msg": "" }`

- **获取当前用户喉镜照片列表**
  - 接口：`POST /Front/Laryngoscope/List`
  - 请求参数：
    - `{ "Page": Integer, "Limit": Integer }`（后台根据登录用户自动过滤）
  - 返回：
    - `PagedResult<TLaryngoscopePhotoDto>`

- **删除喉镜照片记录（软删除）**
  - 接口：`POST /Front/Laryngoscope/Delete`
  - 请求参数：
    - `{ "Id": Integer }`

### 5.3 饮食推荐规则解释增强

- 在现有饮食推荐接口（参考 `饮食管理与推荐模块-需求.md`）中增加返回字段 `MatchedRules`（见 4.3.3），不新增额外接口。

### 5.4 管理端接口（概要）

- `GET/POST /Admin/ModelInterfaceCallLog/List`：模型接口调用日志分页查询；
- `GET /Admin/ModelInterfaceCallLog/Detail?id=xxx`：单条日志详情；
- `POST /Admin/LaryngoscopePhoto/List`：喉镜照片审核列表；
- `POST /Admin/LaryngoscopePhoto/Audit`：审核通过/拒绝；
- `POST /Admin/DietRecommendRule/List`：饮食规则列表；
- `POST /Admin/DietRecommendRule/Save`：新增/编辑规则；
- `POST /Admin/DietRecommendRule/ChangeStatus`：启用/停用规则。

## 九、数据库变更建议（如实施需单独出 SQL 文档）

> 说明：本节仅给出**可选**的结构优化建议，便于后续扩展更高级功能。实际执行时请在 `healthcontrol.sql` 之外，单独新建类似“模块数据库修改.sql”的脚本文件，并按发布流程执行。

### 9.1 `tmodelinterfacecalllog` 优化建议（可选）

1. **增加模型版本与服务标识字段**
   - 目的：区分不同版本的本地网络模型与不同的内部算法服务，便于回溯性能与误差。
   - 建议字段：
     - `ModelVersion` varchar(32) COMMENT '模型版本号，如 v1.0.3'  
     - `ServiceName` varchar(64) COMMENT '内部服务名称/标识，例如 audio_classifier 或 preprocessor_v2'
   - 示例 SQL（仅示意，不直接执行）：
     - `ALTER TABLE tmodelinterfacecalllog ADD COLUMN ModelVersion varchar(32) DEFAULT NULL COMMENT '模型版本号' AFTER ModelInterfaceType;`
     - `ALTER TABLE tmodelinterfacecalllog ADD COLUMN ServiceName varchar(64) DEFAULT NULL COMMENT '内部服务名称' AFTER ModelVersion;`

2. **增加简单质量标签字段**
   - 目的：在后端根据错误码/耗时等计算质量标签，方便前端直接展示“服务状态等级”。
   - 建议字段：
     - `QualityLevel` tinyint(1) COMMENT '服务质量等级：0=正常,1=轻微异常,2=严重异常'

### 9.2 `tlaryngoscopephoto` 优化建议（可选）

1. **显式检查时间与医院字段**
   - 目前 `PhotoDesc` 混合了“医院 + 检查类型 + 备注”，不利于结构化查询和排序。建议新增：
     - `CheckTime` datetime COMMENT '实际检查时间（区别于上传时间）'
     - `HospitalName` varchar(128) COMMENT '检查医院名称'
     - `CheckType` varchar(64) COMMENT '检查方式，如电子喉镜、纤维喉镜等'
   - 这些字段有助于：
     - 在列表中按检查时间排序（更符合用户认知）；
     - 在就诊准备报告中结构化展示医院信息；
     - 后续按医院/检查类型做简单统计。

2. **视图角度/侧别字段（P2，可选）**
   - 如后续需要更精细的影像管理，可增加：
     - `ViewPosition` varchar(32) COMMENT '视角/体位，如正视、左侧位、右侧位'
     - `LesionSide` varchar(16) COMMENT '病变侧别，如左侧/右侧/双侧'
   - 当前阶段可暂不实现。

### 9.3 `tdietrecommendrule` 优化建议（可选）

1. **增加场景编码与展示权重**
   - 为了在前端做“场景卡片”（如“近期咽干 + 中风险的饮食建议”），建议增加：
     - `SceneCode` varchar(64) COMMENT '规则场景编码，例如 DRY_THROAT_MID_RISK'
     - `DisplayWeight` tinyint(4) COMMENT '前端展示权重，数值越大越靠前'

2. **增加命中统计基础字段（轻量级方案）**
   - 若希望在不新增日志表的前提下了解大致命中情况，可增加：
     - `HitCount` int(11) DEFAULT 0 COMMENT '规则累计命中次数（近似统计）'
   - 在推荐接口中，每次命中某条规则时对 `HitCount` 做自增（需注意并发与锁策略，可采用异步批量更新）。

> 再次强调：以上 9.1–9.3 所述字段均为**可选优化**，当前文档规划的功能在多数情况下可以通过现有结构 + 代码逻辑完成。若确定需要修改，请单独维护 SQL 变更文件，而不是直接改 `healthcontrol.sql`。

## 十、典型用户路径示例（跨模块联动）

为了说明本模块与现有“就医辅助、健康档案、模型优化标注、饮食管理”等模块之间的联动关系，下面给出一个从自查到复查的典型用户旅程示例：1. **居家音频自查阶段**
   - 用户在小程序中完成一次音频自查，系统写入 `taudioscreenrecord` 记录；
   - 同时在后端依次调用本地网络模型的预处理、特征提取、分类等服务，产生多条 `tmodelinterfacecalllog` 记录；
   - 检测结果页 `AudioResult.vue` 中，展示“模型服务状态”卡片，让用户知道本次检测服务是否顺畅。

2. **就医与喉镜检查阶段**
   - 若当前风险或初筛结果达到一定阈值，就医辅助模块给出就医建议，引导用户线下就诊；
   - 用户在医院完成喉镜检查后，回到小程序，在“喉镜影像档案”中上传检查照片，填写医院与检查时间，并与本次或历史某次检测记录（`DetectId`）关联；
   - 这些喉镜照片在健康档案中以“既往检查资料”的形式结构化呈现，供用户和医生日后回顾。

3. **确诊标注与隐私授权阶段**
   - 用户根据医院诊断结果，在“模型优化标注模块”中为对应的检测记录填写确诊结果（良性/恶性/正常等），生成 `tmodeloptimizelabel` 记录；
   - 同时可在隐私设置模块中决定是否开启“数据匿名授权”，决定这些确诊数据是否用于后续模型优化训练；
   - 若授权，后端在训练数据准备阶段即可综合利用音频自查记录 + 喉镜影像 + 确诊信息进行模型优化（不在本需求文档内展开）。

4. **饮食与生活方式调整阶段**
   - 系统根据用户的症状日志、风险等级、确诊结果等信息，通过 `tdietrecommendrule` 命中若干饮食规则；
   - 在饮食推荐首页 `DietHome.vue` 中，以“场景卡片 + 推荐解释”的形式呈现个性化饮食建议，引导用户查看具体友好/忌口食物，并记录每日饮食；
   - 随着时间推移，饮食记录与症状变化一起沉淀在健康档案中，为用户观察自我管理效果提供依据。

5. **复查与长期随访阶段**
   - 一段时间后，用户再次进行音频自查或喉镜复查，新的 `taudioscreenrecord`、`tlaryngoscopephoto`、`tmodelinterfacecalllog` 数据被写入；
   - 在健康档案和喉镜影像档案中，用户可以并排查看多次检查的声学结果、喉镜图像与饮食/症状变化，形成“自查 → 就医 → 管理 → 复查”的完整闭环体验。

通过上述用户路径，本需求文档中依托 `tmodelinterfacecalllog`、`tlaryngoscopephoto`、`tdietrecommendrule` 的扩展能力，将模型服务透明度、喉镜影像档案和饮食规则三块原本较“底层”的数据能力，整合成用户可感知、可解释、可回溯的产品功能。
