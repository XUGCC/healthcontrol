# 隐私设置与数据权利模块 - 详细需求文档

## 一、模块概述

### 1.1 模块定位
隐私设置与数据权利模块是居家喉部健康自查小程序的核心合规模块，旨在为用户提供透明的隐私控制权和数据权利保障，满足《个人信息保护法》、《网络安全法》等法规要求，同时为模型优化标注模块提供授权基础，形成"**用户授权 → 数据使用 → 模型优化 → 更准自查**"的合规数据闭环。

### 1.2 核心价值
- **合规保障**：满足个人信息保护法、GDPR等法规要求，确保数据处理的合法性
- **用户信任**：透明的隐私设置和数据权利，提升用户对平台的信任度
- **数据闭环基础**：为模型优化标注模块提供授权机制，确保数据使用的合规性
- **用户权利**：提供数据导出、数据删除等权利，保障用户对个人数据的控制权
- **风险控制**：通过完善的隐私设置，降低数据泄露和合规风险

### 1.3 与其他模块的关系
- **依赖模块**：
  - 用户基础模块（用户信息、登录认证）
  - 模型优化标注模块（数据匿名授权联动）
- **支撑模块**：
  - 为模型优化标注模块提供授权状态校验
  - 为数据导出/删除功能提供基础服务
- **数据来源**：
  - `tuserprivacysetting`：用户隐私设置表（核心表）
  - `appuser`：用户表（关联用户信息）
  - 其他业务表：用于数据导出和删除（音频记录、健康档案、症状日志等）

## 二、功能需求详细说明

### 2.1 隐私设置主页面（PrivacySetting.vue）

#### 2.1.1 功能描述
用户隐私设置的主页面，提供本地存储、数据匿名授权、隐私协议等核心隐私设置的统一管理入口。

#### 2.1.2 页面结构

**顶部导航栏**：
- 标题："隐私设置"
- 返回按钮：返回上一页

**页面内容区域**：

1. **本地存储设置**
   - **标题**："本地存储"
   - **开关**：控制是否在本地设备缓存音频、报告等数据
   - **说明文字**：
     - 开启：允许在本地设备缓存音频记录、检测报告等数据，提升加载速度
     - 关闭：不缓存数据，每次查看需要重新加载，节省存储空间
   - **业务规则**：
     - 默认关闭（`LocalStorageStatus = false`）
     - 用户可随时开启或关闭
     - 关闭时，系统应提示是否清除已缓存的本地数据

2. **数据匿名授权设置**
   - **标题**："数据匿名授权"
   - **开关**：控制是否授权匿名数据用于模型优化
   - **说明文字**：
     - 开启：您的匿名数据将用于模型优化，帮助提升检测准确率，数据将经过脱敏处理
     - 关闭：您的数据将仅用于个人记录，不参与模型优化
   - **业务规则**：
     - 默认关闭（`DataAnonymousAuth = false`）
     - 用户可随时开启或关闭
     - **重要**：关闭总开关不会影响已授权的标注记录（已在标注详情中单独授权的记录保持授权状态）
     - 开启时，系统应提示用户查看隐私协议
     - 与模型优化标注模块联动：标注时如果未开启匿名授权，会提示用户开启

3. **隐私协议状态**
   - **标题**："隐私协议"
   - **状态显示**：
     - 已同意：显示"已同意"标签和同意时间
     - 未同意：显示"未同意"标签和"立即查看"按钮
   - **操作按钮**：
     - "查看隐私协议"：跳转到隐私协议详情页（内嵌或外部链接）
     - "重新同意"：如果协议更新，允许用户重新同意
   - **业务规则**：
     - 首次使用必须同意隐私协议（`PrivacyAgreeStatus = true`）
     - 协议更新时，需要用户重新同意
     - 协议版本号应记录（可选，可在后续版本中实现）

4. **数据权利入口**
   - **标题**："我的数据权利"
   - **功能入口**：
     - "申请导出数据"：跳转到数据导出申请页
     - "申请删除数据"：跳转到数据删除申请页
   - **说明文字**：根据相关法规，您有权导出或删除您的个人数据

#### 2.1.3 业务规则
1. 所有设置变更都需要实时保存到数据库
2. 设置变更时，应显示加载状态和成功提示
3. 关键设置（如数据匿名授权）变更时，应显示确认弹窗
4. 页面加载时，应从后端获取最新的隐私设置状态

### 2.2 数据导出申请页（DataExportRequest.vue）

#### 2.2.1 功能描述
用户申请导出个人数据的页面，支持选择导出数据类型、导出格式，提交申请后由后台异步处理。

#### 2.2.2 页面结构

**表单内容区域**：

1. **导出数据类型选择**（多选checkbox）
   - 音频自查记录、健康档案记录、症状日志、饮食记录、确诊标注记录、问卷答题记录、健康提醒设置、消息通知记录

2. **导出格式选择**（单选radio）
   - JSON格式（推荐）、Excel格式、PDF格式

3. **申请说明**（可选文本输入框，最多200字）

4. **提交按钮**

#### 2.2.3 业务规则
1. 至少选择一种数据类型
2. 导出任务异步处理，状态：待处理/处理中/已完成/失败
3. 完成后生成下载链接，有效期7天
4. 同一用户同时只能有一个进行中的导出任务
5. 导出数据需脱敏处理

1. 至少选择一种数据类型
2. 删除任务异步处理

4. 删除账户时需要身份验证

### 2.6 模型优化授权明细页（AuthUsageDetail.vue）

#### 2.6.1 功能描述
为用户提供一个集中查看和管理“用于模型优化的已授权标注记录”的入口，强化授权透明度和可控性。该页面可以作为独立入口，也可以与现有的`ModelLabelList.vue`进行合并或复用。

#### 2.6.2 页面结构

- **顶部导航栏**：
  - 标题："模型优化授权明细"
  - 返回按钮：返回上一页

- **筛选区**：
  - 授权状态筛选：全部 / 已授权 / 未授权
  - 时间范围筛选：近7天 / 近30天 / 自定义起止时间
  - 检测结果筛选（可选）：良性 / 恶性 / 其他

- **授权记录列表**（基于`tmodeloptimizelabel`与`taudioscreenrecord`联表）：
  - 检测时间（来自`taudioscreenrecord.CreationTime`）
  - 检测结果概览（初筛结果、风险等级等快照字段）
  - 医院确诊结果摘要（`HospitalDiagnoseResult`）
  - 授权状态标签：已授权 / 未授权
  - 操作按钮：
    - "查看详情"：跳转到原有标注详情页
    - "取消授权"：仅对已授权记录展示

- **空状态**：
  - 文案："暂无已授权用于模型优化的标注记录"

#### 2.6.3 业务规则

1. 列表数据来源于`tmodeloptimizelabel`，仅展示`IsDelete = 0`的记录。
2. 已授权记录的认定条件：`AuthStatus = 1` 且对应用户的`tuserprivacysetting.DataAnonymousAuth = 1`（总开关开启时，授权才视为有效）。
3. 用户在此页**只能下调授权（从已授权改为未授权）**，不能在此页批量“强行授权”所有记录，以避免误操作。
4. 取消授权时弹出确认弹窗，明确提示“取消后该条记录将不再用于后续模型优化训练，已经进入历史模型的部分难以从模型中彻底移除”。
5. 授权明细页只做**状态查看与单条收回授权**，不影响总开关逻辑，不提供“一键关闭所有历史授权”的危险操作。

### 2.7 数据留存与自动清理设置（DataRetentionSetting）

#### 2.7.1 功能描述
在隐私设置中增加“数据留存与自动清理”能力，让用户可以选择历史数据的保留时长和自动清理策略，降低长期留存带来的隐私风险。

#### 2.7.2 页面结构（整合进 PrivacySetting.vue）

在隐私设置主页面新增一个设置卡片：

3. **数据留存与自动清理**
   - **标题**："数据留存与自动清理"
   - **说明文字**：
     - "您可以设置历史数据的最长保留时间，到期后系统会自动为您清理对应数据。"
   - **控件设计**：
     - 开关：`开启自动清理（DataRetentionEnabled）`
     - 下拉/单选列表：`数据保留时长（DataRetentionMonths）`
       - 选项示例：3个月 / 6个月 / 12个月 / 永久保留（默认）
     - 补充说明文本：
       - 列出会被纳入清理范围的数据类型（如：音频自查记录、症状日志、饮食记录、消息通知等）

#### 2.7.3 业务规则

1. 默认状态为**关闭自动清理**，且保留时长为“永久保留”（不主动清理历史数据）。
2. 当用户**首次开启自动清理**时，弹出二次确认弹窗，明确说明：
   - 将定期删除早于所选保留时长的历史数据；
   - 删除后可能影响趋势分析、报告生成等功能。
3. 自动清理任务由后台定时任务实现（例如每日凌晨扫描）：
   - 依据用户设置的`DataRetentionMonths`，计算各数据表的阈值时间；
   - 对早于阈值时间的数据优先做软删除（如设置`IsDelete = 1`）。
4. 不同数据表可设置不同的清理力度（在实现层按业务判断）：
   - 检测记录（`taudioscreenrecord`）：可软删除或匿名化处理；
   - 日志类数据（如`tsciencereadlog`、`tsciencesearchlog`、消息通知等）：可直接物理删除或批量清理；
   - 已授权用于模型优化的标注记录（`tmodeloptimizelabel`）：清理前需要特别提示，不建议自动删除，或仅做脱敏/去用户关联。
5. 前端展示上，当用户已经开启自动清理时，应在说明区域明确提示当前生效的保留时长（例如："当前保留最近 6 个月的数据"）。

## 三、数据库设计

### 3.1 核心表：用户隐私设置表（tuserprivacysetting）

**当前表结构**（已在 `healthcontrol.sql` 中存在，节选）：

```sql
CREATE TABLE `tuserprivacysetting` (
  `Id` int(11) NOT NULL AUTO_INCREMENT COMMENT '用户隐私设置主键',
  `CreationTime` datetime DEFAULT NULL COMMENT '创建时间',
  `UserId` int(11) DEFAULT NULL COMMENT '关联用户表主键t_user.',
  `LocalStorageStatus` tinyint(1) DEFAULT NULL COMMENT '本地存储开启状态：0=否，1=是',
  `DataAnonymousAuth` tinyint(1) DEFAULT NULL COMMENT '数据匿名授权状态：0=否，1=是',
  `PrivacyAgreeStatus` tinyint(1) DEFAULT NULL COMMENT '隐私协议同意状态：0=否，1=是',
  `IsDelete` tinyint(1) DEFAULT NULL COMMENT '软删除标记：0=未删除，1=已删除',
  PRIMARY KEY (`Id`) USING BTREE,
  KEY `UserId` (`UserId`),
  CONSTRAINT `tuserprivacysetting_ibfk_1` FOREIGN KEY (`UserId`) REFERENCES `appuser` (`Id`) ON DELETE SET NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='用户隐私设置表';
```

**业务规则**：
1. 每个用户在业务上只应有一条隐私设置记录（`UserId` 维度唯一）
2. 如果用户没有隐私设置记录，系统在首次访问隐私设置页或首次触发相关逻辑时自动创建
3. `IsDelete` 为软删除标记，正常情况下前台只查询 `IsDelete IS NULL OR IsDelete = 0` 的记录

## 四、接口设计

### 4.1 隐私设置接口

#### 4.1.1 获取隐私设置
- **接口**：`POST /Front/Privacy/GetSetting`
- **请求参数**：`{ UserId: Integer }`
- **返回数据**：`TUserPrivacySettingDto`

#### 4.1.2 更新隐私设置
- **接口**：`POST /Front/Privacy/UpdateSetting`
- **请求参数**：`TUserPrivacySettingDto`
- **返回数据**：`TUserPrivacySettingDto`

#### 4.1.3 同意隐私协议
- **接口**：`POST /Front/Privacy/AgreeProtocol`
- **请求参数**：`{ UserId: Integer, ProtocolVersion: String }`
- **返回数据**：`{ Success: Boolean, Msg: String }`

---

### 4.2 数据导出接口

#### 4.2.1 提交导出申请
- **接口**：`POST /Front/Privacy/ExportRequest`
- **请求参数**：
```json
{
  "UserId": Integer,
  "DataType": ["AudioRecord", "HealthRecord", ...],
  "ExportFormat": "JSON|Excel|PDF",
  "RequestDesc": String
}
```
- **返回数据**：`{ Success: Boolean, Data: { RequestId: Integer }, Msg: String }`

#### 4.2.2 查询导出申请列表
- **接口**：`POST /Front/Privacy/ExportRequestList`
- **请求参数**：`{ UserId: Integer, Page: Integer, Limit: Integer }`
- **返回数据**：`PagedResult<TDataExportRequestDto>`

#### 4.2.3 下载导出文件
- **接口**：`GET /Front/Privacy/DownloadExport?RequestId=Integer`
- **返回数据**：文件流

---

### 4.3 数据删除接口

#### 4.3.1 提交删除申请
- **接口**：`POST /Front/Privacy/DeleteRequest`
- **请求参数**：
```json
{
  "UserId": Integer,
  "DataType": ["AudioRecord", "HealthRecord", ...],
  "DeleteReason": String,
  "DeleteReasonDesc": String,
  "Password": String  // 删除账户时需要
}
```
- **返回数据**：`{ Success: Boolean, Data: { RequestId: Integer }, Msg: String }`

#### 4.3.2 查询删除申请列表
- **接口**：`POST /Front/Privacy/DeleteRequestList`
- **请求参数**：`{ UserId: Integer, Page: Integer, Limit: Integer }`
- **返回数据**：`PagedResult<TDataDeleteRequestDto>`

## 十、风险与注意事项

### 10.1 合规风险

1. **数据导出/删除必须符合法规要求**
   - 响应时间要求（如30天内完成）
   - 数据范围要求（必须包含所有用户数据）
   - 脱敏要求（敏感信息必须脱敏）

2. **隐私协议必须完整**
   - 包含所有数据收集和使用条款
   - 用户必须明确同意
   - 协议更新时需要重新同意

### 10.2 技术风险

1. **大数据量导出性能问题**
   - 使用异步任务处理
   - 分批导出，避免内存溢出
   - 文件大小限制

2. **数据删除影响业务**
   - 优先使用软删除
   - 删除前备份关键数据
   - 删除后通知相关模块

### 10.3 用户体验

1. **设置变更及时反馈**
   - 显示加载状态
   - 成功/失败提示
   - 关键操作确认弹窗

2. **数据权利申请流程清晰**
   - 明确告知处理时间
   - 及时通知处理结果
   - 提供进度查询

## 十一、数据库变更清单（基于当前 `healthcontrol.sql`）

### 11.1 必须新增的表

结合当前总表设计，隐私设置与数据权利模块上线前，需至少新增 2 张表：

- **新增表 `tdataexportrequest`（数据导出申请表）**
  - 用途：保存用户“数据导出申请”的任务信息与处理状态
  - 与现有表关系：通过 `UserId` 外键关联 `appuser.Id`
  - **DDL 建议**：使用本文档 3.2 中给出的 `CREATE TABLE tdataexportrequest` 语句执行创建

- **新增表 `tdatadeleterequest`（数据删除申请表）**
  - 用途：保存用户“数据删除申请”的任务信息与处理状态
  - 与现有表关系：通过 `UserId` 外键关联 `appuser.Id`
  - **DDL 建议**：使用本文档 3.3 中给出的 `CREATE TABLE tdatadeleterequest` 语句执行创建

> 说明：当前 `healthcontrol.sql` 中不存在上述两张表，因此为**新增**，不会与现有结构冲突。

### 11.2 建议变更（可选，但推荐）

在不破坏现有功能的前提下，为增强数据一致性与审计能力，建议对 `tuserprivacysetting` 做如下优化：

- **建议 1：为 `UserId` 增加唯一索引（数据清洗后执行）**
  - 目的：保证每个用户只有一条隐私设置记录，避免脏数据
  - 前置条件：需要先清洗现有重复记录（例如只保留每个用户最新一条），再执行以下 DDL：

  ```sql
  ALTER TABLE `tuserprivacysetting`
    ADD UNIQUE KEY `uk_userprivacy_user` (`UserId`);
  ```

- **建议 2：如后续需要记录“最后修改时间”，可再新增 `UpdateTime` 字段**
  - 当前表中只有 `CreationTime`，若需要精确审计隐私设置最近一次修改时间，可按需执行：

  ```sql
  ALTER TABLE `tuserprivacysetting`
    ADD COLUMN `UpdateTime` datetime DEFAULT NULL COMMENT '更新时间' AFTER `CreationTime`;
  ```

  - 同时在 MyBatis-Plus 的公共字段填充（`MyMetaObjectHandler`）中增加对 `UpdateTime` 的自动填充逻辑。

- **建议 3：增加数据留存相关字段，支持“自动清理历史数据”功能**
  - 目的：让用户能配置是否开启自动清理以及保留时长，对应需求文档 2.7 节。
  - 字段设计示例：

  ```sql
  ALTER TABLE `tuserprivacysetting`
    ADD COLUMN `DataRetentionEnabled` tinyint(1) DEFAULT 0 COMMENT '是否开启自动清理：0=否，1=是' AFTER `PrivacyAgreeStatus`,
    ADD COLUMN `DataRetentionMonths` int(11) DEFAULT NULL COMMENT '数据保留时长（月），NULL/0=永久保留' AFTER `DataRetentionEnabled`;
  ```

  - 说明：
    - 默认关闭自动清理（`DataRetentionEnabled = 0`，`DataRetentionMonths = NULL`）
    - 当用户在前端开启自动清理并选择保留时长后，由后台保存具体数值（例如 3/6/12）。

以上变更中，**11.1 为上线必需执行的建表操作**；**11.2 为推荐优化项，可根据实际排期与数据情况择机实施**。
