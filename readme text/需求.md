# 居家喉部健康音频自查小程序 - 需求设计文档

## 1. 项目概述

### 1.1 项目背景
居家喉部健康音频自查小程序是一款基于人工智能技术的健康管理应用，旨在帮助用户通过上传音频进行嗓音病变的初步筛查。系统采用深度学习模型对音频进行特征提取和分类，为用户提供便捷、快速的健康自查服务。

### 1.2 项目目标
- 提供便捷的音频上传和识别功能
- 通过MFCC特征提取和深度学习模型实现嗓音病变识别
- 为用户提供清晰的检测结果和健康建议
- 为管理员提供完整的检测记录管理和数据分析功能

### 1.3 技术架构（对应当前代码目录）
- **小程序前端**：UniApp（`HealthControl.uniapp`，支持微信小程序、H5 等多端）
- **管理端前端**：Vue 3 + Element Plus（`HealthControl.elementui`）
- **后端服务**：Spring Boot 4 + MyBatis Plus（`HealthControl.springboot`）
- **音频处理与 AI 模型**：Python + librosa + PyTorch DenseNet 模型（`voice` 目录，使用 `voice_best_model.pth` 与 `voice_inference.py`/封装脚本）
- **数据库**：MySQL，用于存储用户信息与检测记录（可选，当前实现允许“只检测不落库”）

## 2. 核心功能需求

### 2.1 音频识别流程（同步调用 + 全链路错误透传）

#### 2.1.1 功能描述
用户通过小程序上传音频文件，系统自动完成以下流程：
1. 小程序通过 `uni.uploadFile` 将音频上传到后端 Spring Boot（无需单独文件服务）
2. 后端将音频保存到本机临时目录
3. 后端使用 `ProcessBuilder` 同步调用 Python 推理脚本（封装 `voice_inference.py` 与 `voice_best_model.pth`）
4. Python 读取音频，提取 MFCC 特征并通过 DenseNet 模型进行二分类推理（0=良性，1=恶性）
5. Python 统一以 **单一 JSON** 对象写入标准输出，包含成功结果或完整错误信息
6. 后端解析 Python 输出，将 `success/data/error` 统一封装返回给小程序
7. 小程序根据 `success` 字段分别展示检测结果或完整错误信息，同时可选择将结果写入检测记录

> **补充说明（阶段性实现）**：  
> - 第一阶段以 **命令行脚本形式** 集成 Python（非 Web 服务），保证在单机环境下可稳定跑通 `voice/test_2_82`。  
> - 后续阶段可在不改变前端接口的前提下，将 Python 封装为 Flask/FastAPI 服务，由 Spring Boot 通过 HTTP 或 gRPC 调用。

#### 2.1.2 核心业务与判定规则
- **二分类规则**：  
  - 模型输出 `predicted_class` 与 `confidence`；  
  - 若 `predicted_class == 0` 且 `confidence > 0.5`，则视为 **良性 (0)**；  
  - 否则视为 **恶性 (1)**。
- **错误透传策略**：  
  - 任何环节（上传失败 / Java 调用失败 / Python 解码失败 / 模型推理失败 / JSON 解析失败）都必须在后端聚合为带有 `stage、errorType、errorMessage、stackTrace/rawOutput` 等字段的 JSON，并通过同一 HTTP 响应返回给小程序；  
  - 小程序在结果页将这些错误信息人性化展示，方便用户和开发者定位问题。

- **典型错误阶段枚举（建议后端/脚本统一约定）**：
  - `stage = "resolve_paths"`：组装音频/模型路径时失败（如路径不存在、权限不足）；
  - `stage = "load_model"`：加载 `voice_best_model.pth` 失败（文件缺失、版本不兼容、CUDA/CPU 环境异常等）；
  - `stage = "load_audio"`：读取音频失败（不支持的格式、文件损坏、未安装 ffmpeg 等）；
  - `stage = "preprocess"`：特征提取/MFCC 计算失败；
  - `stage = "inference"`：模型前向推理报错或输出异常；
  - `stage = "postprocess"`：prob0/prob1/benign 等后处理逻辑异常；
  - `stage = "java_parse_python_json"`：后端解析 Python JSON 失败；
  - `stage = "java_exception"`：后端 Java 层其他未捕获异常。

#### 2.1.3 验收与测试数据集
- **内置测试数据集**：`voice/test_2_82`，包含“声带良性”与“喉肿物”等目录。
- **验收要求**：
  - 能够通过小程序前端上传 `test_2_82` 中的任意 MP3 文件进行检测；
  - 对于可正常识别的音频，前端应显示 **良性/恶性结果 + 置信度 + 模型信息**；
  - 若在任何前置环节或推理环节出现问题，前端必须能看到清晰的错误阶段（如 `stage=load_audio`、`stage=inference`、`stage=java_parse_python_json` 等）和具体错误信息（`errorMessage`），不得出现“长期转圈无反馈”的情况。

### 2.2 小程序端功能（UniApp）

  - 上传失败重试机制

    - 选择文件后先在前端校验扩展名与大小；不符合要求时，以对话框或 toast 形式提示原因；

- **结果展示**
  - 检测状态显示（检测中 / 已完成 / 失败 / 超时）
  - 检测结果展示（良性 / 恶性）
  - 判定规则透明化：predicted_class=0 且置信度>0.5 视为良性，否则视为恶性

  - 显示检测时间与音频基础信息（时长、采样率等）
  - 显示模型关键信息：predicted_class、prob0、prob1、method、模型版本号（用于调试与科研）
  - UI 行为细化：
    - 在“检测中”状态时展示统一的加载动画与简短提示文案（例如“请保持网络畅通，通常 3–10 秒可出结果”）；
    - “已完成”状态下，采用卡片式布局：顶部为结果 Tag（良性/恶性）、中部为置信度与基本信息、底部为健康建议与操作按钮；

  - MFCC 特征图展示，支持点击放大预览

  - 健康建议须以“非诊断声明”开头，明确提示“本结果仅供居家自查参考，不作为临床诊断依据，如有不适请及时就医”。

    - 展示字段：检测结果（良/恶性 Tag）、置信度、检测时间、音频时长、检测状态（成功/失败/超时）

## 4. 接口设计（同步检测 + 可选记录管理）

### 4.1 小程序端核心检测接口

#### 4.1.1 上传并检测音频（同步调用 Python）
统一由小程序通过 `uni.uploadFile` 调用后端，后端在同一接口中完成“保存临时文件 + 调 Python + 解析结果 + 返回给前端”。

```
POST /api/audio/detect
Content-Type: multipart/form-data

Request (Form-Data):
- file: 音频文件 (mp3/wav/flac/m4a)
- userId: 用户ID（可选）

Response:
{
  "success": true,          // true=推理成功，false=任意环节失败
  "data": { ... },          // success=true 时，承载 Python 返回的成功结果
  "error": { ... },         // success=false 时，承载 Python 或 Java 的错误信息
  "requestId": "uuid",      // 便于日志与记录追踪
  "message": "检测成功/检测失败（来自模型或音频处理）/后端处理异常"
}
```

- 当 `success = true` 时，`data` 示例（Python 约定）：

```json
{
  "success": true,
  "stage": "inference",
  "audioPath": "C:/.../voice/test_2_82/声带良性/杨俊香2022-10-8.MP3",
  "predicted_class": 0,
  "confidence": 0.87,
  "prob0": 0.87,
  "prob1": 0.13,
  "benign": true,
  "rule": "predicted_class=0 && confidence>0.5 => benign",
  "usingModel": "voice_best_model.pth",
  "device": "cuda"
}
```

- 当 `success = false` 时，`error` 示例：

```json
{
  "success": false,
  "stage": "load_audio",           // 错误发生阶段：resolve_model_path/load_model/load_audio/inference/java_parse_python_json/java_exception 等
  "audioPath": "C:/.../xxx.MP3",
  "errorType": "RuntimeError",
  "errorMessage": "无法加载音频文件 ...",
  "hint": "可能原因：1) 未安装 ffmpeg 2) 文件损坏 3) 格式不支持",
  "stackTrace": "traceback 字符串（可截断）"
}
```

> 说明：小程序端只需要关注 `success` 字段和 `data/error` 内容，无需再单独轮询其他接口即可拿到完整结果或错误信息。

#### 4.1.2 状态码与错误码建议

- HTTP 状态码约定：
  - 200：请求成功（无论检测成功/失败，均返回 200，具体由 `success` 字段区分业务结果）；
  - 400：明显的参数错误（缺少 file、文件过大等，后端可不进入 Python 调用）；
  - 500：后端 Java 出现未捕获异常（极端情况，前端提示“系统异常，请稍后重试”并记录 requestId）。
- 业务错误码（可放在 `error.errorCode` 字段中）：
  - `AUDIO_INVALID_FORMAT`：不支持的音频格式；
  - `AUDIO_TOO_LARGE`：文件超出大小限制；
  - `AUDIO_TOO_SHORT`：音频时长过短；
  - `PYTHON_LOAD_MODEL_FAIL`：加载模型失败；
  - `PYTHON_LOAD_AUDIO_FAIL`：音频读取失败；
  - `PYTHON_INFERENCE_FAIL`：推理阶段失败；
  - `JAVA_PARSE_JSON_FAIL`：后端 JSON 解析失败；
  - `UNKNOWN`：其他未归类错误。

## 5. 数据流程设计

### 5.1 完整检测流程（同步链路 + 可选记录）

```
1. 小程序端（HealthControl.uniapp）
   ├─ 用户录音或从本地选择音频文件（mp3/wav/flac/m4a）
   ├─ 使用 uni.uploadFile 调用 POST /api/audio/detect
   └─ 等待接口同步返回（包含成功结果或完整错误信息）

2. 后端服务（HealthControl.springboot）
   ├─ 接收上传的音频文件，并保存至本机临时目录
   ├─ 使用 ProcessBuilder 调用 Python 脚本（voice/voice_api.py）
   │  ├─ 传入音频绝对路径
   │  ├─ 等待 Python 同步执行完毕（设置超时时间，例如 120 秒）
   │  └─ 从标准输出读取唯一一条 JSON 字符串
   ├─ 解析 Python JSON 输出
   │  ├─ 若 success=true，则组装业务 success=true 的 HTTP 响应
   │  └─ 若 success=false 或 JSON 解析失败，则组装业务 success=false 的 HTTP 响应（包含 stage、errorType、errorMessage 等）
   └─ （可选）将检测结果或错误信息写入数据库检测记录表

3. Python 服务（voice 目录）
   ├─ 解析命令行参数 (--audio)
   ├─ 解析模型路径 voice_best_model.pth（脚本同级或 voice 子目录）
   ├─ 加载 DenseNet 模型（使用 voice_inference.py 中的 load_model）
   ├─ 加载并处理音频
   │  ├─ 使用 librosa.load 读取 MP3/WAV
   │  ├─ 必要时提示安装 ffmpeg 或处理解码错误
   │  └─ 提取 MFCC 特征并插值到模型输入尺寸
   ├─ 模型推理
   │  ├─ 计算预测类别 predicted_class 与置信度 confidence
   │  ├─ 根据规则计算良性/恶性（predicted_class=0 && confidence>0.5 为良性，否则为恶性）
   │  └─ 生成包含 prob0/prob1 的结果 JSON
   └─ 统一输出一条 JSON 到 stdout
      ├─ 成功：success=true + 结果字段
      └─ 失败：success=false + stage + errorType + errorMessage + stackTrace

4. 小程序端结果展示
   ├─ 根据 HTTP 响应中的 success 字段判断分支
   ├─ 若 success=true，展示良性/恶性结果、置信度、模型信息等
   ├─ 若 success=false，展示错误阶段（stage）、错误类型和详细错误信息
   └─ 提供“返回首页”“查看检测记录”等操作入口
```

### 5.2 与测试数据集（voice/test_2_82）的集成

- 在开发与联调过程中，建议直接使用 `voice/test_2_82` 目录下的样本（如“声带良性”“喉肿物”子目录中的 MP3）作为主要测试数据；
- 要求：
  - 后端可通过命令行独立调用 Python 脚本对这些音频进行推理，并得到合理的结果 JSON；
  - 小程序通过上传这些音频文件，能在前端看到与命令行一致的良/恶性判定与置信度；
  - 若由于环境问题（如 Python 环境、依赖库、ffmpeg 缺失等）导致推理失败，前端应展示包含具体错误详情的错误卡片，不能只显示“检测失败”而没有原因。

#### 8.1.1 交互与状态细化

- 页面状态：
  - `idle`：默认状态，尚未开始录音或选择文件；
  - `recording`：正在录音，录音按钮高亮，显示计时；
  - `recorded`：录音完成，可以播放/删除/重新录制；
  - `uploading`：正在上传并检测，不允许重复点击提交；
  - `error`：存在最近一次操作错误，可在顶部或弹窗中展示错误提示。
