## 饮食管理与推荐模块 - 实现介绍文档

## 一、模块概述

饮食管理与推荐模块围绕“**喉部友好/忌口食物库 + 个性化饮食推荐 + 日常饮食记录**”三条主线，为居家喉部健康自查小程序提供饮食指导闭环能力。  

本模块已经完成以下几块能力：

- **饮食推荐首页**：基于用户最近的喉部健康档案、音频初筛结果、症状日志与风险等级，生成当日饮食总览与推荐/忌口分类列表。
- **喉部专用食物库浏览与搜索**：按分类（友好/忌口）浏览食物，支持关键字搜索与分页。
- **食物详情 + 快速记录**：查看单个食物的喉部友好/危害说明，并一键加入“今天的饮食记录”。
- **饮食记录管理**：查看某时间范围内的饮食记录，支持编辑、删除与主观感受反馈。
- **饮食推荐规则后台管理**：在管理端配置规则（症状 + 初筛结果 + 风险等级范围 → 推荐分类），支持启停、排序、批量操作与规则命中预览。

后端基于 **Spring Boot + MyBatis-Plus**，前端基于 **uni-app（微信小程序/H5）**，管理端为 **Element Plus**。接口统一通过 `GlobalResponseAdvice` 封装为：

```json
{
  "Code": "200",
  "Success": true,
  "Msg": "成功",
  "Data": { ... }
}
```

## 二、主要实现功能概览

### 2.1 饮食推荐首页（DietHome）

- **前台入口**
  - 首页功能卡增加“饮食建议”入口：`Index.vue` → 跳转 `pages/Front/DietHome`。
- **页面结构**
  - 顶部：今日喉部状态卡片（风险等级、摘要文案、最近一次检测时间/问卷情况简要说明）。
  - 中部：推荐食物分类区
    - “适合吃”（喉部友好类）：如润喉、清淡、温补等分类。
    - “建议少吃/避免”（忌口类）：如辛辣、刺激、过烫等分类。
    - 每个分类展示若干代表性食物（缩略图 + 名称）。
  - 底部：入口按钮
    - “查看全部友好食物”“查看全部忌口食物”
    - “我的饮食记录”
- **核心交互**
  - 页面加载时调用 `/Front/Diet/Home`，后端根据用户最近的健康档案 + 音频初筛 + 症状日志 + 风险等级匹配规则并生成推荐分类列表。

### 2.2 食物库浏览与搜索（DietFoodList）

- **页面入口**
  - 从 `DietHome` 进入“友好食物”或“忌口食物”列表：`pages/Front/DietFoodList`。
- **主要能力**
  - 顶部 Tab：`喉部友好 / 忌口` 切换（内部以 `CategoryType` 或分类标记区分）。
  - 搜索框：支持按食物名称/别名关键字搜索。
  - 列表项：食物图片、名称、标签（如润喉/辛辣）、简要说明。
  - 上拉分页加载。
- **接口**
  - `POST /Front/Diet/FoodList`：入参包括 `CategoryType`、`Keyword`、`Page`、`Limit` 等。

### 2.3 食物详情 + 快速记录（DietFoodDetail）

- **页面入口**
  - 从 `DietFoodList` 点击任一食物进入 `pages/Front/DietFoodDetail`。
- **主要能力**
  - 展示食物基础信息：图片、名称、别名、所属分类。
  - 对友好类：功效说明、推荐吃法、适合人群、不适合人群。
  - 对忌口类：危害说明、为什么要少吃/避免、替代建议。
  - 动态提醒区：结合用户风险等级与当前食物友好度，给出“适量/谨慎/避免”等提示文案。
  - 底部按钮：“加入今天的饮食记录”。
- **快速记录弹窗**
  - 使用 `uni-popup` 打开表单：
    - 记录时间（默认当前时间，可修改）。
    - 份量/频次（简单输入）。
    - 主观感受（咽干缓解/不适/刺激等，采用枚举编码）。
  - 提交后调用 `/Front/Diet/RecordCreateOrEdit` 新增记录。

### 2.4 饮食记录列表与编辑（DietRecordList）

- **页面入口**
  - 从 `DietHome` 底部“我的饮食记录”进入 `pages/Front/DietRecordList`。
- **能力**
  - 支持按日期范围筛选（如最近 7 天/30 天）。
  - 每条记录展示：时间、食物名称、分类（友好/忌口）、主观感受。
  - 提供“编辑”“删除”按钮：
    - 编辑：打开弹窗修改 `EatFeeling` 等字段（重用 `/Front/Diet/RecordCreateOrEdit`）。
    - 删除：软删除，调用 `/Front/Diet/RecordDelete`。

### 2.5 后台饮食推荐规则管理（TDietRecommendRuleList）

- **页面位置**
  - ElementUI 管理端：`TDietRecommendRuleList.vue` + `TDietRecommendRuleController/TDietRecommendRuleServiceImpl`。
- **功能点**
  - 规则列表：
    - 字段：症状集合、初筛结果关联、推荐分类、推荐优先级、适用风险等级范围（`RiskLevelScope`）、启用状态、软删除标记等。
  - 搜索与过滤：
    - 关键字搜索：在症状、规则说明等字段中模糊匹配。
    - 筛选条件：`PrimaryScreenRelation`、`RecommendPriority`、`RiskLevelScope`、`IsEnabled` 等。
  - 规则增删改：
    - 编辑弹窗支持下拉选择推荐分类、风险等级范围字符串、初筛关系等。
    - 删除为软删除：`IsDelete=true`。
  - 批量操作：
    - `批量启用` / `批量停用`：调用 `/TDietRecommendRule/BatchSetEnabled`，更新多条记录的 `IsEnabled`。
  - **规则命中预览（rule hit explanation）**
    - 工具栏新增“规则命中预览”按钮。
    - 弹窗表单输入：
      - 模拟风险等级（0=低、1=中、2=高 或不限）。
      - 模拟初筛结果（良性/恶性/不限）。
      - 模拟症状（逗号分隔）。
    - 点击“开始预览”调用 `/TDietRecommendRule/PreviewMatch` 返回命中列表：
      - 展示规则 ID、推荐分类、优先级、三项命中标记（风险/初筛/症状）以及一段“命中原因说明”文本，方便运营理解规则生效原因。

## 三、后端实现说明（Spring Boot）

### 3.1 相关表与 Mapper

- 食物分类表：`TLaryngealFoodCategory` + `TLaryngealFoodCategoryMapper`

- 食物表：`TLaryngealFood` + `TLaryngealFoodMapper`
  - 增量脚本：`alter_tlaryngealfood_extend_fields.sql`
    - 新增字段：`FriendlyLevel`（友好度）、`TasteTag`（口味标签）、`SuitPeople`、`AvoidPeople`。
    - 新增索引：`idx_lfood_category`、`idx_lfood_friendly`。
- 用户饮食记录表：`TUserDietRecord` + `TUserDietRecordMapper`
  - 增量脚本：`alter_tuserdietrecord_extend_fields.sql`
    - 新增字段：`MealType`（餐次）、`FeelingCode`（主观感受枚举）、`SourceType`（来源：前台/后台/导入）。
    - 新增索引：`idx_diet_user_time`（按用户+时间查询）。
- 饮食推荐规则表：`TDietRecommendRule` + `TDietRecommendRuleMapper`
  - 增量脚本（最终版）：`alter_tdietrecommendrule_optimize_fields_20260306.sql`
    - 将 `RecommendPriority` 从 `TINYINT(1)`（布尔）调整为 `INT(11)`。
    - 新增 `RiskLevelScope`（字符串，如 `LOW,MID` 或 `0,1`）与 `IsEnabled`（是否启用）。
    - 对历史数据进行默认值填充（优先级、启用状态）。
- 饮食计划模板相关表：
  - `TDietPlanTemplate`、`TDietPlanTemplateItem` + 对应 Mapper
  - 通过脚本 `create_tdietplantemplate_tables.sql` 创建，为后续“标准化饮食方案模版”预留扩展能力（当前版本核心流程尚未完全接入前台）。
- 健康档案与症状、音频记录：
  - `TPersonalLaryngealHealthRecord`、`TSymptomLog`、`TAudioScreenRecord`
  - 在饮食推荐中用于获取风险等级、最近症状、最近一次初筛结果。

### 3.2 前台 DTO 与 Service

#### 3.2.1 前台 DTO（`FrontDietDtos.java`）

  - `FriendlyCategories` / `AvoidCategories`：推荐的友好/忌口分类列表（每项含代表性食物）。
- `FrontDietFoodItem` / `FrontDietFoodListInput` / `FrontDietFoodDetailInput` / `FrontDietFoodDetailDto`
  - 支撑食物列表与详情接口。
- `FrontDietRecordCreateOrEditInput` / `FrontDietRecordListInput` / `FrontDietRecordListItem` / `FrontDietRecordDeleteInput`
  - 支撑饮食记录新增、编辑、列表与删除。

#### 3.2.2 前台 Service（`FrontDietService`）

位置：`HealthControl.springboot/src/main/java/com/example/web/service/front/FrontDietService.java`  
注入的核心 Mapper：

- `TPersonalLaryngealHealthRecordMapper`
- `TAudioScreenRecordMapper`
- `TSymptomLogMapper`
- `TDietRecommendRuleMapper`
- `TLaryngealFoodCategoryMapper`
- `TLaryngealFoodMapper`
- `TUserDietRecordMapper`

主要方法：

- `getDietHomeData(Integer currentUserId)`
  - 组装首页所需的 `FrontDietHomeDto`。
  - 逻辑步骤：
    1. 查询用户最近一条健康档案，获取 `RiskAssessmentLevel`。
    2. 查询最近一次音频初筛记录，获取 `PrimaryScreenResult`（良性/恶性）。
    3. 查询最近一条症状日志，获取 `SymptomType` 列表。
    4. 根据以上信息调用内部方法 `buildRecommendCategories`，命中 `TDietRecommendRule` 并返回推荐分类。
    5. 统计“今天已记录饮食次数”作为额外信息。
- `buildTodayStatus(...)`
  - 根据风险等级、最近检测和症状等信息生成一段友好的中文文案，展示在 `TodayStatus` 卡片中。
- `buildRecommendCategories(Integer userId, boolean generalOnly, List<FrontDietRecommendCategory> friendly, List<FrontDietRecommendCategory> avoid)`
  - 核心推荐逻辑：
    - 从 `TDietRecommendRule` 中取出未删除且启用的规则。
    - 按以下条件过滤命中：
      - 风险等级范围：`RiskLevelScope` 支持 `ANY/LOW/MID/HIGH` 或 `0/1/2`。
      - 初筛关系：`PrimaryScreenRelation` 支持 “良性/恶性/不限”。
      - 症状集合：用户与规则的症状集合求交集。
    - 收集命中规则对应的 `RecommendCategoryId`，再基于 `TLaryngealFoodCategory` 与 `TLaryngealFood` 取回代表性食物。
    - 若无任何规则命中，则回退到所有 `ShowStatus=1` 分类的通用推荐。
- `getFoodList(...)`
  - 根据 `CategoryType`、`Keyword`、分页参数等条件查询 `TLaryngealFood`，并拼装为前台 `FrontDietFoodItem` 列表。
- `getFoodDetail(...)`
  - 根据 `FoodId` 获取单个 `TLaryngealFood` 实体，映射为 `FrontDietFoodDetailDto`。
  - 同时根据用户风险等级与食物友好度生成额外提示文案。
- `recordCreateOrEdit(...)`
  - 支持新增与更新饮食记录：
    - 根据是否有 `Id` 判断是新增还是编辑。
    - 填充用户 ID、时间、感受枚举等字段。
- `recordList(...)`
  - 按用户 ID 与时间范围分页查询饮食记录。
  - 返回列表项中包含食物名称、分类标签与主观感受说明。
- `recordDelete(...)`
  - 软删除记录：设置 `IsDelete=1`，不做物理删除。

### 3.3 后台规则管理 Service（`TDietRecommendRuleServiceImpl`）

位置：`HealthControl.springboot/src/main/java/com/example/web/service/impl/TDietRecommendRuleServiceImpl.java`  

关键点：

- **字段支持与查询构造**
  - `BuilderQuery(TDietRecommendRulePagedInput input)` 中增加对 `RiskLevelScope` 与 `IsEnabled` 的过滤。
  - 默认按 `CreationTime` 倒序排序，也支持前端传入动态排序字段。
- **软删除实现**
  - `Delete(IdInput)` 与 `BatchDelete(IdsInput)` 中将 `IsDelete` 置为 `true`，而不是物理删除。
- **批量启停**
  - `BatchSetEnabled(IdsEnabledInput)`：
    - 使用 `UpdateWrapper` 通过 `Id in (...)` 批量更新 `IsEnabled` 字段。
- **规则命中预览（PreviewMatch）**
  - 方法签名：`List<DietRulePreviewItemDto> PreviewMatch(DietRulePreviewInput input)`。
  - 入参 `DietRulePreviewInput`：
    - `RiskLevel`（0/1/2 或 null 不限）
    - `PrimaryScreenResult`（良性/恶性/不限）
    - `SymptomType`（逗号分隔）
  - 实现逻辑：
    1. 将 `PrimaryScreenResult` 文本转换为布尔/空（良性=FALSE，恶性=TRUE，null=不限）。
    2. 查询所有未删除且启用的规则。
    3. 对每条规则分别计算：
       - `matchRisk`：调用内部 `isRiskScopeMatch(scope, riskLevel)`。
       - `matchPrimary`：根据 `PrimaryScreenRelation` 与用户初筛结果判断。
       - `matchSymptom`：根据症状集合是否有交集判断。
    4. 若三者均为 true，则认为该规则被命中，生成一条 `DietRulePreviewItemDto`：
       - 包含规则 ID、推荐分类 ID/名称、优先级、启用状态，以及各项匹配标记。
       - 通过 `buildExplainText(...)` 生成一段“命中原因说明”，串起风险/初筛/症状三方面信息。
    5. 最终结果按 `RecommendPriority` 升序、`RuleId` 升序排序返回。

### 3.4 后台 Controller（`TDietRecommendRuleController`）

位置：`HealthControl.springboot/src/main/java/com/example/web/controller/TDietRecommendRuleController.java`

### 4.1 小程序页面与路由

`HealthControl.uniapp/pages.json` 已注册饮食相关页面：

- `pages/Front/DietHome`
- `pages/Front/DietFoodList`
- `pages/Front/DietFoodDetail`

首页入口：

- `Index.vue`：新增“饮食建议”功能卡片，点击跳转 `/pages/Front/DietHome`。

### 4.2 小程序端 HTTP 与登录态复用

与健康管理与教育模块保持一致：

- HTTP 封装：`HealthControl.uniapp/utils/http.js`
  - 统一 `Post` 方法，自动注入 `Authorization` 头。
  - 响应结构统一为 `{ Code, Success, Msg, Data }`。
- 登录态管理：`store/index.js` 中的 `commonStore`
  - 提供 `Token/UserInfo/RoleType/UserId` 等计算属性。
  - 在饮食模块接口中直接复用。

### 4.3 具体页面实现

- `DietHome.vue`
  - 在 `onLoad` 或页面展示时调用 `/Front/Diet/Home`。
  - 将 `TodayStatus` 渲染为顶部卡片，风险等级采用颜色区分（低=绿色、中=橙色、高=红色）。

  - 底部“我的饮食记录”按钮跳转到 `DietRecordList`。

- `DietFoodList.vue`
  - 顶部 Tab 控制当前 `CategoryType`。
  - 搜索框变更/确认时重置分页并重新加载数据。
  - 使用下拉刷新与上拉触底实现分页加载 `FoodList`。

- `DietFoodDetail.vue`
  - 页面加载时根据路由参数 `FoodId` 调用 `/Front/Diet/FoodDetail`。
  - 使用图文布局展示食物详情与喉部友好/危害说明。
  - 底部“加入今天的饮食记录”按钮打开 `uni-popup`：
    - 表单字段由 `FrontDietRecordCreateOrEditInput` 驱动。
    - 提交后若成功，弹出 `uni.showToast` 并关闭弹窗。

- `DietRecordList.vue`
  - 页面顶部允许选择日期范围，变化时重新加载记录。
  - 列表项提供：
    - “编辑”按钮：弹出表单修改主观感受，调用 `/Front/Diet/RecordCreateOrEdit`。
    - “删除”按钮：弹出确认框后调用 `/Front/Diet/RecordDelete`。

### 4.4 管理端规则配置页面（`TDietRecommendRuleList.vue`）

- 使用统一的 `PaginationTable` 组件呈现规则列表，支持勾选、多选操作。
- 搜索区域：
  - 输入框 + 下拉框组合，支持对症状、初筛关联、风险范围、启用状态等条件过滤。
- 编辑弹窗：
  - 使用 `el-form` + `el-select` + `el-input-number` + `el-switch` 组合编辑一条规则的所有关键字段。
  - 新增时，自动填充：
    - `CreatorId` 为当前登录管理员。
    - `IsEnabled=true`。
    - `RecommendPriority=3`。
    - `PrimaryScreenRelation="不限"`。
- 批量启停：
  - 工具栏 `批量启用/批量停用` 按钮：
    - 收集已勾选规则的 `Id` 列表，调用 `/TDietRecommendRule/BatchSetEnabled`。
    - 成功后刷新表格，并通过 `ElMessage` 提示结果。
- 规则命中预览：
  - 新增“规则命中预览”按钮打开 `el-dialog`：
    - 表单输入：模拟风险等级、模拟初筛结果、模拟症状。
    - 点击“开始预览”调用 `/TDietRecommendRule/PreviewMatch`。
    - 下方 `el-table` 展示命中结果，并用 `el-tag` 可视化展示三项匹配情况。

1. **确保后端已启动且数据库已执行饮食模块相关增量脚本**：
   - `alter_tlaryngealfood_extend_fields.sql`
   - `alter_tuserdietrecord_extend_fields.sql`
   - `alter_tdietrecommendrule_optimize_fields_20260306.sql`
   - `create_tdietplantemplate_tables.sql`（如需使用饮食计划模板相关功能）
2. **确认 uni-app 环境指向正确后端**：
   - 在 `HealthControl.uniapp/utils/config.js` 中配置 `API_BASE_URL`。
3. **使用 HBuilderX 运行到微信小程序或 H5**，并确保可以正常登录。

### 5.2 饮食推荐首页与规则生效

1. 登录后，从首页点击“饮食建议”卡片进入 `DietHome`。
2. 观察顶部“今日喉部状态”：是否展示风险等级与简要说明。
3. 查看“适合吃/建议少吃”两个区块是否有分类与食物列表：
   - 若为空，可先在后台补充 `TLaryngealFoodCategory` 与 `TLaryngealFood` 数据，并确保 `ShowStatus=1`。
4. 在后台 `TDietRecommendRuleList` 中增加/修改规则，调整 `SymptomType/PrimaryScreenRelation/RiskLevelScope` 后：
   - 回到小程序中重新进入 `DietHome`，观察“推荐/忌口分类”是否随规则变化（可配合风险问卷或音频初筛改变用户风险等级）。

### 5.3 食物库浏览与搜索

1. 从 `DietHome` 点击“查看全部友好食物”进入 `DietFoodList`：
   - 顶部 Tab 显示当前是“友好食物”。
2. 在搜索框输入关键词（如“蜂蜜”“柠檬”），检查列表是否过滤。
3. 切换到“忌口食物”Tab，确认列表显示的是另一类食物。
4. 上拉到底部，若数据较多，应触发分页加载；否则显示“没有更多了”。

### 5.4 食物详情与快速记录

1. 在食物列表中点击任意食物进入 `DietFoodDetail`：
   - 确认显示了图片、名称、分类与友好/危害说明。
2. 点击“加入今天的饮食记录”：
   - 弹出表单，可修改时间与主观感受。
3. 点击确定：
   - 若成功，应看到 Toast 提示“记录成功”。
   - 稍后在“我的饮食记录”中可看到该条记录。

### 5.5 饮食记录列表的查看、编辑与删除

1. 从 `DietHome` 底部进入 `DietRecordList`：
   - 默认展示最近若干天的记录。
2. 修改日期范围（如切换到最近 7 天/30 天），列表随之刷新。
3. 点击某条记录的“编辑”按钮：
   - 弹出表单修改主观感受，提交后该条记录应被更新。
4. 点击“删除”按钮：
   - 确认后，该条记录应从列表中消失（数据库中为软删除）。

### 5.6 后台规则命中预览与前台验证

1. 在 ElementUI 后台打开 `TDietRecommendRuleList` 页面。
2. 点击“规则命中预览”：
   - 输入一组模拟条件（如：风险=高(2)，初筛=恶性，症状=咽干,咽痒）。
   - 点击“开始预览”，观察弹出的表格中有哪些规则被命中，以及原因说明。
3. 根据预览结果，记住被命中的规则关联的分类。
4. 在实际用户环境中，通过修改用户风险等级/症状（可通过风险问卷与症状日志模块）：
   - 再次进入小程序 `DietHome`，观察推荐/忌口分类是否与预览结果一致，从而验证整体规则链路正确。

## 六、常见问题与排查

### 6.1 首页没有推荐分类或食物

- 检查是否存在 `ShowStatus=1 && IsDelete=0` 的 `TLaryngealFoodCategory` 与对应食物。
- 检查 `TDietRecommendRule` 中是否有启用中的规则：
  - 若没有任何规则命中，则系统会回退到“所有可见分类”的通用推荐。

### 6.2 饮食记录看不到或时间范围不对

- 确认 `alter_tuserdietrecord_extend_fields.sql` 已执行，索引创建成功。
- 检查前台传入的日期范围是否正确（时区问题、起止时间顺序）。

### 6.3 规则命中预览与前台行为不一致

- 在后台使用 `PreviewMatch` 时，确认使用的模拟条件与实际用户条件一致：
  - 风险等级：来自 `TPersonalLaryngealHealthRecord.RiskAssessmentLevel`。
  - 初筛结果：来自最近一次 `TAudioScreenRecord.PrimaryScreenResult`。
  - 症状：来自 `TSymptomLog.SymptomType`。
- 确认规则的 `RiskLevelScope` 写法是否正确（如使用 `LOW,MID` 或 `0,1`）。
- 确认规则 `IsEnabled=true` 且 `IsDelete=false`。

### 6.4 食物详情提示文案不符合预期

- 检查 `TLaryngealFood` 中是否正确配置了 `FriendlyLevel/SuitPeople/AvoidPeople`。
- 如需更细粒度控制提示文案，可在 `FrontDietService.getFoodDetail` 中扩展映射逻辑。

## 七、后续可优化点

1. **与健康档案的深度联动**  
   根据长期饮食行为与喉部健康趋势，自动给出“你与同龄人相比更偏爱辛辣/过烫”的客观反馈，并调整推荐权重。

   在后台增加饮食模块的运营报表，如“最常记录的食物 TOP N”“高风险用户常见忌口行为”等，辅助调整规则与内容。
