## 饮食管理与推荐模块（喉部友好 / 忌口 + 个性化推荐）需求文档

> 版本：v1.0  
> 编写日期：2026-03-06  
> 适用项目：居家喉部健康自查小程序（`HealthControl.uniapp` / `HealthControl.springboot` / `HealthControl.elementui` / `voice`）  
> 关联数据库：`healthcontrol`（见 `healthcontrol.sql` 中 `tlaryngealfoodcategory`、`tlaryngealfood`、`tuserdietrecord`、`tdietrecommendrule` 等）  
> 现有后端基础：SpringBoot + MyBatisPlus 通用 CRUD；前端基础：UniApp 小程序端 + ElementUI 管理端；音频自查与健康档案已实现。  

---

## 一、模块定位与目标

### 1.1 模块定位

围绕“**喉部友好饮食 + 忌口管理 + 个性化推荐**”，提供：

- **食物知识库**：按“友好/忌口”分类管理食物，说明对喉部的影响；
- **饮食记录**：用户可记录每日摄入的关键食物与喉部感受；
- **智能饮食推荐**：结合症状、音频自查结果、风险等级，按规则推荐“适合吃什么 / 建议少吃什么”；
- **运营配置能力**：管理端可维护食物库与推荐规则，持续优化推荐效果。

### 1.2 业务目标（可量化）

- 提升用户**饮食记录使用率**（活跃用户中 ≥ 30% 曾添加饮食记录）；
- 高风险或症状明显用户中，**被推荐“忌口”食物后实际减少摄入的比例**；
- 建立可持续运营的食物库和规则体系，方便后续医生/营养师参与协作。

### 1.3 与其他模块的关系

- 与 **音频自查模块**：根据 `taudioscreenrecord.PrimaryScreenResult/PrimaryScreenConfidence` 决定推荐强度；
- 与 **个人喉部健康档案**：根据 `tpersonallaryngealhealthrecord.RecentSymptom/RiskAssessmentLevel` 调整推荐内容；
- 与 **个性化提醒模块**：在提醒中可附带“饮食小贴士”链接跳转到推荐页；
- 与 **健康管理与教育模块**：饮食推荐页可挂接相关科普文章（如“咽干忌口清单”、“护嗓食谱”）。

## 二、范围与角色

### 2.1 用户角色

- **普通用户（RoleType=2）**  
  - 浏览喉部友好/忌口食物；  
  - 记录每日饮食；  
  - 查看系统基于症状+风险的个性化推荐。  
- **管理员（RoleType=1）**  
  - 维护食物分类、食物条目、图片及文案；  
  - 配置饮食推荐规则（症状+初筛结果 → 推荐食物分类）；  
  - 查看推荐规则命中情况与用户饮食行为统计（后续可拓展）。  

### 2.2 业务边界

- 不涉及完整营养学配餐，仅围绕“对喉部健康的友好/危害”进行推荐；
- 不直接给出“诊断级饮食处方”，所有文案需带有“仅供参考，遵医嘱为准”的免责声明；
- 推荐结论不自动改变任何医疗建议与就医推荐，仅作为日常生活辅助。

## 三、数据库现状与差距

### 3.1 已有表（来自 `healthcontrol.sql`）

- **喉部食物库分类表 `tlaryngealfoodcategory`**
  - 核心字段：
    - `CategoryName`：如“润喉类/辛辣类”；
    - `CategoryType`：`0=友好`，`1=忌口`；
    - `CategoryDesc`、`SortNum`、`ShowStatus`、`IsDelete`。
- **喉部食物库表 `tlaryngealfood`**
  - 核心字段：
    - `CategoryId`：关联分类；
    - `FoodName`、`FoodAlias`；
    - `EffectHarmDesc`：功效/危害说明；
    - `SuggestContent`：忌口/食用建议；
    - `PicUrl`、`ShowStatus`、`IsDelete`。  
- **用户饮食记录表 `tuserdietrecord`**
  - 核心字段：
    - `UserId`、`FoodId`、`IntakeTime`、`IntakeFrequency`；
    - `EatFeeling`（食用后喉部感受），`IsDelete`。
- **饮食推荐规则表 `tdietrecommendrule`**
  - 核心字段：
    - `SymptomType`：匹配症状，例如“咽干,咽痒”；
    - `PrimaryScreenRelation`：`良性/恶性/不限`；
    - `RecommendCategoryId`：推荐的食物分类；
    - `RecommendPriority`：1=最高，5=最低；
    - `RuleDesc`、`IsDelete`。

### 3.2 主要差距（为满足目标能力需补充/约定）

- **缺少食物友好程度标签**（如“非常友好/一般友好/轻度刺激/强刺激”等），目前只能通过 `CategoryType` + 文案表达；
- **缺少规则作用范围说明**（如适用于“高风险用户优先”还是“所有用户”），可通过文案与枚举约定弥补；
- **缺少统计字段/索引**（如按 `UserId+IntakeTime` 统计一日饮食，按 `SymptomType` 聚合规则），可通过新增索引/视图后续优化；
- **缺少“症状 + 初筛结果”与推荐联动的明确约定**（虽有 `tdietrecommendrule`，但未定义具体匹配逻辑）。  

> 本需求文档 **暂不强制改表**，优先通过现有结构与索引约定实现；如后续需要可另行编写 SQL 脚本（类似 `健康管理与教育模块-需求.md`）。

## 四、小程序端（UniApp）功能需求

### 4.1 信息架构与页面

建议在 `HealthControl.uniapp` 中新增/补充以下页面（命名示例）：

- `pages/Front/DietHome.vue`：饮食推荐首页（总入口）；
- `pages/Front/DietFoodList.vue`：按分类浏览食物列表；
- `pages/Front/DietFoodDetail.vue`：单个食物详情页；
- `pages/Front/DietRecordList.vue`：我的饮食记录列表；
- `pages/Front/DietRecordEdit.vue`（可选，亦可与列表页合并使用弹窗）：新增/编辑饮食记录；
- （可选）`pages/Front/DietRecommendExplain.vue`：推荐逻辑与免责声明说明。  

入口建议：

- 首页卡片：“饮食建议” → 跳转 `DietHome`；
- 健康档案页：在个人档案概览下增加“饮食建议入口”；
- 个性化提醒推送点击行为（`messagenotice.ActionType=VOICE_CARE` 等）可跳转到 `DietHome`。  

### 4.2 DietHome（饮食推荐首页）

**页面目标**：给用户一个“一眼看懂今天适合/不适合吃什么”的概览，并引导其记录饮食。

#### 4.2.1 结构布局

- 顶部导航：标题“饮食建议”，右侧“饮食记录”按钮（跳转 `DietRecordList`）；
- 内容区：
  1. **今日喉部状态卡片**  
     - 数据来源：
       - 最近一次音频自查记录（`taudioscreenrecord`）结果；
       - 健康档案风险等级（`tpersonallaryngealhealthrecord.RiskAssessmentLevel`）；
       - 最近一次主要症状（`tsymptomlog`）。  
     - 展示：
       - 标题：“今日喉部状态预估：低/中/高风险”；
       - 简要说明（如“近期咽干 + 轻度嘶哑，建议减少辛辣刺激食物”）；
       - CTA 按钮：“查看详细档案”→档案页。
  2. **推荐友好食物卡片区**  
     - 取 `tdietrecommendrule` 中符合当前症状+初筛结果的若干条规则，汇总其 `RecommendCategoryId` 对应的 `tlaryngealfoodcategory`；
     - 展示 3–6 个“更适合吃”的分类卡片（如“润喉水果”、“温和汤水”）；
     - 每个卡片内显示 2–3 个代表性食物（来自 `tlaryngealfood`），点击进入 `DietFoodList`/`DietFoodDetail`。
  3. **推荐忌口食物卡片区**  
     - 类似友好食物区，但展示 `CategoryType=1`（忌口）下、被规则命中的分类；
     - 文案风格：温和提醒而非恐吓，如“建议近期减少：油炸/辛辣等，会加重咽部刺激”。
  4. **饮食记录入口**  
     - 显示今日已记录的关键食物数量（统计 `tuserdietrecord` 中当日记录）；
     - CTA：“记录今天吃了什么”。

#### 4.2.2 UI 规范（与整体项目一致）

- 使用主色 `#11998e`、辅色 `#38ef7d`；
- 卡片：白底、圆角 ~16rpx，阴影 `0 4px 12px rgba(0,0,0,0.06)`；
- 友好食物卡片使用偏绿色角标（如“推荐”），忌口食物使用暖色/橙红角标（如“少吃”）；
- 重要提示区（如“高风险请尽快就医”）使用轻量黄色背景，不制造焦虑。  

### 4.3 DietFoodList（食物列表）

#### 4.3.1 列表功能

- 支持按 `CategoryType` 切换：
  - Tab：`友好食物` / `需要注意`（或“忌口”）；
  - 根据 `tlaryngealfoodcategory.CategoryType` 过滤。  
- 支持按分类过滤：
  - 顶部下拉/横向滚动分类标签；
  - 分类来自 `tlaryngealfoodcategory`，仅展示 `ShowStatus=1 && IsDelete=0`。  
- 支持关键字搜索：
  - 搜索 `FoodName/FoodAlias`；
  - 输入空字符串视为清空搜索。  
- 列表项展示：
  - 食物图片（`PicUrl`，缺省时用占位图）；
  - 名称 + 别名；
  - 简要描述（从 `EffectHarmDesc` 截取前 ~30 字）；
  - 小标签：如“润喉”“温热”“辛辣”“油炸”等（可从 `CategoryName`/`EffectHarmDesc` 中提炼，MVP 可先由运营写入描述文本，不做结构化）。  

#### 4.3.2 交互

- 点击列表项 → `DietFoodDetail`；
- 下拉刷新/上拉加载分页；
- 空态展示：“这里还没有收录食物，欢迎稍后再来或联系管理员补充”。  

### 4.4 DietFoodDetail（食物详情）

#### 4.4.1 展示内容

- 食物名称、别名；
- 所属分类（含 CategoryType 解释：友好/忌口）；
- 图片大图；
- 「对喉部的影响」：展示 `EffectHarmDesc`，分段排版；
- 「食用建议」：展示 `SuggestContent`，明确“适合频率/不适合人群”；
- 「相关提醒」：
  - 根据当前用户风险等级/症状显示简短提示，例如：
    - 高风险 + 忌口类：标红“近期建议减少摄入”；
    - 低风险 + 友好类：标绿“适量摄入有助于缓解不适”。  
- CTA 按钮：
  - “加入今天的饮食记录” → 打开记录表单（带默认 FoodId）。  

#### 4.4.2 记录饮食表单（弹窗或新页）

- 字段：
  - 食物名称（只读）；
  - 摄入时间（默认当前时间，可选择）；
  - 当日摄入频次（1、2、3、3+ 等，可用 `el-input-number` / picker）；
  - 食后感受（`EatFeeling`，自由文本或预设选项：`缓解不适/无明显变化/稍有不适/明显加重`）。  
- 提交后：
  - 写入 `tuserdietrecord`；
  - toast“已记录，感谢反馈，这将帮助系统给你更合适的饮食建议。”。  

### 4.5 DietRecordList（我的饮食记录）

#### 4.5.1 列表功能

- 按日期倒序展示用户近 N 天饮食记录（默认近 7–14 天，可分页）；
- 每条记录展示：日期时间、食物名称、分类（友好/忌口）、感受标签（若有）；
- 支持按日期筛选、按分类类型筛选（只看“忌口类记录”等）。  

#### 4.5.2 交互

- 点击记录可展开/进入编辑页修改 `EatFeeling`；
- 支持删除（软删除：`IsDelete=1`）；
- 底部可显示简单统计：
  - 近 7 天友好食物比率；
  - 忌口食物出现次数；
  - 引导文案：“你最近辛辣摄入稍多，建议结合自查结果适当控制。”。

- 批量调整分类（如“全部移动到 忌口·辛辣类”）。  

### 5.3 饮食推荐规则管理（`tdietrecommendrule`）

#### 5.3.1 列表视图

- 字段：
  - `SymptomType`：以逗号分隔的症状列表（如“咽干,咽痒,嘶哑”）；
  - `PrimaryScreenRelation`：`良性/恶性/不限`；
  - `RecommendCategoryId`+分类名称；
  - `RecommendPriority`：1–5；
  - `RuleDesc`；
  - `IsDelete`。  

#### 5.3.2 规则编辑

- SymptomType：
  - 管理端提供多选控件，值域与 `tsymptomlog.SymptomType` 保持一致（如“咽干/咽痒/嘶哑/疼痛/吞咽困难”等），保存时用逗号拼接；
  - 允许填写“通配”症状（如 `ANY`），表示不限制症状（可选）。  
- PrimaryScreenRelation：枚举下拉：`BENIGN/MALEIGNANT/ANY`（存表继续用“良性/恶性/不限”中文亦可，但前后端需统一）；
- RecommendCategoryId：下拉选择 `tlaryngealfoodcategory`；
- RecommendPriority：1（最高）到 5（最低）；
- RuleDesc：对运营/医生说明该规则的依据与使用场景。  

#### 5.3.3 匹配逻辑约定（供后端实现）

> 仅描述逻辑，不在本需求中写具体代码。

- 当前用户上下文：
  - 症状：从最近一次 `tsymptomlog` 获取 `SymptomType`；
  - 初筛结果：从最近一次 `taudioscreenrecord.PrimaryScreenResult` 获取良性/恶性；
  - 若无检测记录，则 PrimaryScreenRelation 匹配 `不限` 的规则。  
- 规则命中：
  - 若 `SymptomType` 为空：仅匹配 `PrimaryScreenRelation`；
  - 若不为空：用户症状集合与规则的症状集合存在交集即可视为命中；
  - 多条规则命中时：按 `RecommendPriority` 升序、`CreationTime` 倒序排序。  
- 推荐输出：
  - 友好推荐列表：从命中规则的 `RecommendCategoryId` 中选取 `CategoryType=0` 的分类；
  - 忌口提醒列表：同理选取 `CategoryType=1` 的分类。  

### 5.4 用户饮食记录运营视图（可选，P2）

- 简单列表：按日期/用户查看饮食记录；
- 支持按食物分类筛选（例如“查看所有摄入 忌口·辛辣类 的用户”）；
- 可导出 CSV 供后续分析，不做复杂可视化。

## 六、后端接口设计建议（SpringBoot）

> 延续现有风格：`{ Code, Success, Msg, Data }`，分页返回 `Data.Items + Data.TotalCount`。管理端继续使用 `/TLaryngealFood*` 等 CRUD；小程序端建议新增 `/Front/Diet/*` 前缀接口。  

### 6.1 小程序前台接口（建议）

1. `POST /Front/Diet/Home`  
   - 入参：空或可选 `{ UserId }`（从登录态取）；  
   - 返回：
     - `TodayStatus`：风险等级、最近症状摘要、推荐文案；
     - `RecommendFriendlyCategories[]`：每项包含 Category 与若干代表食物；
     - `RecommendAvoidCategories[]`：同上；
     - `TodayRecordStat`：今日已记录条数。  

2. `POST /Front/Diet/FoodList`  
   - 入参：`Page, Limit, CategoryType(0/1), CategoryId(可选), Keyword(可选)`；  
   - 返回：食物列表（Id, 名称, 别名, 分类, PicUrl, 摘要等）。  

3. `POST /Front/Diet/FoodDetail`  
   - 入参：`FoodId`；  
   - 返回：`tlaryngealfood` 全字段 + 所属分类信息 + 根据当前用户状态生成的“风险提示文案”。  

4. `POST /Front/Diet/RecordCreateOrEdit`  
   - 入参：`Id(可选,编辑时)`, `FoodId`, `IntakeTime`, `IntakeFrequency`, `EatFeeling`；  
   - 行为：插入或更新 `tuserdietrecord`。  

5. `POST /Front/Diet/RecordList`  
   - 入参：`Page, Limit, DateStart(可选), DateEnd(可选), CategoryType(可选)`；  
   - 返回：用户饮食记录列表 + 简单统计信息。  

6. `POST /Front/Diet/RecordDelete`  
   - 入参：`Id`；  
   - 行为：设置 `IsDelete=1`。  

### 6.2 管理端扩展接口（建议）

- 在现有 `TLaryngealFoodCategoryController`、`TLaryngealFoodController` 基础上：  
  - 补充分页筛选条件（CategoryType、ShowStatus 等）；

## 七、枚举与文案约定

### 7.1 分类类型（`tlaryngealfoodcategory.CategoryType`）

- `0`：喉部友好（推荐适量/适当多吃）；  
- `1`：喉部忌口（建议减少或避免）。  

### 7.2 推荐规则初筛结果匹配（`tdietrecommendrule.PrimaryScreenRelation`）

- `"良性"`：仅当最近一次 `PrimaryScreenResult=0` 时生效；  
- `"恶性"`：仅当最近一次 `PrimaryScreenResult=1` 时生效；  
- `"不限"`：不关心初筛结果，一律可用。  

### 7.3 前端统一提示风格

## 十二、核心实现流程与状态机设计（详细）

本章从“请求入口 → 数据查询 → 规则匹配 → 结果组装 → 写入/统计”五个维度细化主要功能的实现逻辑，确保后续编码行为一致、可预期。

### 12.1 DietHome 首页数据流（`POST /Front/Diet/Home`）

#### 12.1.1 今日状态计算（TodayStatus）

- 未登录用户：
  - 不读取任何个人数据（档案/检测/症状/记录）；
  - `TodayStatus` 仅返回“未登录提示”与跳转登录的 CTA；
  - 推荐区仅使用“通用规则”（`SymptomType` 为空或含 `ANY`，`RiskLevelScope` 包含 `ANY`）生成基础建议。
- 已登录用户：
  1. **健康档案**：从 `tpersonallaryngealhealthrecord` 中按 `UserId` + `IsDelete != 1`、`CreationTime`/`LastScreenTime` 倒序取 1 条，得到 `RiskAssessmentLevel`（0/1/2 → 低/中/高）。
  2. **最近检测**：从 `taudioscreenrecord` 中按 `UserId` + `IsDelete != 1`、`CreationTime` 倒序取 1 条，将 `PrimaryScreenResult` 映射为“良性/恶性/未知”，保留置信度。
  3. **最近症状**：从 `tsymptomlog` 中按 `UserId` + `IsDelete != 1`、`CreationTime` 倒序取 1 条，获取 `SymptomType/SymptomLevel/SymptomDuration`。
  4. **汇总生成**：
     - `RiskLevel`：来自档案；
     - `PrimaryScreenResultText/PrimaryScreenConfidence`：来自检测；
     - `MainSymptom`：最近症状类型；
     - `SummaryText`：根据风险等级 + 初筛结果 + 症状组合，走一套路由表生成，例如：
       - 高风险 + 明显刺激性症状 → “近期喉部较敏感，请格外注意辛辣油炸等刺激饮食”；
       - 低风险 + 无症状 → “当前整体状况较平稳，可按推荐多选择友好食物，减少偶发刺激食物”。

#### 12.1.2 推荐分类与食物生成

1. **加载规则基表**：从 `tdietrecommendrule` 取 `IsDelete != 1 AND (IsEnabled IS NULL OR IsEnabled = 1)` 的记录。
2. **按初筛结果过滤**：
   - 若最近检测为“良性”：保留 `PrimaryScreenRelation IN ('良性','不限')`；
   - 为“恶性”：保留 `IN ('恶性','不限')`；
   - 为“未知”或无检测：仅保留 `='不限'`。
3. **按风险等级过滤（可选）**：
   - 将用户 `RiskAssessmentLevel` 映射为 `LOW/MID/HIGH`；
   - 若规则 `RiskLevelScope` 非空，则需包含当前等级或 `ANY` 才视为可用。
4. **按症状集合匹配**：
   - 用户症状集 `S_user`：将最近一次 `SymptomType` 按逗号拆分去重；
   - 规则症状集 `S_rule`：同样拆分；
   - 若 `S_rule` 为空或包含 `ANY`：视为对所有症状适用；
   - 否则仅当 `S_user ∩ S_rule` 非空时才命中。
5. **规则排序与去重**：
   - 对命中的规则按 `RecommendPriority` 升序、`CreationTime` 倒序排序；
   - 收集其 `RecommendCategoryId`，去重形成分类 Id 列表。
6. **查询分类与代表性食物**：
   - `tlaryngealfoodcategory`：`Id IN 列表 AND ShowStatus=1 AND IsDelete=0`；
   - 按 `CategoryType` 拆成“友好推荐列表”（0）与“忌口列表”（1）；
   - 对每个分类，从 `tlaryngealfood` 取若干代表性食物：
     - 条件：`CategoryId=当前分类 AND ShowStatus=1 AND IsDelete=0`；
     - 排序建议：`FriendlyLevel` 升序（友好优先）、其次 `CreationTime` 倒序；
     - 截断数量为 2–3 条。
7. **输出结构**：
   - `RecommendFriendlyCategories[]` / `RecommendAvoidCategories[]`：
     - 每项包含分类基础字段 + 代表性食物的精简字段（Id、名称、PicUrl、FriendlyLevel 等）。

#### 12.1.3 今日饮食记录统计

- 从 `tuserdietrecord` 读取：
  - 条件：`UserId=当前用户 AND IsDelete != 1 AND DATE(IntakeTime)=CURDATE()`；
  - 统计总条数 → `TodayRecordStat`；
  - 如需要，可 JOIN `tlaryngealfood -> tlaryngealfoodcategory.CategoryType` 聚合出“友好/忌口次数”，用以构建顶部提示（例如“今日已记录 1 次友好食物、2 次忌口食物”）。

### 12.2 饮食记录创建/编辑流程（`/Front/Diet/RecordCreateOrEdit`）

#### 12.2.1 入参与业务校验

- 必填：`FoodId`；从 `tlaryngealfood` 校验存在且 `IsDelete != 1`，否则返回 `Diet.FoodNotFound`。
- 时间：
  - `IntakeTime` 为空时默认 `NOW()`；如传入，需保证能正确解析为 `DATETIME`。
- 频次：
  - `IntakeFrequency` 为空默认 1；小于 1 时归一为 1，大于如 10 时截断为上限，并在 `Msg` 中友好提示。
- 用餐类型（如已改表）：
  - `MealType` 限定在 0–3 之间（早餐/午餐/晚餐/加餐）；空值视为“未知”，后续可用作统计过滤。
- 食后感受：
  - `FeelingCode` 需在 {0,1,2,3,4} 范围内（0=未填写）；`EatFeeling` 文本仅为补充，不参与判定。

#### 12.2.2 新建与编辑逻辑

- 新建：
  - 插入新记录，写入：
    - `CreationTime=NOW()`；
    - `UserId=当前用户`；
    - 其他字段来自入参；
    - `IsDelete=0`。
  - 不做幂等限制，同一时间可针对同一食物多次记录（表示多次摄入）。
- 编辑：
  - 先按 `Id` + `UserId=当前用户` + `IsDelete != 1` 查找记录；
  - 仅允许修改 `IntakeTime/IntakeFrequency/MealType/FeelingCode/EatFeeling/SourceType`；
  - `UserId/FoodId` 不可修改。

### 12.3 边界与兜底策略

1. **无检测记录**：
   - `PrimaryScreenResultText` 视为“未知”，仅使用 `PrimaryScreenRelation='不限'` 的规则；
   - 若同时无症状日志，则推荐区回落为“通用友好食物分类”+“通用忌口分类”，并设置 `IsGeneralRecommend=true` 以便前端区分文案。
2. **无规则命中**：
   - 即使前置过滤后没有任何 `tdietrecommendrule` 命中，仍返回通用推荐（同上），保证页面不出现空白区域。
3. **分类/食物被隐藏或软删除**：
   - 所有前台查询必须带上：
     - 分类：`ShowStatus=1 AND IsDelete=0`；
     - 食物：`ShowStatus=1 AND IsDelete=0`；
   - 若规则引用了已隐藏分类，最终会在查询阶段被过滤掉，不会出现在推荐结果中。
4. **规则冲突处理**：
   - 同一分类被多条规则命中时：
     - 分类在结果中只出现一次；
     - 其提示文案以“优先级最高（`RecommendPriority` 最小）且最新创建”的规则 `RuleDesc` 为主。

### 12.4 管理端操作与前台一致性

- **软删除策略**：
  - 分类/食物/规则/模板均采用 `IsDelete` 字段软删除，不直接物理删除；
  - 管理端删除操作仅改 `IsDelete=1`，前台全部以 `IsDelete != 1` 为准过滤。
- **规则启停**：
  - 管理端增加 `IsEnabled` 开关，表示是否参与匹配；
  - 停用规则不影响历史记录，仅影响后续推荐计算。
- **计划模板**：
  - 删除模板时仅将模板与明细 `IsDelete=1`，外键约束仍保留；
  - 若后续需要彻底清理，可由运维通过脚本进行物理删除。

### 12.5 接口返回与错误码规范（饮食模块）

- 所有 `POST /Front/Diet/*` 接口统一返回结构：
  - `Code`：业务编码，如 `Success`、`Diet.FoodNotFound`、`Diet.RecordNotFound` 等；
  - `Success`：布尔值；
  - `Msg`：面向用户的友好提示；
  - `Data`：真正业务数据，对错误场景可为 `null` 或空对象。
- 推荐的错误码：
  - `Diet.FoodNotFound`：`FoodId` 无效或已删除；
  - `Diet.RecordNotFound`：编辑/删除时找不到对应记录或不属于当前用户；
  - `Diet.RuleConfigError`：内部规则配置严重异常（如引用不存在分类），前台仅提示“当前推荐配置存在异常，已通知管理员处理”，详细信息写入服务端日志。

### 12.6 并发与幂等性说明

- 饮食记录创建本身不需要幂等（同一时间段多次记录是合法行为），只需保证单次请求事务内写入完整。
- 若后续实现“从推荐卡片一键加入饮食记录”的功能：
  - 建议通过 `SourceType=1` 标记此类记录；
  - 不强制对“同一餐同一食物”做幂等约束，以避免限制真实摄入行为的记录。
